▸ Case 01 · 提示词驱动

让 AI 帮你写"图文并茂"的 HTML 博客

07 / 11

这是我使用频率最高的场景。它是一个典型的 "提示词驱动" 工作流 —— 你不动手写,只动嘴(键盘)。 关键不在 AI,而在那条"提示词"。它就像给大厨的菜谱:菜谱写得好,菜才能好。

🧬 提示词的"四段式骨架"

我所有写博客的提示词,都遵循同一套结构 —— 角色 + 任务 + 思考方法 + 输出要求。这套骨架是我反复打磨出来的"模板",可直接复用。

1

定义角色 Role

告诉 AI 它是"谁" —— 越具体越好。

你是 Cisco 资深系统工程师,精通 Cisco Networking 与 Security 解决方案,持有 CISSP 证书,拥有 20 年网页设计经验……
为什么有效? AI 会根据"角色"调整知识范围、语气和深度。设定 CISSP 专家,它就不会用小学生口吻和你聊安全。
↓
2

设置任务与约束条件 Task & Constraints

明确目标 + 边界 —— AI 最怕"含糊"。

请撰写一篇技术白皮书,采用"总-分-总"结构,使用 Cisco 官方配色,所有图表必须 SVG,必须包含打印样式……
为什么有效? 约束越清晰,AI 越能"对齐"你的期望。"总-分-总"结构、配色规范、SVG 等限制,把发散变成聚焦。
↓
3

指定思考方法 Reasoning

让 AI "按你的方式思考",而不是按它的默认方式。

请用第一性原理 + 苏格拉底提问法构建文章。每个新概念必须给出"精准定义 + 精妙类比"。
为什么有效? 不指定方法时,AI 默认是"列要点"; 指定了,它会层层追问、推导,产出深度文章。这是质变。
↓
4

输出格式与质量标准 Output

明确"成品"长什么样 —— 别让 AI "猜"。

输出形式:单一完整 HTML 网页;Light Mode 风格;字号 ≥16px;Container 自适应;按章节分批输出,等我说"继续"再继续……
为什么有效? 格式定得越死,返工越少。"分批输出 + 等用户确认"还能避开 Token 限制导致的"输出截断"。
本案例的精髓: 写好提示词,本质是"把模糊需求翻译成精确指令"。 它考验的不是技术,而是 "你到底想要什么"的自我澄清能力。

▸ Case 03 · 资料驯服

NotebookLM:让一份文档"开口说话"

09 / 11

通用大模型最大的"原罪",是幻觉 —— 一本正经地胡说八道。
而 NotebookLM 用一个朴素而强悍的设计,几乎根除了这个问题:"我只看你给我的资料,绝不外延、绝不瞎编。"

📖 一个真实场景:让 RoomOS API 文档"自己回答我"

①

遇到的问题

我需要写一段宏脚本(Macro),实现 Cisco RoomOS 设备的"会议屏幕分配"——但 RoomOS 的 API 文档动辄几百页,翻起来眼花。

②

传统做法 vs NotebookLM 做法

传统做法

Ctrl+F 搜关键词 → 在几百页里翻 → 拼凑示例代码 → 反复试错

⏱️ 半天起步
NotebookLM 做法

把 API 文档拖进 NotebookLM → 用自然语言提问:"调用哪些 API 实现会议屏幕分配?"

⚡ 几分钟拿到答案 + 引用页码
③

关键差异:它真的"只读你的资料"

每一条回答都会标注 "引用自原文第 X 页"。如果文档里没有,它会直接告诉你"资料中未提及"——而不是像 ChatGPT 那样,自信地编一个不存在的 API 函数给你。

🧠 为什么"不瞎编"?—— 第一性原理拆解

🎙️ 玩出花:把文档变成"双主播播客"

一键生成,通勤路上听技术

NotebookLM 可以把上传的资料,自动生成双主播形式的中英文音频播客——一个人发问、一个人解答,你听起来就像在听一档真人科技播客。

💬 问答形式 · 一问一答,适合学习
⚔️ 辩论形式 · 正反交锋,激发思考

👂 我做过尝试:把一份枯燥的产品白皮书,变成 10 分钟的播客。开车回家路上听一遍,印象深刻 —— 这才是"信息消费"的现代方式。

📚
类比时间: 通用大模型像"夸夸其谈的演讲家",有问必答,但你不知道哪句是真的; NotebookLM 像"认真的图书管理员",只回答你给他的那本书里有的内容,且每句话都会指给你看出处。
—— 想要"靠谱",就让 AI"先看资料、再开口"。

🎁 实战延伸:我向 Prompt Hub 贡献的一个模板

说到 Prompt Hub,我不光"消费"别人的模板 —— 我也"生产"过一个, 已经提交并通过审核。它解决了一个 TAC 工程师每天都会遇到的真实痛点: 把一团乱麻的 Case Notes,在 30 秒内整理成结构化、可决策的摘要。

🎯 这个 Prompt 是什么?

名称:Cisco Case Summary Structuring

用途:把 TAC Case 中散乱的症状描述、日志片段、配置截图、客户抱怨、工程师笔记, 自动提炼成 "四段式结构化摘要",让任何人在 30 秒内 get 到 case 全貌。

受众:TAC 工程师、客户 IT 管理员、升级团队 —— 任何需要"快速搞清楚发生了什么"的人。

📐 这个 Prompt 的核心:固定的"四段式输出框架"

它强制 LLM 把任何输入的 Case Notes,都按以下四个维度"装进盒子",一段都不能少、一段都不能多:

📋
故障现象 (Fault Symptom)

客户实际观察到、报告了什么? 具体到设备、协议、接口、报错信息、异常行为。2-4 句话讲清楚。

💼
业务影响 (Business Impact)

这个故障对客户业务造成了什么影响?尽可能量化:停机时长、影响用户数、退化的服务。从客户视角看紧迫性与严重性。

🔍
故障根因 (Root Cause)

金字塔原理:结论先行 + 简述技术机制。需要时用类比辅助。这是整段摘要的核心,必须直接、清晰、有逻辑闭环。

✅
解决方案 (Resolution)

编号步骤、可操作。包括配置变更、命令、补丁、临时绕过措施;最后给出预防建议。3-6 步以内,简洁有力。

🏗️
设计哲学: 这个 Prompt 的精髓不在"它让 AI 做什么",而在"它给 AI 的思维脚手架"—— 通过角色定义(资深 TAC)+ 思考方法(第一性原理 + 金字塔)+ 强制结构(四段式)+ 风格约束(精炼、可扫读), 让 LLM 的输出每次都达到"可直接交付给客户"的水准。
—— 这就是前面说的"提示词的四段式骨架"在真实业务场景的落地。
实战收益: 这个 Prompt 上线后,我自己处理 Case 的"现状梳理"环节从 15 分钟压缩到 1 分钟; 团队同事也可以一键复用,无需各自从零设计。这就是 Prompt Hub 的真正价值 —— 把"个人最佳实践"变成"团队公共资产"。
本案例的本质: CIRCUIT 不是一个"另一个 ChatGPT",而是一个 "为 Cisco 人量身定制的 AI 工作台"—— 它的价值不在"模型多强",而在"能让你放心地把真实工作交给它"。
—— 安全合规与生产力,在这里第一次合二为一。
▸ Conclusion · 收束

没有"最好"的 AI,只有"最合适"的 AI

11 / 11

回到最初的问题:"该用哪个 AI 工具?" —— 答案不在工具本身,而在你手头的任务。 下面这张"工具地图",是我四年实战沉淀下来的决策框架,送给你。

🗺️ 我的"工具地图":按任务选工具

🌱 三条心法,送给你

01

选趁手的,不选最贵的

"趁手"比"先进"重要。能帮你高效完成任务的工具,就是好工具。不要为了追新而追新,也不要囤积一堆"用不上的会员"。

02

守住合规底线

外部 AI 再强,也不能投喂 Cisco 机密。在涉及客户数据、内部资料的场景,CIRCUIT 是唯一选择。这条红线,不能碰。

保持开放心态,随时调整

AI 工具发展太快了。三个月前的"最佳实践",今天可能已经过时。比如 Google 搜索的 AI 模式,因为快、方便,我用得越来越多 —— 今天的主力,可能不是明天的主力。

The Real Game-Changer · 最重要的一条

在 AI 时代,
问题,比答案更重要

这是我用 AI 四年最深的体会 —— 当 LLM 能在一秒内生成无数答案, "提出对的问题"就成了真正的稀缺能力。 我反复验证过一件事:问得好,LLM 就能"知无不言、言无不尽";问得糙,再强的模型也只能给你一坨"正确的废话"。

W

What

问"是什么"
—— 让 AI 帮你定义、澄清、精确化

"Agentic AI 到底是什么?它和传统 AI 的本质区别在哪?"
W

Why

问"为什么"
—— 让 AI 帮你追溯本质、暴露假设

"为什么传统防火墙挡不住 Prompt Injection?根因是什么?"
H

How

问"怎么做"
—— 让 AI 帮你给出方案、落地步骤

"用 eBPF 在运行时拦截 Agent 越权,具体怎么实现?"
💡

What → Why → How,这是一条"逼出深度回答"的黄金提问链。 你只需要一个一个抛出去,LLM 就会被你"牵引"着,从概念到本质再到落地,层层深入, 把它脑子里所有相关的知识都倒给你看。
