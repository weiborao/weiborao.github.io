前言与导读

在当今高度数字化的工业环境中，无线通信已经不再仅仅是一种"便利性"技术，而是承载着关键任务数据传输、实时控制信令和安全监控视频流的核心基础设施。无论是矿山中高速行驶的自动驾驶矿卡、港口码头上来回穿梭的自动化跨运车（Straddle Carrier）、城市轨道交通中以 120 公里/小时速度运行的地铁列车，还是大型制造工厂中数以百计的自主移动机器人（AMR），它们都对无线网络提出了传统 Wi-Fi 或蜂窝网络难以完全满足的严苛要求：亚毫秒级确定性时延、零丢包漫游切换、高吞吐量与可靠性并存、以及在恶劣射频环境下的鲁棒运行能力。

Cisco Ultra-Reliable Wireless Backhaul（超可靠无线回传，以下简称 URWB）正是为了解决上述挑战而生的一项专用无线技术。URWB 的技术血脉可追溯至意大利公司 Fluidmesh Networks——该公司自 2005 年起便专注于工业级无线移动网络的研发，其核心技术 Fluidity（流动性无缝漫游）和 Prodigy（专有无线协议）在全球铁路、矿山、港口等场景中积累了超过十五年的规模化部署经验。2020 年 Cisco 完成对 Fluidmesh 的收购之后，将 URWB 技术深度整合到 Cisco 的 Catalyst 产品线和 IOS-XE 软件栈中，使其获得了更强大的平台支撑、更丰富的网络管理能力（如与 Catalyst 9800 无线控制器和 Catalyst Center 的集成），以及更广泛的全球销售与支持网络。

一、什么是 Cisco URWB（超可靠无线回传）

1.1 技术起源与发展历程

要全面理解 Cisco URWB，首先需要回顾其技术源头。URWB 的核心协议和算法并非由 Cisco 从零开始研发，而是来源于一家名为 Fluidmesh Networks 的意大利裔创业公司。Fluidmesh 由几位来自麻省理工学院（MIT）的研究人员于 2005 年在意大利米兰创立，其创始团队最初的研究方向是移动 Ad-Hoc 网络（MANET）在工业场景中的实际应用问题。与学术界侧重理论分析和仿真不同，Fluidmesh 团队很早就意识到，传统的 IEEE 802.11 协议栈在工业移动场景下存在根本性的架构缺陷——尤其是在漫游切换延迟、多跳回传可靠性和确定性时延保障方面。

基于这一洞察，Fluidmesh 走了一条与众不同的技术路线：在保留 802.11 物理层（PHY）的同时，完全重新设计了 MAC 层及以上的协议栈。这一做法带来了两个关键优势。第一，通过复用 802.11 PHY，URWB 可以使用标准的、经过大量验证的 Wi-Fi 射频芯片组，从而在硬件成本和供应链方面保持竞争力，并且能够利用全球各国已有的 ISM/UNII 免许可频段法规框架。第二，通过抛弃 802.11 MAC 层中为通用消费级场景设计的诸多机制（如 BSS/SSID、Association/Reassociation、CSMA/CA 退避算法中的某些随机性等），URWB 得以从底层构建针对工业移动通信优化的确定性媒体访问控制、超快速漫游和自适应链路管理能力。

Fluidmesh 最具代表性的两项核心技术分别是：

Prodigy（神童协议）

Prodigy 是 URWB 系统中替代传统 802.11 MAC 协议的专有无线协议。它实现了一种针对点对点和点对多点回传链路优化的确定性媒体访问控制机制。在 Prodigy 协议中，每对通信节点之间建立的是一条专用的无线"隧道"，而非传统 Wi-Fi 中多个客户端共享一个 BSS 的竞争访问模型。Prodigy 支持带宽的精确分配和优先级调度，能够保证关键流量（如控制信令）在高负载条件下仍然获得确定性的传输时延。此外，Prodigy 在链路层内置了前向纠错（FEC）和自适应调制编码（AMC）机制，能够根据实时射频环境动态调整传输参数，从而在信道质量波动时最大限度地维持链路的可用性和吞吐量。值得注意的是，虽然 Prodigy 不是 802.11 MAC，但它运行在标准 802.11 PHY 之上——即它使用的是相同的 OFDM/OFDMA 调制方式、相同的信道带宽（20/40/80/160 MHz）和相同的频段（2.4 GHz、5 GHz、6 GHz），因此在射频特性和频谱法规方面与标准 Wi-Fi 完全兼容。

Fluidity（流动性无缝漫游）

Fluidity 是 URWB 系统中实现移动终端（如车载接入点）在多个基础设施接入点之间无缝漫游的核心技术。Fluidity 最关键的创新是 Make-Before-Break（先连后断）漫游机制——这与传统 802.11 漫游中的 Break-Before-Make（先断后连）有着本质区别。在 Fluidity 模式下，车载端的接入点会利用其多射频架构（通常具有两个或更多独立射频模块），在仍然保持与当前基础设施接入点连接的同时，使用另一个射频提前扫描和建立到下一个基础设施接入点的链路。只有当新链路完全建立并确认可用后，才会将数据流量切换过去并释放旧链路。这种机制确保在整个漫游切换过程中，数据传输不会中断——理论上和实测中均可实现零丢包切换。Fluidity 还在网络层引入了一套基于 MPLS（多协议标签交换）的 Overlay 转发机制，使得 IP 地址在漫游过程中保持不变，上层应用完全感知不到底层无线链路的切换。关于 Fluidity 的详细工作原理将在第五章中深入分析。

Fluidmesh 在被 Cisco 收购之前，已经在全球范围内积累了大量成功部署案例。其产品被广泛应用于欧洲和澳大利亚的铁路系统（如列车到地面的视频监控回传和乘客信息系统）、南美和非洲的露天矿山（自动驾驶矿卡和远程遥控铲车的通信保障）、全球多个大型港口（如自动化集装箱码头的场桥-岸桥通信和 AGV 调度）、以及石油天然气钻井平台和城市公共交通系统等。这些部署验证了 URWB 技术在极端环境条件下的可靠性和稳定性，也为后来 Cisco 的产品化和市场推广奠定了坚实的基础。

2020 年，Cisco 宣布完成对 Fluidmesh Networks 的收购。这次收购标志着 Cisco 在工业无线领域的一次战略性布局。收购完成后，Cisco 将 Fluidmesh 的技术团队整合到其物联网（IoT）和工业网络事业部，并启动了一系列产品和软件的整合工作。整合的关键里程碑包括：

  • 硬件产品线统一：原 Fluidmesh FM 系列产品（如 FM1300、FM4500 等）逐步过渡到 Cisco Catalyst Industrial Wireless（IW）产品线。新一代旗舰产品包括 Cisco Catalyst IW9165E（外置天线型号，适用于车载或需要定制天线方案的场景）、Cisco Catalyst IW9165D（内置定向天线型号，适用于路侧/基础设施部署）、Cisco Catalyst IW9167E（超高性能外置天线型号，支持更多射频和更高吞吐量）以及最新的 Cisco CW9178I（支持 Wi-Fi 7 / 802.11be 的下一代 URWB 接入点）。
  • 软件平台整合：原 Fluidmesh 设备运行的是基于 Linux 的 Fluidmesh 专有操作系统。收购后，Cisco 将 URWB 功能移植到了 Cisco IOS-XE 操作系统平台上——这是与 Cisco Catalyst 交换机和路由器相同的企业级网络操作系统。截至 IOS-XE 17.18.x 版本，URWB 接入点可以运行在两种管理模式下：（1）独立模式（Standalone / Network Mode）——通过接入点本身的 Web 界面或 CLI 进行配置，适用于小型独立部署；（2）Catalyst 9800 WLC 管理模式——将 URWB 接入点注册到 Cisco Catalyst 9800 无线局域网控制器，通过 WLC 的集中化界面统一管理 URWB 和传统 Wi-Fi 接入点，实现 Day 0/Day 1/Day 2 的全生命周期管理。
  • 网络管理生态整合：URWB 接入点现已支持与 Cisco Catalyst Center（原 DNA Center）集成，实现 URWB 网络的自动化部署、策略管理、健康监控和故障排查。此外，URWB 设备支持现代网络遥测协议（如 gNMI/gRPC、NETCONF/YANG），可以与第三方网络管理和监控平台（如 Grafana、Splunk、Elastic Stack）无缝对接。

1.2 URWB 的技术定位与核心定义

理解 URWB 的技术定位，最关键的一点是认识到：URWB 不是 Wi-Fi 的"增强版"或"工业加固版"，而是一种在 802.11 PHY 之上构建的、完全不同的无线网络体系。这个区分至关重要，因为它决定了 URWB 网络的规划、设计和运维方式与传统 Wi-Fi 网络有着根本性的差异。

从协议栈的角度来看，传统 Wi-Fi（IEEE 802.11）是一个完整的、从 PHY 层到 MAC 层的标准协议族。Wi-Fi 设备之间通过 BSS（基本服务集）、Association（关联）、Authentication（认证）等标准化流程建立连接，数据转发遵循 802.11 帧格式和分布式协调功能（DCF）或增强型分布式信道访问（EDCA）的竞争访问机制。而 URWB 则只复用了 802.11 的 PHY 层规范（物理帧格式、调制编码方案、信道化方案），在 PHY 层之上替换为完全专有的 Prodigy 协议和 Fluidity 控制平面。这意味着：

  • URWB 接入点之间的通信使用的不是标准 802.11 帧，而是 Prodigy 专有帧格式。因此，标准 Wi-Fi 客户端设备（如笔记本电脑、手机、Wi-Fi 传感器）无法直接连接到 URWB 网络——它们需要通过 URWB 接入点的以太网端口以有线方式接入，由 URWB 网络提供无线回传（backhaul）传输。
  • URWB 网络中不存在传统 Wi-Fi 中的"SSID"、"客户端关联"等概念。取而代之的是"URWB 网络名称"（Network Name / Passphrase 或 Network Key）和基于角色的设备配对机制。
  • URWB 链路的建立和维护由 Prodigy 协议和 Fluidity 控制平面自动管理，不依赖 802.11 的扫描-认证-关联流程。

从功能定位来看，URWB 本质上是一个无线回传（Wireless Backhaul）和无线接入回传融合网络。在典型部署中，URWB 在移动载体（如火车车厢、矿卡驾驶室、港口吊具）与固定基础设施网络之间建立高性能无线链路。移动载体上的终端设备（摄像头、PLC 控制器、乘客 Wi-Fi 接入点等）通过以太网连接到车载 URWB 接入点，URWB 接入点再通过无线链路将数据回传到路侧基础设施 URWB 接入点，最终汇聚到有线骨干网络。这种"回传"定位意味着 URWB 网络对终端设备是透明的——终端设备看到的只是一个以太网接口，完全不需要安装任何特殊驱动或客户端软件，也不需要了解底层无线传输的任何细节。

核心定义总结：Cisco URWB 是一种基于标准 802.11 物理层、采用专有 Prodigy MAC 协议和 Fluidity 移动性控制平面的工业级无线回传技术。它专为需要超低时延（<10 ms 端到端）、零丢包漫游切换和高确定性传输的移动关键任务场景而设计，典型应用包括铁路车地通信、矿山自动驾驶、港口自动化、制造业 AGV/AMR 以及体育场/会展等高密度移动场景。

1.3 URWB 的核心价值主张

Cisco URWB 的价值主张可以从以下六个维度来理解。这六个维度共同构成了 URWB 相对于传统无线技术（包括标准 Wi-Fi、专有窄带工业无线和私有 LTE/5G）的差异化竞争优势。

1.3.1 超低确定性时延

在工业控制和自动驾驶场景中，时延的确定性与时延的绝对值同样重要甚至更加重要。传统 Wi-Fi 的典型端到端时延为 10–50 ms，但由于 CSMA/CA 竞争访问机制和重传机制的随机性，其时延抖动（jitter）可能非常大——在高负载或射频干扰条件下，个别数据帧的时延可能突然飙升到数百毫秒甚至发生丢包。这种不可预测性对于实时控制系统来说是不可接受的。URWB 的 Prodigy 协议通过确定性媒体访问调度（取代了 CSMA/CA 的随机退避）和优化的帧处理流水线，将单跳无线链路的时延控制在不超过 3–5 毫秒的水平，且时延抖动极小（通常小于 1 毫秒）。在典型的两跳部署（车载→路侧→汇聚）中，端到端时延（从车载设备以太网口到路侧汇聚交换机以太网口）可以控制在 10 毫秒以内。

1.3.2 Make-Before-Break 零丢包漫游

如前所述，Fluidity 技术实现的 Make-Before-Break 漫游机制是 URWB 最具特色的技术亮点之一。在移动载体高速行驶过程中（典型速度范围从 AMR 的 2 m/s 到高铁的 300+ km/h），车载 URWB 接入点需要频繁地在不同的路侧基础设施接入点之间切换。传统 802.11 的漫游过程通常涉及断开当前连接→扫描新接入点→认证→重新关联→（可选的四次握手密钥交换）这一系列步骤，整个过程即使在 802.11r/k/v 快速漫游优化的情况下也通常需要 50–150 毫秒，期间数据传输完全中断。而 URWB 的 Make-Before-Break 机制利用车载端的多射频架构，在漫游切换的全过程中始终保持至少一条有效的数据传输路径，从而实现零中断、零丢包的漫游体验。对于运行实时视频流、远程操控或自动驾驶控制信令的应用来说，这种连续不间断的连接性是不可或缺的基础保障。

1.3.3 高吞吐量与频谱效率

URWB 在保持超低时延和高可靠性的同时，并未牺牲吞吐量。新一代 URWB 接入点（如 IW9167E 和 CW9178I）支持 Wi-Fi 6/6E/7 级别的物理层能力，包括 160 MHz 信道带宽、OFDMA（正交频分多址）、1024-QAM 乃至 4096-QAM 调制、以及多用户 MIMO（MU-MIMO）。在理想射频条件下，单链路吞吐量可达数百 Mbps 乃至 Gbps 级别。更重要的是，URWB 的 MPO（多径优化）技术允许在一对收发设备之间同时建立多条无线链路（使用不同的射频模块和/或不同的频段/信道），并在这些链路之间实现智能负载均衡和快速故障切换。通过 MPO，系统的聚合吞吐量可以线性叠加，同时链路的韧性（resilience）也得到显著增强——即使其中一条链路因干扰或遮挡而暂时不可用，数据仍然可以通过其余链路无缝传输。

1.3.4 使用免许可频段，降低运营成本

与私有 LTE/5G（需要许可频谱，或依赖 CBRS 等共享频谱方案）不同，URWB 运行在全球通用的 5 GHz UNII 免许可频段（部分型号也支持 2.4 GHz 和 6 GHz）。这意味着部署 URWB 网络无需向任何监管机构申请频谱许可证，无需支付频谱使用费，也无需担心许可证续期和频谱拍卖等问题。对于运营预算有限的工业用户来说，这是一个显著的成本优势。当然，使用免许可频段也意味着需要与其他 Wi-Fi 设备和可能的雷达系统共享频谱，URWB 通过其先进的频谱管理和干扰规避机制来应对这一挑战（详见后续章节关于 DFS 和信道选择的讨论）。

1.3.5 标准以太网兼容性与应用透明性

URWB 网络对上层应用和终端设备呈现为一个标准的二层以太网桥（在 L2 模式下）或一个三层路由网段（在 L3 模式下）。终端设备通过 RJ-45 以太网口或 SFP 光纤口连接到 URWB 接入点，就像连接到一台普通的以太网交换机一样。URWB 网络负责在无线链路上透明地转发以太网帧（L2 模式）或 IP 数据包（L3 模式），终端设备无需安装任何特殊软件或驱动，也无需进行任何网络配置上的特殊适配。这种兼容性极大地简化了与现有工业系统的集成工作——无论是 IP 摄像头、PLC 控制器、Wi-Fi 接入点还是其他任何以太网设备，都可以即插即用地通过 URWB 网络进行通信。

1.3.6 端到端的 Cisco 生态系统整合

作为 Cisco Catalyst 产品家族的一部分，URWB 接入点可以与 Cisco 的整个网络基础设施生态系统无缝集成。这包括：通过 Catalyst 9800 WLC 实现集中化的无线网络管理和策略下发；通过 Catalyst Center 实现网络自动化、Assurance（健康分析）和合规性审计；通过 Cisco Identity Services Engine（ISE）实现基于身份的网络访问控制；通过 Cisco Cyber Vision 实现工业网络中的资产可见性和威胁检测；以及通过 Cisco ThousandEyes 实现端到端的网络性能可观测性。这种深度的生态系统整合意味着企业客户不需要为 URWB 网络单独建立一套管理和运维体系，而可以将其纳入现有的 Cisco 网络管理框架中统一管理，从而降低运维复杂性和总拥有成本（TCO）。

1.5 管理模式：独立模式 vs. Catalyst 9800 WLC 管理模式

URWB 接入点当前支持两种管理模式，每种模式适用于不同的部署规模和管理需求。

独立模式（Standalone / Network Mode）

在独立模式下，每台 URWB 接入点独立运行，通过其内置的 Web 管理界面（GUI）或命令行接口（CLI）进行配置和管理。独立模式的优点是部署简单、无需额外的控制器基础设施、适合小规模或概念验证（PoC）部署。缺点是缺乏集中化的配置管理和监控能力，当网络规模增长时（例如超过 10–20 台接入点），逐台设备管理的效率会显著下降。在独立模式下，设备角色和网络拓扑使用传统的 Fluidmesh 术语体系——即 Mesh End（网状末端）、Mesh Point（网状节点）等。

Catalyst 9800 WLC 管理模式

在 WLC 管理模式下，URWB 接入点通过 CAPWAP（Control And Provisioning of Wireless Access Points）协议注册到 Cisco Catalyst 9800 无线局域网控制器。WLC 负责向接入点下发配置（包括 URWB 网络参数、射频参数、安全策略等），并提供集中化的监控、故障排查和软件升级能力。WLC 管理模式特别适合中大规模部署（数十到数百台接入点），以及需要与传统 Wi-Fi 网络统一管理的混合环境。在 WLC 管理模式下，Cisco 引入了一套新的标准化术语体系来描述设备角色和网络配置——例如，传统的 Mesh End 在 WLC 模式下称为 Coordinator（协调器），Mesh Point 称为 Node（节点），传统的 Passphrase 称为 Network Key（网络密钥），Fluidity 模式称为 Mobility（移动性）模式等。这些术语映射关系将在第三章中详细列表说明。

选择建议：对于新建的 URWB 项目，Cisco 推荐优先选择 Catalyst 9800 WLC 管理模式，以获得更好的可扩展性、集中化管理能力和未来功能升级路径。独立模式主要适用于小规模试点、离线部署或有特殊管理架构要求的场景。需要注意的是，两种模式不能在同一个 URWB 网络中混合使用——即一个 URWB 网络中的所有接入点要么全部工作在独立模式下，要么全部注册到同一个 WLC。

1.6 URWB 的典型应用场景

为了帮助读者建立对 URWB 适用范围的直观认识，以下列出 URWB 技术最常见的几大应用领域和典型用例。

🚂 铁路与轨道交通

铁路是 URWB 最早也是最成熟的应用领域之一。典型用例包括：列车到地面（Train-to-Ground, T2G）的实时视频监控回传——将车厢内外的安防摄像头视频流通过 URWB 无线链路实时传输到轨旁光纤网络和监控中心；列车乘客信息系统（PIS）和列车自动控制系统（ATC/CBTC）的数据回传；车载 Wi-Fi 热点的互联网回传——为乘客提供高速上网服务。在这些场景中，URWB 接入点安装在列车车底或车顶，路侧接入点沿轨道以约 200–500 米的间距部署。列车在高速行驶过程中（速度可达 160–300 km/h），车载接入点通过 Fluidity 技术在各路侧接入点之间无缝切换，确保数据传输不中断。在城市地铁和轻轨场景中，还涉及隧道内部署的特殊挑战，通常采用定向天线沿隧道壁安装，配合较短的接入点间距来保证连续覆盖。

⛏️ 采矿业（露天矿 & 地下矿）

采矿业是 URWB 增长最快的应用领域。在露天矿场景中，自动驾驶矿卡（Autonomous Haulage System, AHS）需要持续不断的高带宽、低时延无线连接来传输 LiDAR 点云数据、高清视频流和车辆控制信令。矿卡的运行速度通常为 30–60 km/h，行驶路线遍布整个矿坑，覆盖面积可达数十平方公里。URWB 基础设施接入点部署在矿坑边缘的高处（如灯塔或专用杆塔上），使用大功率扇区天线覆盖矿坑内部区域。车载接入点安装在矿卡顶部，使用全向或定向天线。Fluidity 技术确保矿卡在各扇区之间移动时无缝切换。在地下矿场景中，部署挑战更大，需要在巷道中沿矿车运行路线部署一系列路侧接入点，通常配合以太网或光纤回传链路连接到地面网络。URWB 在地下矿中的一个重要优势是其对多径衰落和恶劣射频环境的鲁棒性——矿井巷道中的金属壁面和移动机械会造成严重的多径反射和干扰，URWB 的自适应链路管理和 MPO 技术能够有效应对这些挑战。

🚢 港口与航运

在自动化集装箱码头中，多种移动设备需要高可靠无线连接：自动化导引车（AGV）或自动跨运车（Auto Straddle Carrier）在堆场和岸桥之间运输集装箱；轨道式龙门吊（RMG）沿堆场轨道移动进行堆箱和取箱操作；岸桥（STS）操作员可能采用远程操控模式。这些设备的控制系统需要毫秒级的命令响应时间和极高的连接可靠性——一旦无线通信中断，设备将立即触发安全停车，严重影响码头的生产效率。URWB 在港口场景中通常采用沿堆场和码头前沿部署路侧接入点的方式，配合安装在各移动设备上的车载接入点，形成覆盖整个作业区域的移动无线网络。港口环境中的射频挑战包括大量金属集装箱和钢结构造成的多径反射和信号遮挡、以及与港口 Wi-Fi 和其他无线系统的共存干扰问题。

🏭 制造业与仓储物流

在现代智能工厂和自动化仓库中，自主移动机器人（AMR）和自动导引车（AGV）的数量正在快速增长。这些移动平台需要无线网络来接收任务指令、上传传感器数据和实现车队协同调度。虽然单个 AMR 的带宽需求可能不高，但当一个大型仓库中有数百台 AMR 同时运行时，对网络的容量、时延一致性和漫游切换性能都提出了很高的要求。URWB 的 Fluidmax 模式（点对多点架构，详见第四章）特别适合这类高密度移动设备场景——一个 Fluidmax 基站可以同时服务数十台移动端设备，并在它们移动过程中提供无缝漫游。此外，制造业场景中还常见悬挂式输送系统（overhead conveyor）、穿梭车（shuttle）和桥式起重机等移动设备，它们沿固定轨道运行，可以采用 URWB 的定向天线方案来优化覆盖和链路性能。

🏟️ 体育场馆与大型活动

体育场馆中的无线摄像机、移动广播设备和安保系统也是 URWB 的新兴应用场景。例如，在足球比赛或演唱会转播中，手持或肩扛式无线摄像机需要将高清/4K 视频流实时传回转播车，对带宽和时延都有极高要求。URWB 可以为这些移动摄像机提供稳定的无线回传链路，替代传统的微波或有线方案（后者限制了摄像机的移动自由度）。

以上仅列出了几个最具代表性的应用场景。实际上，URWB 的适用范围非常广泛，凡是涉及"移动设备需要高可靠、低时延无线连接"的场景，都可以将 URWB 作为潜在的技术方案进行评估。

二、URWB 与传统 WLAN（Wi-Fi）的本质区别

在第一章中，我们已经从宏观层面介绍了 URWB 的技术定位——它不是 Wi-Fi 的"增强版"，而是一个在 802.11 PHY 之上完全重构的专用无线回传系统。本章将深入到技术细节层面，从架构理念、协议栈设计、漫游机制、组网模型、管理范式和适用场景六个维度，系统性地对比 URWB 与传统 WLAN 的差异。理解这些差异对于正确规划和部署 URWB 网络至关重要——许多常见的部署问题都源于将 Wi-Fi 的设计思维和经验直接套用到 URWB 项目中。

2.1 架构理念的根本差异

传统 WLAN 的设计哲学是"以客户端为中心"（Client-Centric）。整个 802.11 协议族的核心目标是让尽可能多的异构客户端设备（从笔记本电脑到物联网传感器，从智能手机到打印机）能够方便地连接到一个共享的无线网络中，并获得"尽力而为"（best-effort）的网络服务。为此，802.11 采用了 BSS（基本服务集）的概念——一个由 SSID 标识的逻辑网络，任何拥有正确凭证的客户端都可以通过标准化的扫描-认证-关联流程加入。在接入后，所有客户端共享同一信道的空口资源，通过 CSMA/CA（载波侦听多点接入/冲突避免）机制以分布式竞争的方式获取发送机会。这种架构的优点是通用性极强、兼容性极佳、部署和管理相对简单，因此成为了全球最普及的无线局域网技术。然而，其固有的随机竞争访问特性和为通用场景设计的漫游流程，使得它在需要确定性时延和无缝移动性的工业场景中表现不佳。

URWB 的设计哲学则是"以链路为中心"（Link-Centric）和"以移动性为中心"（Mobility-Centric）。URWB 网络中不存在传统意义上的"客户端"——每一个 URWB 接入点都是网络中的一个节点，节点之间建立的是专用的、确定性的无线链路（而非共享的广播媒体）。网络的核心目标不是服务大量异构终端设备，而是在一组预先规划和部署的 URWB 节点之间，建立高性能、高可靠的无线传输路径。终端设备通过以太网口"挂载"到 URWB 节点上，URWB 网络为其提供透明的二层或三层转发服务。这种架构使得 URWB 可以从底层针对链路性能和移动性进行深度优化，但代价是丧失了与标准 Wi-Fi 客户端的直接互操作性。

常见误区提醒：由于 URWB 运行在与 Wi-Fi 相同的频段上，且使用的硬件外观也类似于 Wi-Fi 接入点，初次接触 URWB 的工程师很容易产生一个误解——认为可以把 URWB 接入点当作"更好的 Wi-Fi 接入点"来使用，让笔记本电脑或手机直接连接。这是不可能的。URWB 接入点的无线接口运行的是 Prodigy 专有协议，不兼容标准 802.11 MAC 帧——标准 Wi-Fi 客户端在扫描时甚至看不到 URWB 网络（它不会广播 Beacon 帧）。如果项目中需要同时提供设备端的 Wi-Fi 接入服务（例如为矿卡驾驶室中的平板电脑提供 Wi-Fi 连接），应在 URWB 接入点的以太网下游另外部署一台标准 Wi-Fi 接入点（如 Cisco Catalyst 9100 系列），由 URWB 为该 Wi-Fi AP 提供无线回传。自 IOS-XE 17.18.1 版本起，Cisco 引入了新的无线架构——在同一台 AP 上同时运行 Wi-Fi 和 URWB（Simultaneous URWB and Wi-Fi operations）。这意味着你绝对不需要再像过去那样，在 AGV 或厂房内部署并排的两套物理 AP 基础设施。

2.2 协议栈的详细对比

从协议栈的角度来看，URWB 与传统 WLAN 的差异主要体现在数据链路层（第二层）及其上方的控制平面和转发平面。

物理层（PHY Layer）

在物理层，URWB 和传统 Wi-Fi 使用的是相同的 802.11 PHY 规范。这意味着：

  • 使用相同的调制编码方案（MCS），包括 OFDM（802.11a/n/ac/ax 中使用的正交频分复用）和 OFDMA（802.11ax 引入的正交频分多址）。
  • 支持相同的信道带宽选项：20 MHz、40 MHz、80 MHz、160 MHz（Wi-Fi 6/6E），以及 320 MHz（Wi-Fi 7）。
  • 运行在相同的频段：2.4 GHz ISM、5 GHz UNII（UNII-1 到 UNII-3 及部分 UNII-4 子频段，视各国法规而定）和 6 GHz（Wi-Fi 6E/7 频段）。
  • 遵守相同的国家/地区射频法规（发射功率限制、DFS/TPC 要求等）。

复用 802.11 PHY 的主要好处是：URWB 可以使用成熟的、批量生产的 Wi-Fi 射频芯片组（如 Qualcomm 的平台），从而在硬件成本、性能和可靠性方面享受 Wi-Fi 芯片产业的规模经济效应。同时，由于使用的是合规的 802.11 PHY 信号，URWB 设备在各国的射频合规认证（如 FCC、CE、MIC、SRRC 等）方面也可以复用已有的 Wi-Fi 认证框架。

MAC 层与链路管理

这是 URWB 与传统 Wi-Fi 最核心的分界点。

传统 Wi-Fi MAC 层（IEEE 802.11 MAC）的关键特征包括：

  • BSS 机制：网络以 BSS 为单位组织。一个 BSS 由一个接入点（AP）和与其关联的若干客户端（STA）组成。BSS 通过 SSID 标识。客户端通过扫描（主动或被动）发现 BSS，然后执行 Open System Authentication → Association Request → Association Response 流程加入网络。在安全场景下（如 WPA2/WPA3），还需在关联后执行四次握手（4-way handshake）来协商会话密钥。
  • CSMA/CA 竞争访问：所有关联到同一 BSS 的设备共享同一信道的传输时间。每个设备在发送前必须先侦听信道是否空闲（carrier sense），如果信道忙则进入随机退避（random backoff），退避计数器归零后才可尝试发送。这种机制在低负载时效率尚可，但在高密度高负载场景下，退避冲突和隐藏节点问题会导致严重的吞吐量下降和时延飙升。802.11ax 引入的 OFDMA 和 BSS Coloring 在一定程度上缓解了密集部署的效率问题，但基本的竞争访问范式并未改变。
  • 漫游基于重关联：当客户端移动到另一个 AP 的覆盖范围时，需要执行解除关联（Disassociation）→ 重新扫描 → 重新认证 → 重新关联的流程。即使使用了 802.11r（快速 BSS 转换）来预先协商密钥，仍然需要执行 Reassociation Request/Response 帧交换，期间至少有几十毫秒的数据传输中断。
  • QoS 分类基于 EDCA：802.11e / WMM 定义了四个访问类别（AC_VO、AC_VI、AC_BE、AC_BK），通过不同的竞争参数（AIFS、CWmin、CWmax）实现差异化的优先级处理。但这本质上仍是一种概率性的优先级机制，无法提供硬实时保障。

URWB Prodigy MAC 层的关键特征包括：

  • 专用链路模型：每对 URWB 节点之间建立的是一条专用的、点对点或点对多点的 Prodigy 无线链路。链路的建立不依赖 BSS/Association 机制，而是通过 URWB 网络名称和预共享密钥（PSK）进行配对。一旦链路建立，两端节点之间拥有一条逻辑上独占的传输通道——不会有"其他客户端"来竞争空口资源。在点对多点（Fluidmax）场景中，基站侧会对多个移动端进行调度式资源分配，而非竞争式。
  • 确定性媒体访问调度：Prodigy 协议在链路层实现了一种确定性的时分/资源分配机制（具体的调度算法为 Cisco 专有，不公开完整细节）。其核心思想是由链路的一端（通常是基础设施侧/Coordinator）对另一端（Node/Mobility Client）进行集中调度，预先分配传输时隙和资源块，从而消除了随机退避带来的不确定性。这种调度机制与 802.11ax 的 OFDMA 触发帧机制有概念上的相似之处，但 Prodigy 的调度粒度更细、对时延的控制更严格，且不受 802.11 标准帧格式和流程的约束。
  • 内置 FEC 和 ARQ：Prodigy 链路层内置了前向纠错（FEC）编码和自适应自动重传请求（ARQ）机制。FEC 可以在不需要重传的情况下恢复一定比例的传输错误，从而减少由重传引起的额外时延。ARQ 则作为 FEC 的补充，对 FEC 无法恢复的错误帧进行快速重传。这两种机制的参数会根据实时信道条件动态调整，以在可靠性和效率之间取得最优平衡。
  • 链路层加密：URWB 支持在 Prodigy 链路层使用 AES 加密，对无线链路上传输的所有数据帧进行端到端加密。加密过程对上层协议透明，不引入额外的握手延迟（密钥在链路建立阶段一次性协商完成）。

控制平面

传统 Wi-Fi 的控制平面主要由两个层面组成：（1）AP 级别——每个 AP 独立管理其 BSS，处理客户端的关联/解关联请求、维护关联状态表、执行射频资源管理（如 DFS 信道切换、TPC 功率调整）等；（2）WLC 级别（如果使用控制器模式）——无线控制器集中管理所有 AP 的配置、安全策略、射频资源协调（RRM）、客户端漫游状态同步等。客户端漫游时，WLC 负责在旧 AP 和新 AP 之间同步客户端状态和密钥材料。

URWB 的控制平面则由 Fluidity 协议和 URWB 网络协调机制组成。Fluidity 协议运行在所有参与移动性管理的 URWB 节点上，负责以下关键功能：

  • 邻居发现和链路质量监测：车载端（Mobility Client / Vehicle）的 URWB 接入点持续监测与各个可见的路侧端（Mobility Base / Infrastructure）接入点之间的链路质量指标（包括 RSSI、SNR、丢帧率、往返时延等）。
  • 切换决策（Handoff Decision）：基于收集到的链路质量数据，Fluidity 算法决定何时以及向哪个目标路侧接入点发起切换。切换决策不仅考虑当前的瞬时信号质量，还会结合历史趋势和预测模型（例如，根据信号变化的速率推断移动方向和速度），以提前做出最优切换决策，避免"乒乓效应"（即在两个接入点之间频繁来回切换）。
  • Make-Before-Break 切换执行：当切换决策确定后，车载端使用其空闲的射频模块向目标路侧接入点发起链路建立，同时保持在当前射频模块上的数据传输不中断。新链路建立完成后，数据流量切换到新链路，旧链路释放。整个过程由 Fluidity 协议自动协调，无需上层干预。
  • Overlay 转发路径更新：每次切换完成后，Fluidity 协议会触发 Overlay 网络的转发路径更新——具体来说，是更新 MPLS 标签转发表，使得从有线网络发往车载端的数据包能够被正确地路由到车载端当前连接的路侧接入点。这个过程将在第六章中详细讨论。

数据转发平面

传统 Wi-Fi 的数据转发有两种主要模式：（1）本地转发（Local Switching / FlexConnect Local Mode）——AP 直接在本地将无线客户端的数据帧桥接到其有线端口所连接的 VLAN；（2）集中转发（Central Switching）——AP 将无线客户端的数据帧封装在 CAPWAP 数据隧道中，通过有线网络发送到 WLC，由 WLC 解封装后转发到目标网络。集中转发模式的好处是便于在 WLC 上实施统一的安全策略和 QoS 策略，但会增加数据路径的长度和时延。

URWB 的数据转发平面则基于一种分布式的 MPLS Overlay 架构（在 L2 Fluidity 模式下），或标准的 IP 路由（在 L3 模式下）。在 L2 模式中：

  • URWB 网络中的每个节点都运行一个轻量级的 MPLS 标签交换引擎。
  • 当一个以太网帧从终端设备进入车载 URWB 接入点的以太网口时，接入点查询本地的 MAC 地址-MPLS 标签映射表，为该帧添加一个 MPLS 标签头，然后通过 Prodigy 无线链路将其发送到路侧接入点。
  • 路侧接入点（如果它是 Mesh End / Coordinator）会剥除 MPLS 标签头，还原出原始的以太网帧，并将其从有线端口转发到上游网络。
  • 如果路侧接入点不是最终的汇聚点（例如在多跳 Mesh 网络中），它会根据 MPLS 标签转发表进行标签交换（swap）操作，将数据帧转发到下一跳节点，直到到达 Mesh End / Coordinator。
  • 反方向的数据流（从有线网络到车载端）遵循类似的 MPLS 标签转发逻辑。Mesh End / Coordinator 根据目标 MAC 地址查找对应的 MPLS 标签和下一跳节点，将数据帧封装后通过无线链路逐跳转发到车载接入点，最终从车载接入点的以太网口以原始以太网帧的形式送达终端设备。

这种基于 MPLS Overlay 的转发机制是 URWB 实现零丢包漫游的关键技术基础之一。由于 MPLS 标签的分配和管理完全由 URWB 控制平面控制，当车载端发生漫游切换时，只需更新相关的 MPLS 标签转发条目，即可将数据流无缝重定向到新的路侧接入点——这个过程比传统 Wi-Fi 漫游中的 MAC 地址表刷新和 ARP 表更新要快得多、可靠得多。

2.3 漫游机制的本质差异

漫游（Roaming / Handoff）机制是 URWB 与传统 Wi-Fi 最显著的技术差异点之一，其重要性怎么强调都不为过。对于在移动场景中评估无线技术方案的工程师来说，这往往是决定选择 URWB 还是 Wi-Fi 的关键因素。

传统 Wi-Fi 的 Break-Before-Make 漫游

传统 802.11 漫游本质上是一个"先断后连"（Break-Before-Make）的过程。其基本流程如下：

  1. 触发阶段：客户端检测到当前关联 AP 的信号质量下降到某个阈值（如 RSSI 低于 -75 dBm），触发漫游搜索。
  2. 扫描阶段：客户端在各信道上发送探测请求帧（Probe Request）或被动监听信标帧（Beacon），以发现可用的候选 AP。这个扫描过程需要客户端暂时离开当前数据信道，切换到其他信道上进行探测——这意味着在扫描期间，客户端无法在当前信道上收发数据。虽然 802.11k（邻居报告）和优化的扫描算法可以减少扫描时间，但完全消除扫描阶段的数据中断是不可能的（在单射频客户端上）。
  3. 决策阶段：客户端从扫描结果中选择最佳的目标 AP。
  4. 切换阶段：客户端向当前 AP 发送 Disassociation 帧（或直接停止在当前 AP 上的通信），然后向目标 AP 发送 Authentication 请求和（Re）Association Request。在安全网络中（WPA2/WPA3），还需要执行密钥协商——即使使用了 802.11r 的快速 BSS 转换（FT），仍然需要在 Reassociation 过程中完成密钥的派生和安装。整个切换阶段的耗时通常在 50–150 毫秒（使用 802.11r）到 500 毫秒以上（不使用 802.11r）。
  5. 恢复阶段：切换完成后，客户端在新 AP 上恢复数据传输。上层协议栈可能还需要额外的时间来恢复状态——例如，DHCP 续租（如果跨子网漫游还需重新获取 IP 地址）、TCP 连接的恢复（如果漫游中断超过了 TCP 重传超时）等。

在整个切换阶段（第 4 步）和可能的恢复阶段（第 5 步），客户端的数据传输是完全中断的。对于传输电子邮件或网页浏览这类非实时应用，几十到几百毫秒的中断通常不会被用户感知。但对于实时视频流、VoIP 通话或工业控制信令，即使 50 毫秒的中断也可能导致可见的卡顿、语音断续或控制超时。在高速移动场景（如车辆以 60 km/h 行驶，漫游间隔仅数秒），频繁的漫游中断会累积成严重的服务质量问题。

URWB 的 Make-Before-Break 漫游

URWB 的 Fluidity 技术实现了"先连后断"（Make-Before-Break）漫游。其基本流程与传统 Wi-Fi 有着本质区别：

  1. 持续监测阶段：车载 URWB 接入点（Mobility Client / Vehicle）利用其多射频架构中的一个射频模块（称为"扫描射频"或"备用射频"）持续不断地监测周围所有可见路侧接入点（Mobility Base / Infrastructure）的链路质量。注意这里的关键区别：传统 Wi-Fi 客户端需要在数据信道和扫描信道之间切换，导致扫描和数据传输互斥；而 URWB 的多射频架构允许一个射频模块专门用于数据传输，另一个射频模块同时用于环境扫描和备选链路的建立。
  2. 预建链路阶段：当 Fluidity 算法判断车载端即将移出当前路侧接入点的最佳覆盖范围，并且已经识别到一个更优的目标路侧接入点时，它会指示备用射频模块提前与目标路侧接入点建立 Prodigy 链路。这个链路建立过程在后台进行，不会影响主射频模块上正在进行的数据传输。
  3. 无缝切换阶段：当备用链路完全建立并经过质量验证后，Fluidity 协议执行数据路径的切换——将数据流量从主射频模块/旧路侧接入点切换到备用射频模块/新路侧接入点。由于新链路在切换之前就已经完全就绪，切换操作本身只涉及转发表的更新（MPLS 标签重映射），不涉及任何空口信令交互或密钥协商，因此可以在微秒级别完成——数据流量的中断时间为零或接近零。
  4. 旧链路释放阶段：切换完成后，旧射频模块释放与旧路侧接入点的链路连接，转变为新的"备用射频"，开始扫描下一个潜在的切换目标。

整个 Make-Before-Break 过程的核心优势在于：在任何时刻，车载端都保持着至少一条有效的数据传输路径——数据流从未中断。这使得 URWB 能够支持对连续性要求极高的应用，如自动驾驶的实时感知数据流、远程操控的控制回路和高清视频直播流，而不会出现传统 Wi-Fi 漫游中不可避免的传输间隙。

2.4 组网模型的差异

传统 Wi-Fi 的组网模型相对固定：通常由一组部署在固定位置的接入点（AP）组成一个扩展服务集（ESS），这些 AP 通过有线骨干网络（通常是以太网 + WLC）互联。客户端在各 AP 之间漫游时，始终是"客户端 ← → AP"的星形拓扑关系。虽然 802.11s 标准定义了 Mesh 网络拓扑，允许 AP 之间通过无线链路互联，但 802.11s Mesh 主要用于 AP 间的回传（解决布线困难的场景），在移动性支持方面功能有限。

URWB 的组网模型则更加灵活和多样化，支持以下几种基本拓扑类型（将在第四章中详细讨论）：

  • Point-to-Point (P2P / 点对点)：两个 URWB 节点之间建立单一的高性能无线链路，用于固定站点间的连接（如两栋建筑之间的无线桥接，或矿山坑口与矿区控制中心之间的回传链路）。这种模式下不涉及移动性。
  • Point-to-Multipoint (P2MP / 点对多点)：一个中心节点同时与多个远端节点建立无线链路，形成星形拓扑。适用于一个汇聚点需要连接多个分散固定站点的场景（如一个中心塔站覆盖多个周边监控点）。
  • Mesh（网状）：多个 URWB 节点通过无线链路互联形成网状拓扑，数据可以通过多跳路径在网络中转发。Mesh 拓扑提供了路径冗余和自愈能力——如果某条链路中断，数据可以自动切换到备选路径。
  • Fluidity（流动性移动）：这是 URWB 最具特色的组网模式，专门为移动场景设计。网络由固定的路侧基础设施节点（Infrastructure / Mobility Base）和移动的车载节点（Vehicle / Mobility Client）组成。车载节点在路侧节点之间移动时，通过 Make-Before-Break 机制无缝切换。路侧节点之间可以通过有线或 Mesh 无线回传互联。
  • Fluidmax（大规模点对多点移动）：这是 Fluidity 的扩展版本，采用类似蜂窝网络的点对多点架构。一个 Fluidmax Primary / Fixed Base 节点（类似于蜂窝基站）可以同时服务多个 Fluidmax Secondary / Fixed Client 或 Fluidity 移动客户端。Fluidmax 特别适合高密度移动设备场景（如仓库中的大量 AMR），可以显著减少所需的基础设施节点数量。

2.5 管理范式的差异

传统 Wi-Fi 网络的管理已经非常成熟，形成了以 WLC 为核心的集中化管理范式。WLC 负责向 AP 下发 WLAN 配置（SSID、安全策略、QoS 策略等）、RF 配置（信道、功率、RRM 策略等）和系统配置（管理 IP、NTP、Syslog 等）。AP 通过 CAPWAP 协议与 WLC 通信，WLC 维护所有 AP 和客户端的状态视图。网络管理员通过 WLC 的 Web UI 或 CLI 进行统一管理。更高层的管理平台（如 Catalyst Center / DNA Center）则在 WLC 之上提供自动化、Assurance 和合规性管理能力。

URWB 网络的管理范式取决于所选择的管理模式（参见 1.5 节）：

  • 独立模式下：每台接入点独立管理。配置通过每台设备的 Web GUI 或 CLI 进行。网络级别的协调（如射频信道分配、漫游参数调优）需要管理员在各设备上手动确保一致性。这种模式适合小规模部署，但缺乏集中化的配置同步和状态监控。
  • WLC 管理模式下：URWB 接入点注册到 Catalyst 9800 WLC，WLC 通过 URWB 专用的配置 Profile（包括 URWB Network Profile、URWB RF Profile 等）向接入点下发统一配置。管理员在 WLC 上定义这些 Profile，然后将其关联到相应的 AP 或 AP 组。WLC 还提供 URWB 接入点的实时状态监控、告警管理和软件升级功能。在与 Catalyst Center 集成时，可以进一步实现自动化部署工作流和网络健康分析。
管理模式选型的实际考量：在实际项目中，选择独立模式还是 WLC 管理模式，需要综合考虑以下因素：（1）网络规模——超过 10–15 台接入点时，WLC 管理的效率优势开始显现；（2）现有基础设施——如果客户已经部署了 Catalyst 9800 WLC 来管理传统 Wi-Fi 网络，添加 URWB 管理的边际成本很低；（3）功能需求——某些高级功能（如与 Catalyst Center 的集成、集中化的安全策略管理）仅在 WLC 管理模式下可用；（4）环境约束——在完全离线或网络连接极度受限的环境中（如深海平台、偏远矿山），独立模式可能更实际。

2.7 适用场景决策指南

在实际项目评估中，选择传统 Wi-Fi 还是 URWB，应根据项目的具体需求来决定。以下是一个简明的决策框架：

选择传统 Wi-Fi 的场景特征

  • 终端设备是标准 Wi-Fi 客户端（笔记本电脑、手机、平板、Wi-Fi 传感器等），需要直接通过无线方式接入网络。
  • 应用类型以"尽力而为"的数据通信为主（如办公上网、邮件、文件传输），对时延的确定性和漫游的连续性没有严格要求。
  • 终端设备的移动速度较低（步行到低速车辆），漫游事件不频繁。
  • 需要服务大量异构类型的终端设备，通用兼容性是核心需求。
  • 部署环境为室内办公区、零售商店、校园、医院等传统企业场景。

选择 URWB 的场景特征

  • 核心需求是为移动平台（车辆、列车、吊具、机器人等）提供高可靠、低时延的无线回传连接。
  • 应用类型包括实时视频监控、远程操控、自动驾驶控制信令、工业控制协议（如 Profinet、EtherNet/IP）等对时延和可靠性有严格要求的关键任务流量。
  • 移动平台的运行速度较高（中高速车辆到高铁），漫游事件频繁，漫游中断不可接受。
  • 终端设备通过以太网接口接入（或可以在车载端额外部署 Wi-Fi AP 提供客户端接入）。
  • 部署环境为工业场所——露天矿、港口码头、铁路沿线、工厂车间、仓库等。
  • 网络必须在恶劣的射频环境（大量金属反射、电磁干扰、多径衰落）中保持稳定运行。
混合部署提示：在许多实际项目中，URWB 和传统 Wi-Fi 并不是非此即彼的关系，而是互补的。一个典型的混合部署方案是：在移动载体（如矿卡）和路侧基础设施之间使用 URWB 提供回传连接，在移动载体内部（如矿卡驾驶室）使用标准 Wi-Fi 接入点（连接到 URWB 回传链路的以太网端口）为驾驶员的平板电脑和手持设备提供 Wi-Fi 接入。这种"URWB 回传 + Wi-Fi 接入"的分层架构既保证了回传链路的高性能和高可靠性，又保持了终端设备的兼容性和灵活性。在 Cisco 的产品生态中，车载端的 URWB 接入点（如 IW9165E）和 Wi-Fi 接入点（如 Catalyst 9100 系列）可以同时注册到同一个 Catalyst 9800 WLC 进行统一管理。

第三章  设备角色详解——理解 URWB 网络中的"每一个人"

在传统 Wi-Fi 网络中，角色划分非常简单：AP（接入点）负责发射信号，STA（站点/客户端）负责连接。管理层面再加一个 WLC（无线局域网控制器）就构成了全部角色。但当你第一次打开 URWB 的配置界面，你会看到一长串令人困惑的名词：Mesh End、Mesh Point、Infrastructure、Vehicle、Fluidmax Primary、Fluidmax Secondary……如果你再切换到 Catalyst 9800 WLC 的管理视角，这些名词又变成了 Coordinator、Node、Mobility Base、Mobility Client、Fixed Base、Fixed Client……

这种"双重术语体系"是许多工程师初次接触 URWB 时最大的心理障碍。本章的目标，就是用一次性、彻底的方式，把每一个角色的定义、职责、在拓扑中的位置、与其他角色的协作关系、典型硬件选型、以及多功能复合能力，全部讲清楚。

💡 为什么会有两套术语？
URWB 技术源自 Fluidmesh Networks，在被 Cisco 收购之前已经拥有一套完整的术语体系（我们称之为 Standalone 术语 或 经典术语）。2020 年 Cisco 完成收购后，为了将 URWB 融入 Catalyst 无线产品线，在 Catalyst 9800 WLC 的管理界面中引入了一套新术语。两套术语描述的是同一套技术概念，只是命名风格不同。经典术语更偏向"网状网络"的学术传统，而新术语更贴近 Cisco Catalyst 产品线的命名规范。在实际项目中，你需要能在两套术语之间自由切换，因为独立管理模式使用经典术语，WLC 管理模式使用新术语，而很多技术文档和 TAC 工单中两套术语是混用的。

3.2  概念层次模型：模式 → 角色 → 设备

为了避免在众多术语中迷失方向，我们需要建立一个清晰的三层概念模型：

🧩 三层概念模型

第一层：无线电模式（Radio Mode）——决定"这条无线链路如何工作"

  • Fixed：静态点对点链路，两端固定不动
  • Fluidmax (Fixed P2MP)：静态一对多链路，基于 TDMA 调度
  • Fluidity (Mobility)：动态移动链路，支持 Make-Before-Break 漫游

第二层：设备角色（Device Role）——决定"这台设备在拓扑中扮演什么"

  • 每种无线电模式下有自己的角色划分（例如 Fluidity 模式下分 Infrastructure / Vehicle）
  • 角色决定了设备的行为逻辑：是主动发起连接还是被动等待？是汇聚流量还是中继流量？

第三层：硬件设备（Hardware）——决定"用哪台物理设备来承载角色"

  • 同一款硬件可以配置为不同的角色（例如 IW9165E 既可以做 Mobility Base 也可以做 Mobility Client）
  • 但硬件的物理特性会影响最佳角色选择（例如 IW9165D 的内置定向天线更适合做固定基站）

用一个类比来理解：把 URWB 网络想象成一支足球队。无线电模式相当于战术体系（442 阵型、433 阵型）；设备角色相当于场上位置（前锋、中场、后卫）；硬件设备相当于具体的球员（梅西、C罗）。同一个球员可以踢不同位置，但他的身体素质决定了哪个位置最适合他。同理，同一台 IW9167E 可以配置为不同角色，但它三射频的硬件优势在某些角色上发挥得更好。

3.3  Mesh End / Coordinator —— 网络的"根"与"锚"

3.3.1 定义

Mesh End（经典术语）或 Coordinator（9800 术语）是整个 URWB 网络的根节点（Root Node）。它是无线域与有线域之间的网关，所有来自 URWB 无线网络的流量最终都汇聚到 Mesh End，再通过其以太网接口进入有线骨干网（交换机、路由器、防火墙等）。

3.3.2 核心职责

  • 流量汇聚与转发：将无线侧的 MPLS 封装流量解封装，转换为标准以太网帧，送入有线网络；反方向则将有线侧的以太网帧封装为 MPLS 后送入无线域。
  • MPLS 标签分配：作为 MPLS 域的边缘节点（类似 PE 路由器），负责为进入无线域的流量分配标签、为离开无线域的流量剥离标签。
  • 网络协调：在 Fluidity（Mobility）场景下，Coordinator 参与漫游决策的协调——当车载设备需要切换基站时，Coordinator 需要更新 MPLS 标签交换路径（LSP），确保流量无缝切换到新路径。
  • 时钟与同步：在部分部署场景中，Coordinator 还承担网络时钟参考源的角色，确保各节点的 TDMA 时隙对齐（尤其在 Fluidmax 模式下）。
  • 有线侧集成：Coordinator 的以太网口通常需要配置为 VLAN Trunk，以便承载多个 VLAN 的流量。对于 MPLS 开销，连接 Coordinator 的交换机端口 MTU 需要设置为至少 1544 字节。

3.3.3 在拓扑中的位置

Coordinator 始终位于 URWB 无线网络的"有线侧边缘"。你可以把它想象成一棵树的树根——所有营养（数据）最终都通过树根进出。在一个标准部署中，通常有一台主 Coordinator 和一台备份 Coordinator（通过 TITAN/High Availability 机制实现故障切换）。

📌 关键设计原则
每个独立的 URWB 无线域必须有且仅有一个活跃的 Coordinator。如果你需要冗余，可以部署两台 Coordinator 并启用 TITAN（High Availability），其中一台为活跃（Active），另一台为备用（Standby），故障切换时间典型值 <500ms。但在任意时刻，只有一台 Coordinator 处于活跃状态。

3.3.4 典型硬件选择

由于 Coordinator 通常部署在室内机房或受保护的控制室中，且需要稳定的有线连接，常见的硬件选择包括：

  • IW9165D：IP67 防护等级，内置定向天线，适合部署在室外但位置固定的场景（如港口控制塔、矿山基站杆）。当 Coordinator 需要直接参与无线链路（例如 P2P 链路的一端）时，IW9165D 是首选。
  • IW9167E：三射频高性能平台，适合大型部署中 Coordinator 需要同时连接多条无线链路的场景。
  • IW9165E：双射频、外置天线，适合空间受限或需要灵活天线配置的场景。但由于 IW9165E 主要定位为车载设备，在实际项目中较少用作 Coordinator。

3.3.5 与其他角色的协作关系

Coordinator 与网络中的所有其他角色都有交互：

  • 与 Mobility Base / Infrastructure：Coordinator 为基站分配 MPLS 标签，基站将移动客户端的流量通过 LSP 转发给 Coordinator。
  • 与 Mobility Client / Vehicle：当移动客户端发生漫游时，Coordinator 负责更新 LSP，确保流量路径的连续性。
  • 与 Fixed Base / Fluidmax Primary：在 P2MP 拓扑中，Coordinator 通常就是 Fixed Base 本身，或者通过一跳 P2P 链路连接到 Fixed Base。
  • 与备份 Coordinator：通过 TITAN 协议保持心跳检测，活跃 Coordinator 故障时备份自动接管。

3.4  Infrastructure / Mobility Base —— 移动网络的"灯塔"

3.4.1 定义

Infrastructure（经典术语）或 Mobility Base（9800 术语）是在 Fluidity（Mobility）模式下部署在固定位置的基站设备。它们沿着移动载体的运行路径（铁轨、矿道、港口堆场通道、工厂 AGV 导引线）按照一定间距部署，形成连续的无线覆盖带。

如果把 Coordinator 比作"树根"，那么 Mobility Base 就是"树枝"——它们从树根延伸出去，为移动的"树叶"（Mobility Client）提供连接点。或者用另一个更直观的类比：Mobility Base 就像高速公路旁的手机信号塔——你开车行驶时，手机不断在不同信号塔之间切换，但你的通话不会中断。Mobility Base 扮演的就是这个"信号塔"的角色，区别在于 URWB 的切换速度比手机网络快几个数量级。

3.4.2 核心职责

  • 无线覆盖：在 Fluidity 模式下持续广播信标，让移动客户端能够发现并评估信号质量。
  • 漫游协助：当移动客户端从相邻基站漫游过来时，Mobility Base 需要在漫游发生之前就与客户端建立好新的无线连接（Make-Before-Break 的"Make"阶段）。这要求相邻基站之间有足够的信号重叠区域。
  • 流量中继：将移动客户端的数据流量通过 MPLS LSP 转发到 Coordinator，再进入有线网络。
  • 信号质量报告：持续向网络控制平面报告与各移动客户端之间的信号质量（RSSI、SNR、PER 等），为漫游决策提供依据。

3.4.3 部署间距设计

Mobility Base 的部署间距是 URWB 网络设计中最关键的参数之一。间距取决于以下因素：

  • 射频环境：开阔地带（港口、旷野）可以拉大间距（200-500m），密集遮挡环境（隧道、仓库）需要缩短间距（50-150m）。
  • 移动速度：速度越快，相邻基站之间需要的重叠覆盖区域越大，以确保 Make-Before-Break 有足够时间完成。高速列车场景（300km/h+）对重叠区域的要求远高于 AGV 场景（5-10km/h）。
  • 频段选择：5 GHz 频段的传播衰减大于 2.4 GHz，相同条件下 5 GHz 部署需要更密的间距。
  • 天线增益与方向性：高增益定向天线可以在特定方向上延伸覆盖范围，但需要精确的天线对准。
💡 经验法则
在 Cisco 的官方设计指南中，建议相邻 Mobility Base 之间的信号重叠区域至少覆盖移动载体行驶 2-3 秒的距离。例如，如果列车速度为 80 km/h（约 22 m/s），则重叠区域至少应有 44-66 米。这确保了 Make-Before-Break 漫游过程有足够的时间窗口完成所有步骤（信号评估 → 建立新连接 → 流量切换 → 断开旧连接）。

3.4.4 典型硬件选择

  • IW9165D：最常见的 Mobility Base 硬件选择。内置定向天线无需额外安装外部天线，IP67 防护等级适合户外长期部署，安装简便（仅需固定在杆或墙上并接通 PoE 线缆）。
  • IW9167E：当基站需要同时服务大量移动客户端或需要同时维护多条射频链路（例如一条面向移动客户端、另一条用于基站间回传）时选用。
  • IW9165E + 外部定向天线：在需要特殊天线方向图（如环形全向覆盖）的场景下使用。

3.4.5 回传连接方式

Mobility Base 需要将收到的移动客户端流量回传到 Coordinator。回传方式有两种：

  • 有线回传：每台 Mobility Base 通过以太网线缆直接连接到有线交换机网络，再到 Coordinator。这是性能最佳、延迟最低的方式，但需要在每个基站位置铺设网线或光纤。
  • 无线回传（Wireless Backhaul）：Mobility Base 之间通过无线链路（P2P 或 Mesh）回传流量。这减少了布线需求，但会增加一跳延迟并消耗部分无线带宽。在铁路、矿山等布线困难的场景中很常见。

3.5  Infrastructure Relay / Base Relay —— 覆盖的"延伸臂"

3.5.1 定义

Infrastructure Relay（经典术语）或 Base Relay（9800 术语）是 Mobility Base 的无线中继延伸。当某些区域无法直接铺设有线回传到 Mobility Base，但又需要为移动客户端提供覆盖时，可以在该区域部署 Base Relay。Base Relay 通过无线链路连接到最近的 Mobility Base（或另一个 Base Relay），形成多跳中继链路。

3.5.2 核心职责

  • 扩展覆盖范围：在无法铺设有线连接的区域提供 Fluidity 无线覆盖。
  • 流量中继：将覆盖区域内移动客户端的流量通过无线多跳路径转发到最近的有线连接点（Mobility Base 或 Coordinator）。
  • 漫游支持：与 Mobility Base 一样参与 Make-Before-Break 漫游过程，移动客户端可以在 Mobility Base 和 Base Relay 之间无缝切换。

3.5.3 使用场景与限制

Base Relay 最典型的使用场景是铁路隧道：隧道内无法每隔几百米就接一根以太网线，但需要连续的无线覆盖。此时可以在隧道口部署一台有线连接的 Mobility Base，然后在隧道内每隔一定距离部署 Base Relay，通过无线多跳回传。

⚠️ 多跳中继的延迟影响
每增加一跳无线中继，端到端延迟大约增加 2-5ms（取决于射频条件和流量负载）。对于延迟敏感的控制类应用（如紧急制动信号），建议无线中继跳数不超过 3 跳。如果需要更多跳数，应优先考虑增加有线回传点的数量。此外，每一跳中继会消耗额外的无线带宽（因为中继设备需要同时收和发），多跳场景下可用吞吐量会显著下降。

3.5.4 典型硬件选择

Base Relay 的硬件选择与 Mobility Base 类似，常用 IW9165D（户外场景）或 IW9167E（需要独立射频做回传和接入的场景）。使用 IW9167E 时，可以将其中一个射频用于面向移动客户端的接入，另一个射频用于与上游 Mobility Base 的回传，避免接入与回传争用同一射频资源。

3.6  Vehicle / Mobility Client —— 移动的"旅行者"

3.6.1 定义

Vehicle（经典术语）或 Mobility Client（9800 术语）是安装在移动载体上的 URWB 设备。"移动载体"可以是铁路列车、地铁车厢、矿用卡车、港口龙门吊/跨运车、工厂 AGV、体育场内的摄像车——任何需要在运动过程中保持不间断网络连接的平台。

Mobility Client 是 URWB Fluidity 技术最核心的受益者——正是为了让它在高速移动过程中获得零丢包、亚毫秒级切换的网络体验，Fluidmesh 团队才发明了 Make-Before-Break 漫游机制和 MPLS-based Prodigy 协议。

3.6.2 核心职责

  • 持续扫描与评估：Mobility Client 的射频模块持续扫描周围的 Mobility Base 信号，评估 RSSI、SNR 等指标，维护一张"候选基站列表"。
  • 主动发起漫游：与传统 Wi-Fi 中由 AP 或 WLC 决定漫游不同，URWB 中的漫游决策由 Mobility Client 自主发起。当它检测到前方有更优的基站时，会在保持当前连接的同时，主动向新基站发起连接请求。
  • 双连接维持：在 Make-Before-Break 的过渡阶段，Mobility Client 同时与旧基站和新基站保持活跃连接，确保流量不中断。
  • 本地接入服务：Mobility Client 的以太网口连接着车载设备（PLC、摄像头、乘客 Wi-Fi 网关、工控终端等），为这些有线设备提供到无线网络的接入。从有线设备的视角看，Mobility Client 就是一台普通的以太网交换机——它们感知不到底层的无线传输和漫游过程。

3.6.3 L2 Fluidity 与 L3 Fluidity

Mobility Client 的部署有两种网络层级模式，这是 URWB 网络设计中非常重要的架构决策：

📡 L2 Fluidity（二层漫游）

特征：所有 URWB 设备（Coordinator、Mobility Base、Mobility Client）都在同一个 VLAN 和同一个广播域中。

优点：配置简单，所有设备共享同一子网，ARP/广播可以自由传播。

缺点：当移动客户端数量较多时，广播风暴风险增大；车载设备的广播流量会扩散到整个 URWB 域。

适用场景：小规模部署（<20 台移动载体）、车载设备数量少、网络结构简单的场景。

📡 L3 Fluidity（三层漫游）

特征：每台移动载体拥有独立的广播域/VLAN，车载设备的流量在 Mobility Client 处被隔离。不同移动载体之间的通信需要经过三层路由。

优点：广播域隔离，可扩展性强，支持大量移动载体并行；车载设备的广播不会影响其他载体。

缺点：配置相对复杂，需要在 Coordinator 或上游路由器上配置三层路由和 VLAN 间路由。

适用场景：大规模部署（20+ 台移动载体）、每台载体上有多个有线设备、需要流量隔离的场景（如地铁系统中每列车独立 VLAN）。

WLC 模式下的 L3 移动性架构支持已被列入未来的路线图中。

3.6.4 典型硬件选择

  • IW9165E：最典型的 Mobility Client 硬件。双射频、外置天线接口（允许连接车载专用天线，如鲨鱼鳍天线、低剖面天线）、IP30 防护等级（适合安装在车厢内部或设备柜中）、紧凑外形。
  • IW9167E：三射频，适合需要同时连接多个频段基站或需要更高冗余度的高端场景（如高速列车需要在两个频段上同时维护连接以提升可靠性）。
💡 为什么 Mobility Client 通常选择 IW9165E 而不是 IW9165D？
IW9165D 的内置定向天线是为固定部署优化的——天线方向图呈窄波束，需要精确对准远端设备。而移动载体需要的是全向或扇形覆盖，以便在行进过程中能接收到来自不同方向的基站信号。IW9165E 的外置天线接口允许连接各种车载天线（全向天线、双极化天线、MIMO 天线），提供更灵活的射频覆盖。此外，IW9165E 的 IP30 防护等级虽低于 IW9165D 的 IP67，但车载设备通常安装在设备柜内，有额外的物理防护。

3.7  Vehicle to Vehicle / Mobility Client-to-Client Handoff

3.7.1 定义

这是一种特殊的通信能力，允许两台 Mobility Client（移动载体）之间直接建立无线链路并交换数据，同时保持各自与基站网络的连接和漫游能力。

3.7.2 使用场景

想象一个矿山场景：两辆矿卡在同一条巷道中相向行驶，它们之间需要交换避碰信息。如果所有数据都必须经过 Mobility Base → Coordinator → Mobility Base 的路径，延迟会显著增加。Vehicle-to-Vehicle 能力允许两辆矿卡直接通信，将延迟降到最低。

另一个场景是列车编组：当两列车需要连挂运行时，车载设备之间需要直接通信以协调制动和牵引。Vehicle-to-Vehicle 提供了不依赖基站的直接通信通道。

3.7.3 技术约束

Vehicle-to-Vehicle 通信需要两台 Mobility Client 在射频可达范围内，且需要额外的射频资源（通常使用独立的射频链路或时隙）。启用此功能会消耗一定的系统资源，不建议在所有场景中默认开启。

3.8  Fluidmax Primary / Fixed Base —— P2MP 的"指挥中心"

3.8.1 定义

Fluidmax Primary（经典术语，也写作 Fluidmax Master）或 Fixed Base（9800 术语）是在 Fluidmax / Fixed P2MP 模式下的中心基站。它像一个"指挥中心"，同时与多个远端站点（Fixed Client）通信，并通过 TDMA（时分多址）类似的调度机制来协调每个远端站点的发送时隙，避免冲突。

3.8.2 核心职责

  • TDMA 时隙调度：Fixed Base 是 P2MP 拓扑中的时隙调度器——它决定每个 Fixed Client 何时可以发送数据、何时必须接收数据。这种中心化的调度方式消除了随机竞争带来的延迟不确定性，是 Fluidmax 能够提供确定性低延迟的关键。
  • 流量汇聚：将所有 Fixed Client 的上行流量汇聚后转发到 Coordinator 或有线网络。
  • 链路质量监控：持续监控与每个 Fixed Client 之间的链路质量，动态调整调制编码方案（MCS）和发送功率。
  • 带宽分配：根据配置策略为每个 Fixed Client 分配上行/下行带宽比例。

3.8.3 与传统 Wi-Fi AP 的关键区别

虽然 Fixed Base 也是"一对多"通信的中心节点，但它与传统 Wi-Fi AP 有本质区别：传统 Wi-Fi 使用 CSMA/CA（载波侦听多路访问/冲突避免）——本质上是"谁先抢到信道谁先说话"的随机竞争机制。当连接设备较多时，竞争加剧，延迟显著增加且不可预测。而 Fixed Base 使用的 TDMA 类调度是"排好队轮流说话"——每个设备有固定的发言时间，延迟确定可控。

3.8.4 典型硬件选择

  • IW9165D：内置定向天线，适合覆盖特定扇区。可以部署多台 IW9165D 面向不同方向，组合覆盖 360°。
  • IW9167E：三射频平台，可以在多个频段上同时运行多个 P2MP 扇区，适合大规模 P2MP 部署。

3.9  Fluidmax Secondary / Fixed Client —— P2MP 的"远端站点"

3.9.1 定义

Fluidmax Secondary（经典术语，也写作 Fluidmax Slave）或 Fixed Client（9800 术语）是在 Fluidmax / Fixed P2MP 模式下的远端站点。它接受 Fixed Base 的时隙调度，在分配给它的时隙内发送和接收数据。

3.9.2 核心职责

  • 遵循调度：严格按照 Fixed Base 分配的 TDMA 时隙进行数据收发。
  • 本地接入：通过以太网口连接本地有线设备（监控摄像头、传感器、PLC 等），为它们提供到中心网络的连接。
  • 链路质量反馈：向 Fixed Base 报告链路质量指标，协助 MCS 调整和功率控制。

3.9.3 使用场景

Fixed Client 通常部署在需要固定无线回传的远端位置。典型场景包括：

  • 视频监控回传：城市或工业园区中，多个远端摄像头通过 Fixed Client 将视频流回传到中心机房。
  • 港口岸桥连接：多台岸桥通过 Fixed Client 连接到港口控制中心的 Fixed Base。
  • 矿山远端传感器：分布在矿区各处的环境监测传感器通过 Fixed Client 回传数据。

3.9.4 典型硬件选择

IW9165D 是 Fixed Client 最常见的选择——内置定向天线直接对准 Fixed Base，安装简单，无需额外天线调试。在距离较远或需要更高增益的场景下，也可使用 IW9165E + 高增益定向天线 的组合。

3.10  Fixed 模式（P2P）—— 两点之间的"专属通道"

3.10.1 定义

Fixed 模式是 URWB 最简单的无线电模式——两台设备之间建立一对一的静态点对点链路。两端设备位置固定不变，形成一条稳定的无线"桥梁"。

3.10.2 使用场景

  • 建筑间互联：两栋建筑之间铺设光纤成本过高或物理条件不允许时，用 P2P 无线桥接。
  • 基站间回传：在 Fluidity 部署中，多个 Mobility Base 之间通过 P2P 链路级联回传到 Coordinator，形成 Daisy-chain 拓扑。
  • 跨河流/跨公路连接：无法铺设物理线缆的地理障碍两侧通过 P2P 链路连接。

3.10.3 Fixed Automatic 变体

在经典术语中有一个 Fixed Automatic 变体，它允许 P2P 链路两端自动协商参数（如信道宽度、MCS 等级）。在 9800 管理模式下，这种自动协商能力被内置到 Fixed 配置中，不再作为独立选项出现。

3.11  Overlay Only / AutoTap —— 防止二层环路的"智能阀门"

3.11.1 为什么需要环路防护？

在 URWB 网络中，同一台设备可能同时有有线连接（以太网口连到交换机）和无线连接（通过 MPLS overlay 连到 Coordinator）。如果设备同时在有线侧和无线侧桥接二层流量，就可能形成二层环路——广播帧从有线口进入，通过无线 overlay 到达 Coordinator，Coordinator 再从有线口发出，又通过交换机回到起始设备的有线口……如此循环往复，广播风暴在瞬间就能瘫痪整个网络。

3.11.2 传统方案的不足

传统以太网使用 STP/RSTP（生成树协议）来防止环路。但 STP 有两个关键问题：

  • 收敛慢：即使是 RSTP，收敛时间也在秒级，这对于延迟敏感的工业应用不可接受。
  • 不理解 MPLS overlay：STP 只能看到物理链路拓扑，无法感知 URWB 的 MPLS overlay 路径，因此可能做出错误的阻塞决策。

3.11.3 AutoTap 的工作原理

AutoTap（在 9800 术语中对应 Autotap Loop Avoidance）是 URWB 特有的环路防护机制。其核心思路是：

  • 当 URWB 设备检测到自己同时有有线和无线两条到达 Coordinator 的路径时，自动禁用本地以太网口的二层桥接，仅通过 MPLS overlay 转发流量。
  • 这就是 Overlay Only 模式的含义——设备只通过 overlay 路径转发流量，不在本地以太网口做任何二层桥接，从根本上消除了环路可能性。
  • AutoTap 是自动检测、自动生效的，不需要手动配置。当设备检测到有线连接断开后，AutoTap 自动解除 Overlay Only 限制。
📌 与 STP 的关系
AutoTap 不是 STP 的替代品，而是 URWB overlay 网络的专属补充机制。在 URWB 设备的以太网侧（连接到有线交换机的部分），仍然建议在交换机上启用 STP 作为额外的安全保障。但在 URWB 无线域内部，环路防护完全由 AutoTap 负责，不依赖 STP。

3.12  Handoff —— 漫游的"那一瞬间"

Handoff 这个术语在两套命名体系中保持一致，不需要翻译对照。它指的是 Mobility Client 从一个 Mobility Base 切换到另一个 Mobility Base 的过程——不是某个角色，而是一个动作/事件。

URWB 的 Handoff 流程（Make-Before-Break）将在第五章进行深入解析。这里只需要知道：在 URWB 中，Handoff 的核心创新是在断开旧连接之前就建立好新连接，确保在切换过程中数据流不中断。典型的 Handoff 时间为 <1ms（理想条件下接近 0ms），这远优于传统 Wi-Fi 的 50-150ms 漫游延迟，甚至优于 4G/5G 蜂窝网络的切换延迟。

3.13  Fastfail / High Availability —— TITAN 协议与快速故障恢复

3.13.1 定义

Fastfail（经典术语）或 High Availability（9800 术语）指的是 URWB 网络中的快速故障检测与恢复机制，其核心是 Cisco 专有的 TITAN 协议。

3.13.2 TITAN 协议的工作原理

TITAN 通过以下机制实现快速故障恢复：

  • 心跳检测：网络中的关键节点（尤其是 Coordinator）之间持续交换心跳报文，检测间隔通常为 100-200ms。
  • 快速切换：当心跳检测到故障（如活跃 Coordinator 宕机、关键链路中断）后，备用路径或备用设备在 <500ms 内自动接管。
  • MPLS 路径重计算：故障切换后，所有受影响的 MPLS LSP 会自动重新计算并收敛，确保流量快速恢复。

3.13.3 TITAN 与 Fluidity 的协作

TITAN 负责的是设备/链路级别的故障恢复（例如某台 Mobility Base 硬件故障、某条回传链路断裂），而 Fluidity 负责的是移动性带来的正常漫游切换。两者是互补关系：

  • 正常漫游（Mobility Client 从基站 A 移动到基站 B）→ 由 Fluidity 处理 → 切换时间 <1ms
  • 故障切换（基站 A 突然宕机，Mobility Client 紧急切换到基站 B）→ 由 TITAN 处理 → 恢复时间 <500ms

3.14  Passphrase / Network Key —— 网络安全的"第一道门"

Passphrase（经典术语）或 Network Key（9800 术语）是 URWB 网络的预共享密钥（Pre-Shared Key, PSK）。所有属于同一个 URWB 网络的设备必须配置相同的 Network Key才能互相通信。这是 URWB 网络安全体系的第一道防线。

  • 身份验证：当一台 URWB 设备尝试加入网络时，它必须用 Network Key 进行身份验证。密钥不匹配的设备将被拒绝。
  • 数据加密：Network Key 用于派生数据加密密钥，保护空口传输的数据不被窃听。
  • 管理安全：建议定期轮换 Network Key，尤其在人员变动后。在 WLC 管理模式下，Network Key 的变更可以集中推送到所有设备。

3.15  多功能复合能力——一台设备，多种角色

URWB 硬件的一个重要特性是多功能复合能力：同一台物理设备的不同射频接口可以配置为不同的无线电模式和角色。这种灵活性在实际部署中非常有价值。

🔧 典型的多功能配置示例

场景：一台 IW9167E（3 射频）部署为 Mobility Base

  • Radio 0（5 GHz）：配置为 Mobility 模式 / Mobility Base 角色 → 面向移动客户端提供 Fluidity 覆盖
  • Radio 1（5 GHz）：配置为 Fixed 模式 → 与相邻 Mobility Base 建立 P2P 无线回传链路
  • Radio 2（2.4 GHz 或 5 GHz）：配置为 Fixed P2MP / Fixed Base 角色 → 同时为几台固定位置的监控摄像头提供 P2MP 连接

这样一台设备就同时承担了移动接入基站、回传中继、固定接入基站三种功能！

💡 IW9165E/D 的多功能限制
IW9165E 和 IW9165D 只有两个射频，因此多功能复合能力相比 IW9167E（三射频）会受到一定限制。典型的双射频配置是：一个射频用于接入/覆盖，另一个射频用于回传。如果需要更复杂的多角色配置，建议选择 IW9167E。

3.16  特别说明：Cisco Wireless with URWB（IOS-XE 17.18.1+）

从 IOS-XE 17.18.1 版本开始，Cisco 引入了一种全新的融合模式——Cisco Wireless with URWB。在这种模式下，同一台 Cisco Catalyst 无线接入点可以同时运行传统 Wi-Fi 和 URWB。例如，一台 CW9178I 的一个射频可以配置为标准 Wi-Fi 接入（为员工手机、平板提供网络），而另一个射频配置为 URWB 模式（为工业控制设备提供确定性低延迟连接）。

这种融合模式的意义在于减少硬件部署数量——在很多工业场景中，IT 网络（办公 Wi-Fi）和 OT 网络（工业控制 URWB）需要共存。以前需要分别部署两套硬件，现在可以用一套设备同时覆盖两种需求。在 WLC 管理界面中，这通过 URWB Profile、Radio Profile 和 RF Tag 来配置。

3.17  本章小结

📋 角色关系速查

用一段话概括所有角色的关系：

在一个典型的 URWB Fluidity 网络中，Coordinator（Mesh End）作为根节点连接有线网络，是所有流量的汇聚点。多台 Mobility Base（Infrastructure）沿移动路径部署，通过有线或无线回传连接到 Coordinator。在无法铺设有线回传的区域，Base Relay（Infrastructure Relay）通过无线多跳延伸覆盖。Mobility Client（Vehicle）安装在移动载体上，在 Mobility Base 之间无缝漫游（Handoff），通过 Make-Before-Break 机制确保零丢包切换。当网络中任何关键节点发生故障时，TITAN（High Availability / Fastfail）在 500ms 内完成故障切换。所有设备通过Network Key（Passphrase）进行身份验证和数据加密。AutoTap（Overlay Only）自动检测并防止二层环路。

而在静态场景中，Fixed Base（Fluidmax Primary）和 Fixed Client（Fluidmax Secondary）通过 TDMA 调度建立确定性的点对多点链路，用于视频监控、传感器数据回传等不涉及移动性的应用。

第四章  网络拓扑类型——URWB 的"战术编队"

如果说设备角色回答的是"每台设备做什么"的问题，那么网络拓扑回答的是"这些设备如何组合排列"的问题。在军事术语中，角色相当于兵种（步兵、炮兵、装甲兵），而拓扑相当于阵形（一字长蛇阵、方阵、楔形阵）。同样的兵种在不同阵形中发挥的作用完全不同。

URWB 支持五种主要的网络拓扑类型，每种拓扑适用于不同的应用场景。理解这五种拓扑以及它们之间的组合方式，是进行 URWB 网络设计的基础。

4.1  点对点拓扑（Point-to-Point, P2P）

4.1.1 定义与结构

点对点拓扑是最简单的 URWB 拓扑——两台设备，一条链路。一端配置为 Coordinator（Mesh End），另一端作为远端节点，两者之间建立一条专属的固定无线链路。

用最简单的类比：P2P 就是一根"无线网线"——它在两个物理位置之间建立一条透明的二层桥接通道，两端连接的有线设备就像直接插在同一台交换机上一样。

4.1.2 技术特性

  • 延迟：单跳延迟 <2ms，是所有 URWB 拓扑中延迟最低的。
  • 吞吐量：取决于信道带宽和射频条件，80 MHz 信道下最高可达 800+ Mbps 聚合吞吐量（使用 Wi-Fi 6 物理层）。
  • 距离：取决于天线增益和射频环境。使用 IW9165D 内置定向天线，典型可达 2-5 km（视距条件下）；使用高增益外部天线可延伸至 10+ km。
  • 可靠性：由于只有一条路径，没有冗余。如果链路中断（如恶劣天气、物理遮挡），通信即中断。建议对可靠性要求高的场景使用双 P2P 链路（不同频段）做冗余。

4.1.3 典型应用

  • 两栋建筑之间的网络互联（替代铺设光纤）
  • 河流两岸、高速公路两侧的网络桥接
  • 临时施工现场到办公室的网络连接
  • 作为 Fluidity 网络中基站间的回传链路

4.2  点对多点拓扑（Point-to-Multipoint, P2MP / Fluidmax）

4.2.1 定义与结构

点对多点拓扑由一台 Fixed Base（Fluidmax Primary）和多台 Fixed Client（Fluidmax Secondary）组成。Fixed Base 位于中心位置，同时与多个远端 Fixed Client 保持无线连接，形成"星形"拓扑结构。

类比理解：P2MP 就像一位老师（Fixed Base）同时给多名学生（Fixed Client）上课。老师通过点名（TDMA 时隙分配）来决定哪位学生可以发言，避免了所有人同时说话的混乱。

4.2.2 技术特性

  • TDMA 调度：这是 P2MP 与传统 Wi-Fi 的核心区别。Fixed Base 通过 TDMA 时隙分配为每个 Fixed Client 分配专属的发送和接收时间窗口，消除了 CSMA/CA 的随机竞争，提供确定性延迟。
  • 容量共享：所有 Fixed Client 共享 Fixed Base 的总带宽。例如 Fixed Base 总带宽为 500 Mbps，三个 Fixed Client 可能各分到 150-170 Mbps。具体分配取决于配置策略和各链路质量。
  • 扇区化部署：在需要 360° 覆盖时，可以部署多台 Fixed Base（或使用多射频设备的不同射频），每台覆盖一个扇区（如 90° 或 120°），组合实现全向覆盖。
  • 最大客户端数：单个 P2MP 扇区通常建议不超过 8-10 个 Fixed Client，以保证每个客户端获得足够的时隙和带宽。

4.2.3 典型应用

  • 港口/码头：一台 Fixed Base 安装在控制塔上，同时连接多台岸桥/场桥上的 Fixed Client
  • 城市视频监控：一台 Fixed Base 安装在高点，同时回传多个远端摄像头的视频流
  • 矿山环境监测：一台 Fixed Base 安装在矿区制高点，收集多个远端传感器站点的数据
  • 农业/水利：一台 Fixed Base 连接多个远端水位/气象监测站

4.3  Mesh 拓扑——自愈的"蜘蛛网"

4.3.1 定义与结构

Mesh 拓扑是由多台 URWB 设备通过 P2P 链路互联形成的网状结构，其中至少部分设备之间存在冗余路径。当某条链路故障时，流量可以自动切换到备用路径，无需人工干预。

如果把 P2P 比作"一根绳子"，P2MP 比作"一把伞"，那么 Mesh 就是"一张渔网"——任何一个节点断裂，网的其他部分仍然完整连通。

4.3.2 技术特性

  • 自愈能力：URWB 的 MPLS 控制平面会自动计算最优路径。当某条链路故障时，MPLS LSP 自动重新收敛，流量切换到备用路径。配合 TITAN 协议，故障恢复时间 <500ms。
  • 多路径负载均衡：当存在多条等价路径时，URWB 可以在多条路径之间进行流量负载均衡，提高整体吞吐量。
  • 跳数与延迟：每增加一跳 Mesh 中继，端到端延迟增加 2-5ms。因此 Mesh 拓扑不适合对延迟极度敏感的场景（如实时控制），更适合需要高可靠性但可以容忍适度延迟的场景。
  • 规划复杂度：Mesh 拓扑的射频规划比 P2P 和 P2MP 复杂得多——需要考虑信道分配（避免相邻链路干扰）、功率控制、路径选择策略等。

4.3.3 典型应用

  • 大型工业园区的网络覆盖：多栋建筑之间通过 Mesh 互联，任何一条链路断裂不影响整体连通性
  • 港口/码头回传网络：多个基站之间形成 Mesh 拓扑，提供冗余回传路径
  • 城市公共安全监控：多个监控点之间形成 Mesh，确保关键视频流在链路故障时仍能传输

4.4  Mobility 拓扑（Fluidity）——为运动而生的"移动走廊"

4.4.1 定义与结构

Mobility（Fluidity）拓扑是 URWB 最具技术含量、也是最核心的拓扑类型。它由以下元素组成：

  • 一台 Coordinator：连接有线网络的根节点
  • 多台 Mobility Base（可能包含 Base Relay）：沿移动路径部署的固定基站，形成连续覆盖带
  • 一台或多台 Mobility Client：安装在移动载体上的车载设备
  • 回传网络：连接各 Mobility Base 到 Coordinator 的有线或无线回传

用铁路类比：Coordinator 是车站调度中心，Mobility Base 是沿铁路线部署的信号基站，Mobility Client 是列车上的通信设备。列车行驶过程中不断在基站之间切换，但车上的乘客（有线设备）感知不到任何网络中断。

4.4.2 Make-Before-Break 漫游流程概览

Mobility 拓扑的核心价值在于 Make-Before-Break (MBB) 漫游机制（详细机制将在第五章深入分析）。这里先给出一个简化的流程概述：

  1. 持续扫描：Mobility Client 持续扫描前方基站的信号（不中断当前连接）
  2. 评估决策：当检测到前方基站信号优于当前基站时，触发漫游决策
  3. 建立新连接（Make）：Mobility Client 与新基站建立无线连接，同时保持与旧基站的连接——此时 Mobility Client 同时连接两个基站
  4. 流量切换：Coordinator 更新 MPLS LSP，将流量从旧路径切换到新路径
  5. 断开旧连接（Break）：确认新路径稳定后，断开与旧基站的连接

整个过程中，数据流从未中断——因为在断开旧连接之前，新连接已经在承载流量了。这就是"Make-Before-Break"名称的由来，也是 URWB 在移动场景中能实现零丢包漫游的根本原因。

4.4.3 技术特性

  • 漫游延迟：典型值 <1ms，理想条件下接近 0ms
  • 支持速度：已验证支持 300+ km/h 的移动速度（高速列车场景）
  • 并发移动客户端：单个 Mobility Base 可同时服务多个 Mobility Client（具体数量取决于射频条件和带宽需求）
  • 双栈支持：同时支持 L2 Fluidity 和 L3 Fluidity（见第三章 3.6.3 节）

4.4.4 典型应用

  • 铁路/地铁：列车与地面控制中心之间的实时通信（CBTC、PIS、CCTV）
  • 矿山：矿卡、铲车等移动设备的远程控制和视频回传
  • 港口：跨运车、AGV、龙门吊等移动设备的网络连接
  • 制造业：AGV/AMR 在工厂车间内的无缝网络覆盖
  • 体育/演艺：移动摄像车、无人机的实时高清视频回传

4.5  混合模式拓扑——现实世界的"组合拳"

4.5.1 定义与结构

在实际项目中，很少有网络只使用单一拓扑类型。更常见的是多种拓扑的组合，我们称之为混合模式拓扑。典型的混合模式包括：

🔗 混合拓扑模式 A：Mobility + P2P 回传

这是最常见的混合模式。多台 Mobility Base 沿移动路径部署为移动客户端提供覆盖，但基站之间通过 P2P 无线链路级联回传（Daisy-chain），而不是每台基站独立拉有线回传。这大幅减少了布线成本，特别适合铁路和矿山场景。

示例：铁路线沿线每隔 500m 部署一台 IW9165D 作为 Mobility Base。每 5 台基站通过 P2P 链路级联，最终汇聚到一台有线连接的 Coordinator。

🔗 混合拓扑模式 B：Mobility + Mesh 回传

与模式 A 类似，但基站之间的回传使用 Mesh 拓扑而非简单的 P2P 级联。这提供了回传层面的冗余——如果某条回传链路故障，流量可以自动绕行。适合对可靠性要求极高的场景（如矿山安全系统）。

🔗 混合拓扑模式 C：P2MP + Mobility

在同一个 URWB 网络中，部分设备运行 P2MP 模式（连接固定摄像头、传感器），另一部分设备运行 Mobility 模式（服务移动载体）。这种模式在港口场景中非常常见——岸桥通过 P2MP 连接到控制中心（固定位置），而 AGV/跨运车通过 Mobility 模式移动作业。

🔗 混合拓扑模式 D：Multi-Radio 融合

利用 IW9167E 的三射频能力，同一台设备的不同射频运行不同的拓扑模式。例如 Radio 0 运行 Mobility（面向移动客户端），Radio 1 运行 Fixed P2P（回传到 Coordinator），Radio 2 运行 Fixed P2MP（连接本地固定传感器）。这是 URWB 多功能复合能力的终极体现。

4.5.2 混合模式设计注意事项

  • 信道规划：不同射频链路应使用不同的信道（或频段），避免同频干扰。IW9167E 的三射频天然适合这种需求。
  • 带宽预算：无线回传链路的带宽是共享的——多跳回传的每一跳都会消耗带宽。设计时需要计算最坏情况下的总带宽需求，确保回传带宽充足。
  • 延迟预算：每增加一跳无线中继，延迟增加 2-5ms。对于控制类应用，需要确保端到端延迟在应用的容忍范围内。
  • MPLS 路径复杂度：混合拓扑中的 MPLS LSP 路径更复杂，需要确保 Coordinator 有足够的处理能力来管理大量 LSP。

4.7  拓扑选型决策流程

面对一个新项目，如何选择合适的拓扑类型？以下是一个实用的决策流程：

🧭 拓扑选型五步法

  1. 第一步：识别移动性需求
    项目中是否有设备需要在运动过程中保持不间断网络连接？
    • 如果是 → 必须包含 Mobility（Fluidity） 拓扑
    • 如果否 → 进入第二步
  2. 第二步：识别连接点数量
    需要连接的远端站点有多少个？
    • 1 个 → P2P
    • 2-10 个（且集中在一个方向扇区内） → P2MP
    • 多个（分散分布） → 考虑 Mesh 或多个 P2MP 扇区
  3. 第三步：评估可靠性需求
    是否需要链路冗余和自愈能力？
    • 如果是 → 引入 Mesh 拓扑或双 P2P 冗余链路
    • 如果否 → P2P 或 P2MP 即可满足
  4. 第四步：设计回传网络
    各基站如何连接到 Coordinator？
    • 每台基站都有有线连接 → 最简单，性能最优
    • 部分基站无法布线 → 使用 P2P 级联或 Mesh 无线回传
  5. 第五步：验证延迟与带宽预算
    计算最坏情况下的端到端延迟和带宽需求，确认所选拓扑能满足应用要求。如果不满足，考虑增加有线回传点、减少无线中继跳数、或升级到更高性能的设备（如 IW9167E）。

4.8  本章小结

本章系统介绍了 URWB 支持的五种网络拓扑类型：P2P（两点直连的"无线网线"）、P2MP / Fluidmax（一对多的"星形广播"）、Mesh（多路径自愈的"蜘蛛网"）、Mobility / Fluidity（移动无缝漫游的"移动走廊"）、以及混合模式（多种拓扑的"组合拳"）。

没有万能拓扑——每种拓扑都是在覆盖距离、吞吐量、冗余度、移动性和部署成本之间做出取舍。在实际项目中，工程师需要首先明确业务的核心需求（是固定回传？还是移动接入？还是两者兼有？），然后依据本章提供的五步选型法，选定主拓扑并叠加辅拓扑，最终形成一张既满足 SLA 又可持续演进的无线回传网络。

在确定拓扑之后，下一个关键问题便是：当车辆或移动终端从一个基站切换到另一个基站时，数据流如何不间断？ 这正是第五章将要深入剖析的 Make-Before-Break 漫游机制。

第五章 Make-Before-Break 漫游机制深度解析

对于任何需要支撑移动设备（矿用卡车、地铁列车、AGV、轮式起重机……）的无线回传网络来说，漫游（Handoff / Handover / Roaming）是最核心的技术挑战之一。在传统 Wi-Fi 世界中，漫游通常意味着短暂的连接中断；而在 URWB 体系下，Cisco 采用了与传统方案截然不同的Make-Before-Break（MBB）策略，将漫游切换时间从数百毫秒甚至秒级压缩到几乎为零。本章将从传统 Wi-Fi 漫游入手，逐步深入到 URWB 的 MBB 内部机制、双射频架构需求、切换量化指标以及硬件层面的射频约束，帮助读者全面理解这一关键技术。

5.1 传统 Wi-Fi 漫游回顾

5.1.1 基本漫游流程

在标准 802.11 协议体系下，一个无线客户端（STA）在移动过程中需要从当前关联的接入点（AP-old）切换到信号更强的接入点（AP-new）。这个过程大致分为以下几个阶段：

传统 Wi-Fi 漫游的典型阶段

阶段名称描述典型耗时
①检测（Detection）客户端检测到当前 AP 的信号质量低于漫游阈值（RSSI 下降、丢包率上升等），判定需要寻找更好的 AP。因实现而异；通常连续数帧低于阈值后触发
②扫描（Scanning）客户端在各信道上发送 Probe Request 或侦听 Beacon，发现候选 AP 列表。主动扫描需要在每个信道上停留 10-30 ms 发出 Probe Request 并等待 Probe Response，被动扫描则需等到下一个 Beacon（间隔通常为 100 ms）。2.4 GHz 三个信道 × ~30 ms ≈ 90 ms；5 GHz 可达 20+ 信道，耗时更长。40 ms – 300+ ms（取决于频段与信道数量）
③断开（Disassociation / Deauthentication）客户端向 AP-old 发送 Disassociation 或 Deauthentication 帧，正式断开连接。此时数据帧无法收发。< 5 ms（帧交换）
④认证（Authentication）客户端向 AP-new 发送 Authentication Request；在 Open System 模式下仅需两帧交互（Auth Request → Auth Response）。若使用 WPA2-Enterprise（802.1X），则需完整 EAP 交互，可能涉及 RADIUS 往返。Open: 2-5 ms
802.1X: 50-700+ ms
⑤关联（Association / Reassociation）客户端向 AP-new 发送 (Re)Association Request，AP-new 回复 (Re)Association Response。若使用 WPA2-PSK / WPA3-SAE，还需完成 4-Way Handshake 以建立 PTK/GTK。Association: 2-5 ms
4-Way Handshake: 10-25 ms
⑥网络层恢复如果漫游前后子网不同（L3 漫游），客户端还需要获取新 IP 或通过隧道维持原 IP。即使在同子网漫游（L2 漫游），交换机的 MAC 表也需要更新，ARP 缓存可能过期。L2: ~0 ms
L3: 500 ms – 数秒（DHCP）

将上述阶段累加，一次传统的 Wi-Fi 漫游通常需要 50 ms（最理想条件：Open Auth、同信道、FT）到 1-2 秒（复杂环境：802.1X 全握手、多信道扫描、L3 漫游）不等。对于网页浏览或电子邮件等应用而言，偶尔几百毫秒的中断是可接受的；但对于工业控制信号（周期 ≤ 10 ms）、高清视频流（中断 >100 ms 即出现画面冻结）或列车信号系统（安全容限极小）来说，这是完全不可接受的。

5.1.2 802.11 快速漫游优化（802.11r/k/v）

为了缩短漫游时间，IEEE 和 Wi-Fi 联盟先后推出了多项优化标准：

标准名称核心机制改善点
802.11r快速 BSS 转换（FT）引入 Mobility Domain 概念和 FT 密钥层次结构（PMK-R0 / PMK-R1），允许客户端在 Reassociation 帧中直接携带密钥协商信息，跳过完整 4-Way Handshake。支持 Over-the-Air（OTA）和 Over-the-DS（ODS）两种模式。将认证 + 密钥协商从 50-700 ms 压缩到 < 50 ms
802.11k无线资源管理（RRM）AP 通过 Neighbor Report 向客户端提供周边 AP 的信道和 BSSID 信息，客户端无需全信道扫描，只需扫描报告中列出的候选信道。将扫描时间从 100-300+ ms 缩减到 < 40 ms
802.11v无线网络管理（WNM）AP 可通过 BSS Transition Management Request 帧主动建议客户端切换到特定 AP，实现网络侧驱动的漫游决策。解决客户端"粘滞"（Sticky Client）问题
OKC / PMK 缓存（非标准，厂商扩展）客户端首次完成 802.1X 全认证后，WLC/AP 将 PMK 分发到同域内其他 AP，后续漫游时客户端只需通过 PMKID 匹配即可跳过 EAP 认证。消除 EAP 交互，节省 50-700 ms

即使同时启用 802.11r + 802.11k + 802.11v + OKC，在理想条件下（同一 WLC 域内、客户端完整支持所有标准），最优漫游时间仍在 20-50 ms 量级。这已经是标准 802.11 协议栈能做到的极限了。但这里存在几个根本性的限制：

传统快速漫游的根本性限制
  1. Break-Before-Make（BBM）本质不变：无论如何优化，802.11 漫游仍然是"先断后连"——客户端必须先从 AP-old 断开（或至少停止在该链路上收发数据），再与 AP-new 建立连接。在"断开"和"连接完成"之间，必然存在一个无法传输用户数据的间隙。
  2. 客户端依赖：漫游决策（何时触发、选择哪个 AP）高度依赖客户端的驱动程序实现。不同厂商、不同操作系统、不同驱动版本的漫游行为差异极大，网络管理员几乎无法控制。802.11v 虽允许 AP "建议"漫游，但客户端可以忽略此建议。
  3. 单射频限制：绝大多数 Wi-Fi 客户端在任一时刻只有一个射频处于已关联状态。即使设备内部有多个射频，802.11 协议栈也是按"每个 STA"独立漫游的。
  4. 管理帧开销：即使最快的 FT Over-the-Air 漫游，也至少需要交换 Reassociation Request / Response 两帧管理帧。考虑到管理帧通常以较低速率（如 24 Mbps）发送，信道竞争（CSMA/CA 退避）也会引入额外延迟。

正因为这些根本性限制，传统 Wi-Fi 快速漫游虽然能满足企业办公、仓储等对延迟不太敏感的场景，但对于工业移动回传场景——特别是列车以 350 km/h 时速通过基站间切换区域、矿用卡车在巷道拐角处切换、港口 ASC 在轨道上往复运动——仍然力不从心。这正是 URWB Make-Before-Break 技术诞生的背景。

5.2 URWB Make-Before-Break（MBB）原理

5.2.1 核心思想：先建后断

URWB 的 Make-Before-Break 漫游策略，从名称上就揭示了其核心思想——"先建（Make）新连接，再断（Break）旧连接"。与传统 Wi-Fi 的 BBM 模式截然相反，MBB 确保在任何时刻，移动设备至少有一条活跃的数据通路，从而实现零丢包或接近零丢包的无缝漫游。

要理解 MBB 的可行性，首先需要认识到 URWB 与传统 Wi-Fi 在架构上的关键差异：

MBB 的架构前提
  • 多射频硬件：URWB 移动设备（如 IW9165E 配置为 Mobility Client / Vehicle 角色）通常装备两个或以上的 5 GHz 射频模块。其中一个射频维持与当前基站（MB / Base）的活跃数据链路，另一个射频可用于扫描并预连接到下一个基站。
  • 网络侧控制：漫游决策不是由移动客户端自主做出，而是由网络侧（Coordinator / Mesh End）基于全局拓扑信息、各链路的 RSSI/SNR/PER 指标综合决策。这消除了传统 Wi-Fi 中客户端驱动漫游带来的不确定性。
  • 专有协议栈：URWB 使用 Prodigy 协议栈替代标准 802.11 MAC，因此不受 802.11 状态机（Unauthenticated → Authenticated → Associated）的约束，可以在协议层面自由实现 MBB 逻辑。
  • Overlay 隧道：URWB 的 MPLS overlay 隧道可以在 handoff 前就预建立到新基站的路径，在切换瞬间只需将数据流从旧隧道切到新隧道，无需重建 IP 连接。

MBB 漫游阶段详解

阶段名称详细描述数据中断？
① 稳态通信 AGV 通过 Radio-0 与 MB-A 建立了稳定的 Prodigy 无线链路。用户数据通过 MPLS overlay 隧道经由 MB-A → 有线网络 → Coordinator 转发到目的地。Radio-1 处于周期性扫描状态，监听环境中其他基站的 Beacon 或 Pilot 信号。 否 ✓
② 候选发现
（Neighbor Discovery）		 Radio-1 检测到 MB-B 的信号，并将 MB-B 的信号质量指标（RSSI、SNR、PER 估计等）上报给 Coordinator。由于 Radio-1 是独立射频，其扫描行为完全不影响 Radio-0 上正在进行的数据传输。这是与传统 Wi-Fi 最关键的区别——传统客户端扫描时必须离开当前信道，导致数据传输暂停。 否 ✓
③ 预连接
（Pre-Association）		 当 Radio-1 上报的 MB-B 信号满足一定阈值条件时，Coordinator 指示 AGV 的 Radio-1 与 MB-B 建立 Prodigy 层面的连接（不是标准 802.11 的 Association，而是 Prodigy 协议内的链路建立过程）。由于使用 Network Key（PSK）认证模式，密钥交换可在极短时间内完成。此时 AGV 同时拥有两条活跃链路：Radio-0 ↔ MB-A 和 Radio-1 ↔ MB-B。 否 ✓
④ 双路并行
（Dual-Path）		 在预连接成功后，Coordinator 会在 MPLS overlay 层面预计算经由 MB-B 到达 AGV 的新路径。此时系统处于双路状态——可以同时通过两条路径转发数据。根据 Fluidity 算法的实现，系统可能会短暂地通过两条路径同时发送数据（类似 1+1 冗余），也可能仅准备好新路径等待切换指令。这一阶段的持续时间取决于 AGV 在两个基站重叠覆盖区域内行驶的时间。 否 ✓
⑤ 隧道切换
（Overlay Handoff）		 当 Coordinator 判定切换条件成熟（例如 MB-B 的链路质量明显优于 MB-A，或 MB-A 的信号已接近下限），它会执行 overlay 层面的路径切换——将 AGV 的 MPLS 标签从指向 MB-A 的 LSP 切换到指向 MB-B 的 LSP。由于新路径已经在阶段④中预建立，这个切换操作仅涉及转发表的原子更新，耗时在微秒到毫秒级。由于旧路径在切换瞬间仍然存在，即使切换期间有数据帧正在传输中，也不会丢失。 否 ✓
⑥ 旧链路释放
（Old Link Teardown）		 切换完成后，Radio-0 与 MB-A 的链路不再承载用户数据。Coordinator 在确认所有流量都已迁移到新路径后，指示 Radio-0 断开与 MB-A 的连接。Radio-0 随即进入扫描状态，为下一次漫游做准备。此时角色翻转：Radio-1 变为活跃数据射频，Radio-0 变为扫描射频。 否 ✓

从上表可以看出，在整个漫游过程中，没有任何一个阶段出现数据传输中断。这就是 Make-Before-Break 的精髓——通过时间上的"重叠"（新连接在旧连接断开之前就已建立并可用），消除了传统 BBM 漫游中不可避免的中断窗口。

5.3 双射频架构的关键作用

Make-Before-Break 之所以能实现"先建后断"，其最根本的硬件前提是移动设备必须拥有至少两个可独立操作的射频模块。这一要求与传统 Wi-Fi 形成了鲜明对比。

5.3.1 为什么传统 Wi-Fi 难以实现 MBB？

传统 Wi-Fi 客户端（笔记本电脑、手机、IoT 传感器等）通常只有一个 WLAN 射频接口。即使某些设备拥有多个天线（MIMO），这些天线也是服务于同一个射频链路的空间流复用，而非独立的射频模块。这意味着：

  • 当客户端需要扫描其他信道时，必须暂时离开当前工作信道，导致数据传输中断（即所谓的"off-channel scanning"）。
  • 客户端在任一时刻只能与一个 AP 保持关联状态。要建立新关联，必须先断开旧关联。
  • 即使少数高端设备拥有两个物理 Wi-Fi 适配器（如某些专用工业网关），标准 802.11 协议栈也没有原生的机制来协调两个适配器之间的无缝切换。

5.3.3 射频角色翻转（Radio Role Swap）

在 URWB 的 MBB 机制中，两个射频的角色并非固定不变，而是在每次漫游后动态翻转：

切换完成释放 MB-A → 扫描模式活跃数据（连接 MB-B，CH 149）

这种乒乓式角色翻转机制确保了：

  • 在任何时刻都有一个射频承载用户数据，另一个射频用于扫描和预连接。
  • 无论 AGV 经过多少次基站切换，MBB 机制始终可以工作，不会出现"两个射频都在扫描"或"两个射频都在传数据但没有射频可以扫描"的死锁状态。
  • 射频角色的翻转由 Prodigy 协议引擎统一管理，对上层应用完全透明。

5.3.4 信道规划与 MBB 的关系

MBB 的有效运作对信道规划有特殊要求：

MBB 对信道规划的要求
  • 相邻基站应使用不同信道：如果 MB-A 和 MB-B 使用相同信道，则 Radio-1 在扫描/预连接到 MB-B 时可能与 Radio-0 在 MB-A 上的数据传输产生同频干扰。理想情况下，相邻基站应分配不同的 5 GHz 信道（例如 MB-A 用 CH 36，MB-B 用 CH 149）。
  • 至少需要两个不重叠信道：在最简单的场景中，可以使用两个 80 MHz 信道交替分配给奇数和偶数编号的基站（类似蜂窝网络的频率复用）。例如：MB-1(CH36) → MB-2(CH149) → MB-3(CH36) → MB-4(CH149)。
  • 三信道规划更优：使用三个不重叠信道（如 CH36、CH52、CH149）可以提供更大的信道间隔，降低同频干扰风险。这类似于蜂窝网络中的三扇区频率复用。
  • 80 MHz vs 40 MHz vs 20 MHz：信道带宽越宽，吞吐量越高，但可用不重叠信道数越少。80 MHz 在 5 GHz UNII-1 + UNII-3 频段大约有 2-4 个不重叠信道（取决于国家的频谱政策）；40 MHz 有更多选择。工程师需要在吞吐量和频率复用灵活性之间权衡。
  • DFS 信道注意事项：在某些国家，5 GHz 的部分信道属于 DFS（动态频率选择）范围，需要在检测到雷达信号时自动切换。在工业环境中，如果依赖 DFS 信道作为 MBB 的备用路径，可能面临 DFS 事件导致链路不可用的风险。建议在规划时优先使用非 DFS 信道（UNII-1 和 UNII-3）。

总结来说，双射频架构是 MBB 的硬件基础，而合理的信道规划是 MBB 发挥最优效果的射频保障。两者缺一不可。

5.4.1 延迟抖动（Jitter）对比

除了绝对的中断时间外，延迟抖动也是评估漫游质量的重要指标。在工业控制场景中，周期性控制报文（如 PLC 扫描周期 10 ms）不仅要求低延迟，还要求延迟稳定可预测。

  • 传统 Wi-Fi 漫游：在漫游前后，端到端延迟可能从 5 ms 突变到 500 ms 甚至出现丢包超时，形成巨大的抖动尖峰。对于 TCP 连接，这可能触发重传超时（RTO），导致吞吐量骤降；对于实时控制，这可能触发安全停机。
  • URWB MBB：由于数据流从未中断，漫游前后的延迟变化非常小。Overlay 隧道切换导致的延迟增加通常在 1 ms 以内，且是一次性的短暂抖动，不会形成持续的延迟劣化。这使得上层应用（PLC 控制循环、视频流、VoIP）几乎无法感知到漫游的发生。

5.5 网络侧驱动的漫游决策

URWB MBB 的另一个关键创新在于漫游决策由网络侧（Coordinator / Mesh End）统一做出，而非像传统 Wi-Fi 那样依赖客户端自主决策。

5.5.1 传统 Wi-Fi 的客户端驱动漫游问题

在传统 Wi-Fi 中，漫游决策完全由客户端执行。客户端的驱动程序根据内置算法（通常是供应商私有的、不可配置的）来判断何时触发漫游、选择哪个 AP。这带来了多个问题：

  • 行为不可预测：不同厂商、不同型号、不同驱动版本的客户端漫游行为差异极大。同一环境下，Apple 设备可能在 -70 dBm 触发漫游，Intel 网卡可能在 -75 dBm，Qualcomm 网卡可能在 -65 dBm。
  • "粘滞客户端"（Sticky Client）：某些客户端即使当前 AP 信号已经很差，仍然不愿切换到信号更好的 AP，导致性能严重下降。802.11v 的 BSS Transition Management 尝试解决此问题，但客户端可以选择忽略 AP 的建议。
  • 无法考虑全局负载：客户端只能看到各 AP 的信号强度，无法了解目标 AP 的当前负载、信道利用率、有线侧带宽等信息。可能盲目切换到一个信号强但已过载的 AP。
  • 网络管理员无法干预：即使网络管理员知道最优的漫游策略，也无法强制客户端执行，只能通过间接手段（调整 AP 功率、启用 802.11v）来引导客户端行为。

5.5.2 URWB 的网络侧决策机制

在 URWB Mobility / Fluidity 拓扑中，漫游决策由 Coordinator（Mesh End）基于全局信息做出：

  1. 可预测性：漫游行为完全由网络侧控制，工程师可以精确配置漫游阈值和策略，确保在所有设备上获得一致的行为。
  2. 全局优化：Coordinator 可以在做出漫游决策时考虑整个网络的状态，避免将多个移动设备同时切换到同一个基站导致过载。
  3. 主动式漫游：Coordinator 可以预判移动设备的运动趋势（基于信号变化趋势），在信号质量下降到临界值之前就提前触发 MBB 流程，确保始终在最优时机完成切换。
  4. 故障感知：如果某个基站发生故障或有线链路中断，Coordinator 可以立即触发所有受影响移动设备的漫游，而不是等待移动设备自己检测到问题。这一特性与 TITAN（Fastfail）协议紧密配合。

5.5.3 Handoff 角色的配合

在第三章中我们提到了 Handoff 这一特殊角色。在 MBB 漫游过程中，Handoff 射频起着关键的"信令通道"作用。Handoff 射频通常配置在 Mobility Base 上，用于在 Prodigy 控制平面上与移动设备的扫描射频交换漫游相关的控制消息。这些控制消息包括：

  • 邻居通告（Neighbor Advertisement）：基站定期广播自身的身份和链路参数，供移动设备的扫描射频接收。
  • 预认证请求/响应：在移动设备正式切换到新基站之前，通过 Handoff 信道完成密钥交换和身份验证。
  • 切换命令：Coordinator 通过 Handoff 信道向移动设备下发切换指令，包括新基站的标识和信道信息。

Handoff 信道的存在使得 MBB 的控制信令可以在不占用数据信道带宽的情况下完成，进一步保障了数据传输的连续性。

5.6 MBB 与 Overlay 隧道迁移的协同

MBB 不仅仅是射频层面的无缝切换，还涉及数据转发路径（Overlay 隧道）的无缝迁移。在第六章中我们将详细讨论 URWB 的 Overlay/Underlay 架构，但这里有必要先简述 MBB 与 Overlay 隧道迁移的协同关系。

5.6.1 Overlay 隧道在漫游中的角色

在 URWB 网络中，用户数据不是直接通过无线链路"裸传"的，而是封装在 MPLS overlay 隧道中。每个移动设备到 Coordinator 之间有一条 LSP（Label Switched Path），数据帧在网络中的转发基于 MPLS 标签而非 IP 地址或 MAC 地址。

当 AGV 从 MB-A 漫游到 MB-B 时，不仅无线射频链路需要切换，MPLS LSP 也需要从"经由 MB-A"更新为"经由 MB-B"。如果这个 LSP 更新不够快速或不够原子化，即使射频层面实现了 MBB，用户数据仍然可能因为 overlay 路径断裂而丢失。

5.6.2 预建立新 LSP

URWB 的 MBB 流程将 overlay 路径的预建立作为关键步骤（前述阶段④）。在移动设备的扫描射频成功预连接到 MB-B 之后，Coordinator 会：

  1. 计算从自身到 MB-B 再到移动设备的新 LSP。
  2. 沿新路径的每个节点预写入 MPLS 转发表条目。
  3. 但不激活新 LSP——旧 LSP（经由 MB-A）仍然是活跃路径。

5.6.3 原子路径切换

当 Coordinator 下达切换命令时，切换操作在 MPLS 层面是原子化的：

  • Coordinator 同时在自身和相关中间节点上执行标签替换操作。
  • 下行数据（从有线网络到移动设备）：Coordinator 将进入标签从 MB-A 对应的标签替换为 MB-B 对应的标签。
  • 上行数据（从移动设备到有线网络）：移动设备将数据帧通过新活跃射频发送到 MB-B，MB-B 使用已预配置的标签将帧转发到 Coordinator。
  • 旧 LSP（经由 MB-A）在标签替换完成后被标记为过期，短暂保留后删除。

由于新 LSP 在切换前就已完全建立（所有中间节点的转发表都已就绪），切换操作仅涉及少量节点的标签更新，耗时极短。

5.6.4 下行数据的处理

一个特别需要关注的问题是：在切换的瞬间，如果有下行数据帧正在从 Coordinator 经由 MB-A 向移动设备传输，这些"在途"帧会如何处理？

在途帧处理机制

URWB 的 Fluidity 引擎采用了一种短暂双投递（brief dual-delivery）或数据面屏障（data plane barrier）机制来处理在途帧：

  • 在执行标签切换之前，Coordinator 会短暂地将下行流量同时通过旧 LSP 和新 LSP 双发。
  • 移动设备在两个射频上都可以接收数据帧，并通过序列号去重，确保上层只收到一份数据。
  • 双发窗口通常只有几毫秒，之后 Coordinator 停止通过旧 LSP 发送，完全切换到新 LSP。

这种机制确保了即使在切换瞬间有"在途"帧，也不会被丢弃。代价是短暂的带宽双倍使用和接收端的去重开销，但相对于零丢包的收益而言，这个代价微不足道。

5.7 特殊场景下的 MBB 行为

5.7.1 U 型转弯 / 往复运动

在某些工业场景中（如港口 ASC 在轨道两端往复运动、矿井卡车在调度区掉头），移动设备可能在短时间内多次在两个基站之间来回切换。URWB 的 MBB 机制通过以下方式处理这种情况：

  • 漫游迟滞（Hysteresis）：Coordinator 在漫游决策中内置了信号强度迟滞机制。只有当新基站的信号优于旧基站超过一定阈值（如 6 dB）时才触发切换。这避免了在信号接近的区域频繁"乒乓漫游"（Ping-Pong Handoff）。
  • 回切优化（Handback Optimization）：如果移动设备刚从 MB-A 切换到 MB-B，然后立即检测到 MB-A 的信号又变好了（例如掉头），系统可以快速执行反向 MBB，因为 MB-A 的连接信息仍在缓存中，无需重新扫描和全新认证。
  • 扫描射频的选择性：在刚完成一次漫游后，系统会让扫描射频优先监听刚释放的基站，以便在回切场景中快速重新连接。

5.7.2 多移动设备同时漫游

在大型工业场景（如港口同时运行 30+ 辆 AGV）中，可能出现多辆 AGV 同时穿过基站切换区域的情况。Coordinator 作为集中式决策点，可以：

  • 串行化切换：将同时到达切换区域的多辆 AGV 的切换命令错开几毫秒执行，避免基站和网络层面的瞬时过载。
  • 负载均衡：如果一个基站覆盖区域内有两个可选的 MB（例如双 MB 冗余部署），Coordinator 可以将部分 AGV 切换到 MB-B1，另一部分切换到 MB-B2，实现负载均衡。
  • 优先级区分：对于运载关键物资或处于紧急操作中的 AGV，可以优先执行其漫游切换。

5.7.3 基站故障触发的强制漫游

如果一个 Mobility Base 发生故障（如掉电、有线链路中断），连接到该 MB 的所有移动设备需要紧急切换到其他可用 MB。这种情况下的漫游不同于正常的 MBB 流程：

  • TITAN（Fastfail）协议启动：Coordinator 在检测到 MB 故障后（通常在 <500 ms 内通过 TITAN 心跳超时检测），立即向所有受影响的移动设备下发紧急切换命令。
  • 扫描射频加速连接：如果移动设备的扫描射频恰好已经预连接到备用 MB，则可以立即完成 MBB 切换。
  • 回退到 BBM：如果扫描射频尚未找到可用的备用 MB，移动设备可能需要进行一次快速扫描，此时会出现短暂的数据中断。但即使在这种最坏情况下，URWB 的 Prodigy 协议也比标准 802.11 的恢复速度快得多（通常在 100-500 ms 内恢复）。
设计建议：冗余覆盖消除 BBM 回退

为了确保即使在基站故障时也能实现 MBB（而非 BBM 回退），建议在关键区域部署冗余 Mobility Base——即在任何位置，移动设备的扫描射频至少能探测到一个备用 MB。这通常意味着将基站间距设计为覆盖半径的 1.5-2 倍重叠，而非刚好相切。虽然增加了基站数量和成本，但对于安全关键的工业应用（如矿井自动驾驶、港口 ASC 自动化），这一投资是值得的。

5.7.4 L3 Fluidity 模式下的 MBB

在第三章和第四章中提到，URWB 支持 L2 Fluidity 和 L3 Fluidity 两种模式。在 L3 Fluidity 模式下，移动设备与不同基站之间可能跨越 IP 子网。即便如此，MBB 机制仍然有效，因为：

  • MPLS overlay 隧道独立于 IP 子网。即使 MB-A 的管理接口在 10.0.1.0/24，MB-B 在 10.0.2.0/24，overlay 隧道可以跨越这些子网边界。
  • 移动设备的 overlay IP 地址不随漫游改变——它是分配在 overlay 隧道内的虚拟 IP，与 underlay 的物理 IP 编址无关。
  • Coordinator 在执行 LSP 切换时会同时更新 L3 路由信息，确保上游路由器知道如何通过新路径到达移动设备。

因此，在 L3 Fluidity 模式下，MBB 不仅消除了射频层面的中断，还消除了传统 L3 漫游中不可避免的 IP 层中断（如 DHCP 重获地址、ARP 更新等）。这对于跨大面积区域部署的工业网络（如矿山、大型港口）尤为重要。

5.8 MBB 的局限性与工程注意事项

尽管 URWB 的 MBB 机制在性能上远优于传统 Wi-Fi 漫游，但它并非没有局限性和需要注意的工程约束：

MBB 的局限性与应对建议

局限性详细说明应对建议
硬件成本 MBB 要求移动设备具备至少两个独立 5 GHz 射频，增加了硬件复杂度和成本。IW9165E 的单台成本远高于普通 Wi-Fi 客户端。 在成本敏感场景中评估是否真正需要 MBB；对于低速、允许短暂中断的场景，可考虑使用更经济的方案。

重叠覆盖区域设计 MBB 的有效性依赖于相邻基站之间有足够的重叠覆盖区域。如果重叠区太短（例如因为地形遮挡导致信号急剧衰减），扫描射频可能没有足够时间完成预连接。 在射频勘测（RF Survey）阶段仔细规划重叠覆盖区域；在信号衰减快的场景（如矿井弯道）增加中继基站。
Coordinator 单点 MBB 的集中式决策依赖于 Coordinator 的可用性。如果 Coordinator 故障，新的漫游决策可能无法及时做出。 在 Catalyst 9800 WLC 托管模式下，WLC 可提供 HA（高可用）对来消除控制平面单点故障。在 Standalone 模式下，需要在网络设计中考虑 Coordinator 的冗余。

5.9 本章小结

第五章核心要点
  1. 传统 Wi-Fi 漫游是 Break-Before-Make（先断后连）：无论如何优化（802.11r/k/v），都存在不可避免的数据中断窗口，典型时间 20-50 ms（FT 优化后）到 1-2 秒（无优化）。
  2. URWB 使用 Make-Before-Break（先建后断）：在新连接完全建立并可用之后才断开旧连接，实现理论上零中断的漫游。
  3. MBB 的硬件基础是双射频架构：一个射频传输数据，另一个射频扫描和预连接，两者角色在每次漫游后动态翻转。
  4. MBB 的控制核心是网络侧决策：由 Coordinator 基于全局信息（链路质量、负载、拓扑、MPLS 路径）统一决策，消除了客户端驱动漫游的不确定性。
  5. MBB 与 MPLS Overlay 协同工作：隧道路径的预建立和原子切换确保了不仅射频层面无缝，数据转发层面也无缝。
  6. MBB 支持高速场景：已在 350+ km/h 的铁路场景中得到验证，远超传统 Wi-Fi 的能力范围。
  7. 工程实施需要注意：信道规划（至少两个不重叠信道）、重叠覆盖区域设计、Coordinator 冗余等因素直接影响 MBB 的效果。

Make-Before-Break 是 URWB 技术最核心的差异化能力之一，也是使其能够在铁路、矿山、港口等高速移动场景中替代传统有线回传的关键技术。在下一章中，我们将深入讨论 MBB 所依赖的底层架构——Underlay（无线链路层）与 Overlay（MPLS 隧道层）的协同工作机制，以及一个完整数据帧是如何从车载终端设备经过 URWB 网络到达有线网络的。

第六章   Underlay 与 Overlay 架构深度解析

在前面的章节中，我们已经了解到 URWB 使用专有的 Prodigy 协议栈取代了标准 Wi-Fi 的 MAC 层，并通过 Make-Before-Break 实现了无中断漫游。但一个核心问题始终悬而未决：当数据帧进入 URWB 无线域后，到底经历了怎样的封装、转发和解封装过程？本章将对 URWB 的"双层架构"——Underlay（底层无线承载层）与 Overlay（上层叠加转发层）——进行逐字节级别的深度拆解，并在最后以一个完整的 AGV（自动导引车）数据包从应用程序到服务器的端到端旅程作为综合示例，将所有概念串联起来。

6.1   为什么需要"双层架构"？

在传统的有线以太网中，交换机之间使用 STP/RSTP 来避免环路，使用 VLAN Trunk 来承载多业务。但在无线 Mesh 或移动场景下，这套机制遇到了严峻挑战：

表 6-1 ｜ 有线交换 vs. 无线移动场景的核心矛盾
挑战维度有线以太网做法无线移动场景的问题
环路避免STP 阻塞冗余端口Mesh 拓扑天然存在多条路径，STP 会阻断大量链路，浪费宝贵的无线带宽
收敛速度RSTP 收敛约 1-2 秒移动节点漫游时需 <10 ms 收敛，STP 根本无法满足

漫游连续性N/A（设备不移动）终端从一个 AP 移动到另一个 AP 时，MAC 表需要即时更新，帧不能丢失

面对上述所有矛盾，URWB 的设计者（源自 Fluidmesh Networks 的核心团队）借鉴了电信运营商网络中经过大规模验证的 MPLS（Multi-Protocol Label Switching，多协议标签交换） 技术思路，将 URWB 的数据转发平面分成了两个独立但紧密协作的层次：

🔑 核心设计思想：关注点分离（Separation of Concerns）

  • Underlay 层：专注于"如何把比特从 A 点可靠地搬到 B 点"——负责无线射频、调制解调、信道接入、链路可靠性。
  • Overlay 层：专注于"这个帧应该走哪条路径、属于哪个业务、到了目的地后交给谁"——负责标签交换、路径计算、业务隔离、漫游连续性。

这种分离使得两层可以独立演进优化：Underlay 可以升级射频技术（Wi-Fi 5 → Wi-Fi 6 → Wi-Fi 7）而不影响转发逻辑，Overlay 可以增加新业务功能而不需要改动无线驱动。

📖 类比理解
可以把 Underlay 想象成高速公路的路面和车道线——它决定了车能跑多快、几条车道、路面是否平整。Overlay 则是高速公路上的导航系统和 ETC 收费系统——它决定了车从哪个入口上、走哪条路线、在哪个出口下、属于哪个运输公司。路面可以从沥青升级为混凝土（Underlay 升级），而导航系统可以从 GPS 升级为北斗（Overlay 升级），两者互不干扰。

6.2.1   Underlay 层的定义与边界

URWB 的 Underlay 层指的是从射频天线到 Prodigy MAC 帧交付之间的所有处理，具体包括：

6.2.2   Underlay 层的 PHY 合规性与互操作性影响

这是一个经常被提问的关键点（对应问题 Q10 的补充）：URWB 的 PHY 层是否符合 802.11 标准？它能否与标准 Wi-Fi 设备互操作？

✅ PHY 层：完全合规
URWB 设备使用的射频芯片组与标准 Wi-Fi AP 完全相同（基于 Qualcomm/Atheros 平台）。它们发射的无线信号在物理层面完全符合 IEEE 802.11ac/ax/be 标准，包括：调制方式（OFDM/OFDMA）、信道带宽（20/40/80/160 MHz）、空间流数量（最多 4×4 MIMO）、MCS 索引范围、帧前导码格式（L-STF、L-LTF、L-SIG、HE-SIG 等）。因此，URWB 设备可以正常通过各国的无线电管理认证（FCC、CE、TELEC 等），并且使用与 Wi-Fi 相同的免许可频谱（2.4 GHz、5 GHz、6 GHz）。
⚠️ MAC 层：不互操作
虽然 PHY 层完全兼容，但由于 Prodigy MAC 替换了标准的 802.11 MAC 层，URWB 设备无法与标准 Wi-Fi 客户端或 AP 建立关联（Association）。这意味着：
  • 普通的笔记本电脑、手机、IoT 传感器不能直接连接到 URWB 无线网络。
  • URWB 设备不能作为标准 Wi-Fi AP 使用（CW9178I 例外，它可以同时运行 URWB 射频和标准 Wi-Fi 射频）。
  • 频谱分析工具可以看到 URWB 的无线信号（因为 PHY 合规），但无法解码 MAC 层内容（因为是 Prodigy 专有格式）。
  • URWB 网络与同频段的标准 Wi-Fi 网络之间存在射频干扰关系（因为共享相同频谱），需要在信道规划时统一考虑。

因此，URWB 在网络设计中的定位是有线以太网的无线延伸——它连接的不是终端用户设备，而是以太网端口。各种终端设备（PLC、摄像头、Wi-Fi AP、工业 PC）通过有线以太网连接到 URWB 设备的 PoE 端口，然后 URWB 负责将这些以太网帧透明地"搬运"到网络的另一端。

6.2.3   Underlay 层的链路质量度量

Underlay 层持续监测每条无线链路的质量，并向 Overlay 层提供以下关键指标：

📊 链路质量综合评分模型

  • RSSI（接收信号强度指示）：以 dBm 为单位，反映信号的绝对功率水平。
  • SNR（信噪比）：以 dB 为单位，反映信号相对于背景噪声的质量。在工业环境中，SNR 比 RSSI 更能反映真实链路质量，因为工业设备产生的电磁噪声会降低 SNR 而不一定影响 RSSI。
  • PER（误包率）：实际传输中的丢包比例，是最终极的链路质量指标。
  • 可用带宽估算：基于当前 MCS 速率、信道利用率和重传率，估算链路可用吞吐量。
  • 链路稳定性：信号质量的变化方差，高速移动场景中此指标尤为重要。

Overlay 层的路径计算引擎会综合以上所有指标，使用加权算法为每条链路计算出一个 Metric（度量值），然后基于 Metric 运行修改版的链路状态路由算法（类似 OSPF 的 SPF 算法），为每对源-目的节点计算最短路径。

6.3   Overlay 层详解：MPLS 标签交换转发平面

6.3.1   为什么选择 MPLS？

在众多可选的 Overlay 技术中（VXLAN、GRE、MPLS、LISP、Geneve 等），URWB 选择了 MPLS 作为其 Overlay 转发机制。这个选择背后有深刻的技术考量：

表 6-3 ｜ MPLS vs. 其他 Overlay 技术对比（针对无线移动场景）
技术封装开销路径切换速度标签/隧道机制适用性评估
MPLS4 字节/标签极快（预建立 LSP，原子切换）短标签（20 bit），硬件可快速查找✅ 最优选择：开销最小、切换最快、天然支持多标签栈实现业务隔离
VXLAN50 字节较慢（需更新 VTEP 映射）24-bit VNI，需 UDP 封装❌ 封装开销过大，无线带宽宝贵；收敛速度不满足要求
GRE24-38 字节中等无标签概念，需外层 IP 路由❌ 缺乏细粒度路径控制能力

MPLS 的核心优势可以归纳为以下几点：

  1. 极小的封装开销：每个 MPLS 标签仅 4 字节（32 bit）。即使使用双标签栈（传输标签 + 业务标签），总开销也仅 8 字节。相比 VXLAN 的 50 字节，在无线带宽受限的环境中，这个差距意义重大。
  2. 预建立路径（LSP）：MPLS 的 Label Switched Path 可以在数据传输之前就预先建立好，当发生漫游时，只需将数据流从旧 LSP 原子地切换到新 LSP，无需重新建立隧道或更新路由表。这正是 MBB 能够实现零丢包的关键所在。
  3. 标签栈实现业务隔离：通过堆叠多个标签，一个 MPLS 标签可以标识传输路径（哪些节点需要转发），另一个标签可以标识业务类型（属于哪个 VLAN / 哪个 VPN），从而在不引入 STP 的情况下实现多业务隔离。
  4. 标签交换速度极快：MPLS 转发不需要查找 IP 路由表（最长前缀匹配），只需要查找标签表（精确匹配），处理速度更快，延迟更低。
  5. 天然支持多路径：可以为同一对源-目的同时维护多条 LSP，实现快速故障切换（Fast Reroute）。

6.3.2   MPLS 标签格式与封装结构

为了完整理解 URWB 的数据封装，我们需要首先了解 MPLS 标签的精确格式：

各字段含义详解：

  • Label（20 bits）：标签值，范围 0~1,048,575。URWB 内部使用标签来标识两种信息：
    • 外层标签（Transport Label）：标识到达目的节点的 LSP 路径。每一跳的节点会查找本地标签表，将入标签替换为出标签（Label Swap），然后将帧转发到下一跳。
    • 内层标签（Service Label / VPN Label）：标识帧所属的业务（等同于 VLAN 映射）。目的节点收到帧后，剥离外层标签，根据内层标签将帧还原到对应的 VLAN 接口。
  • TC — Traffic Class（3 bits）：用于 QoS 标记，类似 802.1p 的 CoS 或 IP 头的 DSCP。URWB 会将入站以太网帧的 802.1p 优先级映射到 MPLS TC 字段，从而在整个无线 Mesh 中保持 QoS 策略一致。
  • S — Bottom of Stack（1 bit）：当值为 1 时，表示这是标签栈中的最后一个标签。URWB 使用双标签栈时，外层标签的 S=0，内层标签的 S=1。
  • TTL — Time to Live（8 bits）：与 IP 的 TTL 功能相同，每经过一跳递减 1，到达 0 时丢弃。防止配置错误导致的帧在网络中无限循环。

6.3.4   LSP 的建立、维护与切换

MPLS 的核心机制是 LSP（Label Switched Path，标签交换路径）的管理。在 URWB 网络中，LSP 的生命周期如下：

表 6-4 ｜ LSP 生命周期管理
阶段详细说明
1. 拓扑发现每个 URWB 节点启动后，通过 Underlay 层的邻居发现协议识别直连邻居，并通过链路状态通告（类似 OSPF 的 LSA）将自己的邻居信息和链路质量指标洪泛到整个 URWB 域。每个节点因此拥有完整的网络拓扑视图。
2. 路径计算基于完整的拓扑数据库，每个节点独立运行 SPF（Shortest Path First）算法，计算到达所有其他节点的最优路径。路径的"代价"基于 Underlay 层报告的链路 Metric 综合值（而非简单的跳数）。
3. 标签分配路径计算完成后，沿路径的每个节点为其下游邻居分配本地标签（类似 LDP — Label Distribution Protocol）。标签是本地有意义的（Local Significance），即同一个标签值在不同节点上代表不同的含义。
4. 转发表建立每个节点建立 LFIB（Label Forwarding Information Base），表项格式为：{入标签, 入接口} → {出标签, 出接口, 操作(SWAP/PUSH/POP)}。数据帧到达时，只需查找 LFIB 即可完成转发，无需 IP 路由查找。
5. 数据转发帧沿 LSP 逐跳转发。每一跳执行 Label Swap（替换标签）操作，直到倒数第二跳执行 PHP（Penultimate Hop Popping，倒数第二跳弹出外层标签），最后一跳根据内层 Service Label 将帧分发到对应的有线 VLAN 接口。
6. 持续监控Underlay 层持续监测链路状态。当检测到链路质量劣化或链路中断时，触发拓扑变更通告，所有受影响的节点重新运行 SPF 并更新 LSP，整个过程通常在 <50 ms 内完成。
7. MBB 切换当 Vehicle 节点移动并决定漫游到新的 Infrastructure 基站时（参见第五章），新 LSP 在旧 LSP 尚存的情况下预先建立（Make-Before-Break），然后在一个精确的时间点将数据流从旧 LSP 原子地切换到新 LSP。切换期间，in-flight 帧通过旧 LSP 的残余路径继续传递，不会丢失。

6.4   Overlay 层的业务功能

6.4.1   VLAN 透传与隔离

URWB 的 Overlay 层能够在无线域中透明地传递多个 VLAN，而不需要在无线节点上运行 STP。这是通过 MPLS Service Label 实现的：

  • VLAN-to-Label 映射：在 Mesh End / Coordinator 节点上配置 VLAN 到 Service Label 的映射关系。例如，VLAN 100（控制流量）映射到 Service Label 1001，VLAN 200（视频流量）映射到 Service Label 2001。
  • 透明传输：进入 URWB 域的带有 802.1Q Tag 的帧，其原始 VLAN Tag 被保留在 MPLS 封装内部。当帧从 URWB 域出去时，原始的 802.1Q Tag 被完整恢复。
  • 广播域隔离：不同 VLAN 的广播帧只会在对应的 Service Label 路径上传播，不会跨 VLAN 泛洪。

6.4.2   ARP 处理与广播优化

在传统网络中，ARP 广播是一个已知的问题——尤其在大型二层域中。在无线 Mesh 网络中，广播帧的危害被进一步放大，因为每个无线节点都需要在空口重复传输广播帧，消耗宝贵的无线带宽。URWB 的 Overlay 层采取以下策略来优化 ARP 处理：

  1. ARP 代理/缓存：Mesh End / Coordinator 节点学习其下游所有节点的 IP-MAC 映射关系，当收到 ARP Request 时，如果本地缓存中已有目标 IP 的 MAC 地址，直接在本地回复 ARP Reply，而不将 ARP Request 泛洪到无线域。
  2. 广播帧的选择性转发：对于确实需要泛洪的广播帧（如 DHCP Discover、未知 ARP Request），Overlay 层使用点到多点的 LSP 树来分发，而非简单的泛洪。这确保了每条无线链路上只传输一份广播帧，避免了广播风暴。
  3. 未知单播的处理：当目的 MAC 地址未在任何节点的 FDB（Forwarding Database）中找到时，帧会被发送到 Mesh End / Coordinator，由其在有线侧进行常规的 L2 泛洪处理，而非在无线域中泛洪。

6.4.3   AutoTap / Overlay Only 机制

在第三章中我们提到了 AutoTap（Overlay Only）角色，现在可以更深入地理解它的 Overlay 层工作原理：

🔁 AutoTap 的环路避免逻辑

考虑以下场景：一个 URWB Infrastructure 节点同时连接了有线上联和无线回传，形成了一个潜在的二层环路：

有线交换机 ←──ETH──→ Infra A ←──Radio──→ Infra B ←──ETH──→ 有线交换机
（同一个交换机或同一个 L2 域）

如果没有任何机制，一个广播帧可以从有线侧进入 Infra A → 无线传到 Infra B → 从 Infra B 的有线口出去 → 通过有线交换机回到 Infra A → 再次进入无线……形成无限循环。

AutoTap 机制的解决方式：

  • 被配置为 AutoTap 的节点，其有线口仅处理 Overlay 层的 MPLS 封装帧，不处理原始以太网帧。
  • 这意味着 AutoTap 节点的有线口"看不到"原始的广播帧，自然不会将其从有线口转发出去，从而打断了环路。
  • 但 AutoTap 节点仍然参与 MPLS Overlay 的转发——它可以作为中间跳转发 MPLS 封装的帧，只是不会在本地解封装并传递到有线侧。
  • 管理流量（如 SSH 访问 AutoTap 节点自身）仍然可以通过 Management VLAN 正常工作。

6.5   Overlay 对管理与故障排除的影响

MPLS Overlay 在带来强大转发能力的同时，也对网络管理和故障排除提出了新的挑战。以下是管理者需要了解的关键点：

6.5.1   Overlay 带来的管理优势

✅ 优势一：统一的转发平面

无论底层无线拓扑如何变化（Mesh 路径切换、Vehicle 漫游、链路故障），上层应用看到的始终是一个透明的二层网络。网络管理员不需要关心每一个无线帧走了哪条射频路径，只需要确保 Overlay 的 LSP 正常建立。这极大地降低了管理复杂度。

✅ 优势二：集中的拓扑可视化

由于每个节点都拥有完整的链路状态数据库，管理平面（Cisco IIoT Gateway Manager 或 Catalyst 9800 WLC）可以从任何节点获取完整的网络拓扑视图，包括：所有节点的角色和状态、所有无线链路的质量指标、所有 LSP 的路径和标签信息、所有 Vehicle 的当前关联状态和漫游历史。

✅ 优势三：QoS 端到端一致性

MPLS TC 字段确保了 QoS 优先级标记在整个无线域中的一致性。从入口到出口，帧的优先级不会丢失或被重新标记。这对于工业控制场景中"控制流量优先于视频流量优先于普通数据"的需求至关重要。

6.5.2   Overlay 带来的管理挑战

⚠️ 挑战一：抓包分析的复杂性

在 URWB 域内的有线镜像点抓包时，看到的是 MPLS 封装后的帧，而非原始以太网帧。管理员需要：

  • 熟悉 Wireshark 的 MPLS 解码过滤器（mpls、mpls.label、mpls.bottom）
  • 了解 URWB 的标签分配规则，才能正确地将标签值映射回业务 VLAN
  • 在 Mesh End / Coordinator 的有线上联口抓包是最佳位置，因为帧在该点会被解封装回原始格式

⚠️ 挑战二：MTU 配置的敏感性

如前所述，MPLS 封装增加了 26~30 字节的开销。如果上游交换机的 MTU 未正确配置（≥1544），会导致：

  • 大帧被静默丢弃（交换机不转发超过 MTU 的帧）
  • 表现为间歇性的连接问题——小帧（如 ARP、ICMP）正常通过，而大帧（如 TCP 数据传输、视频流）被丢弃
  • 故障表现类似于"能 ping 通但不能传文件"，常被误诊为无线链路问题

排查方法：首先检查上游交换机端口的 MTU 配置，确认 ≥ 1544；其次在交换机端口上查看计数器中是否有"Giants"或"Oversize"丢弃。

⚠️ 挑战三：标签空间管理

虽然 20-bit 标签空间理论上支持超过 100 万个标签，但在实际部署中需要注意：

  • 标签分配是本地有意义的（Local Significance），不同节点可以为同一条 LSP 使用不同的标签值
  • 标签的分配和回收是自动的，通常无需手工干预
  • 但在大规模 Mesh 网络（>50 个节点）中，需要关注标签分配协议的收敛时间

⚠️ 挑战四：与传统网络工具的兼容性

标准的网络管理工具（如简单的 SNMP 监控、NetFlow/sFlow 分析）可能无法完全理解 MPLS 封装的流量。建议使用 Cisco 提供的专用管理工具：

  • Cisco IIoT Gateway Manager（FM Gateway）：专为 URWB 设计的管理平台，可视化显示拓扑、链路质量、节点状态
  • Catalyst 9800 WLC Dashboard：当 URWB 设备注册到 9800 后，WLC 的监控页面可以显示 URWB 特有的指标
  • URWB CLI 命令：设备本地 CLI 提供 MPLS 标签表查看、LSP 追踪、链路质量统计等诊断命令

同样的场景如果使用传统 Wi-Fi AP 方案（AGV 上装 Wi-Fi 客户端 + 沿途部署企业 AP + WLC），典型的单向延迟为 3~8 ms（取决于 Wi-Fi 争用程度和 WLC 集中转发的额外延迟）。URWB 方案的 <0.4 ms 延迟比传统方案低了 一个数量级。对于 AGV 的实时控制（紧急停车、路径微调、协同避障）来说，这个差距意味着更短的反应距离（以 1.2 m/s 速度计算，8 ms 延迟对应 9.6 mm 的位置不确定性，0.4 ms 延迟仅对应 0.48 mm）。

6.7.5   下行方向（服务器 → AGV）

调度服务器向 AGV 发送控制指令（如路径指令、速度调整）的下行过程是上行过程的镜像：

  1. 调度服务器生成控制指令 → TCP/IP 封装 → 以太网帧（目的 MAC = PLC MAC, VLAN 100）
  2. 核心交换机根据 MAC 地址表将帧转发到 Mesh End 的有线上联口
  3. Mesh End 作为 Overlay 入口节点（Ingress LER）：
    • 查找 FDB：目的 MAC = PLC MAC → 下一跳 = 当前服务 Vehicle 的 Infra-B
    • PUSH Transport Label（到 Vehicle 的 LSP）+ Service Label（VLAN 100 映射）
    • 添加新以太网头（Dst MAC = Infra-B 或 Vehicle 的无线 MAC）
  4. 帧经 Underlay 层加密发送到 Infra-B → Infra-B 执行 Label Swap → 发送到 Vehicle IW9165E
  5. Vehicle IW9165E 作为 Overlay 出口节点：POP 双标签 → 还原原始以太网帧 → 从 PoE 端口发送到 PLC
  6. PLC 接收控制指令 → 执行动作（调整速度、转向等）
📖 双向对称性
上行和下行使用的是不同的 LSP（标签值不同），但路径通常是对称的（沿相同的物理链路但方向相反）。在某些 Mesh 拓扑中，上行和下行可能走不同的物理路径（如果 SPF 算法基于当前链路质量为两个方向计算出了不同的最优路径），但这对上层应用是完全透明的。

6.8 本章小结

第六章是本指南中篇幅最长、也最为核心的章节之一，我们从"为什么需要双层架构"出发，逐层拆解了 Underlay 与 Overlay 各自的职责、协议机制与工程细节。以下要点值得反复回顾：

  1. 职责分离是 URWB 架构的灵魂：Underlay 负责"把比特可靠地搬到对端"，Overlay 负责"让上层业务感知不到无线的存在"。这种分离使系统在面对射频环境剧变、节点漫游、链路故障等挑战时，仍能向应用层呈现一张稳定的以太网。
  2. Underlay 五子层模型（PHY → Prodigy MAC → 链路度量 → 邻居发现 → 路径计算）为上层提供带质量评分的逐跳隧道，其核心创新在于 Prodigy MAC 对时隙的确定性调度与对信道的快速感知。
  3. Overlay 选择 MPLS而非 GRE / VXLAN 等方案，根本原因在于标签交换的硬件亲和性与固定 4 字节开销的可预测性。双层标签栈（Transport + Service）将"到达哪个出口"与"属于哪条业务流"正交解耦。
  4. MTU 必须 ≥ 1544 字节（1500 有效载荷 + 14 以太头 + 8 MPLS 双标签 + 4 内层 VLAN + 18 Prodigy MAC 额外开销），上游有线交换机需开启 Jumbo Frame 或至少将 MTU 设为 1600。
  5. AGV 数据包端到端旅程展示了一帧数据从 PLC 以太网口到 MES 服务器所经历的 15+ 处理步骤，总延迟预算约 320-360 μs（单跳无线场景），充分体现了 URWB 对工业级实时性的支撑能力。
  6. 漫游期间的 Overlay 行为是零丢包的关键：LSP 在 MBB 的 Phase-2 就开始预建，新旧路径在 Phase-4 实现原子切换，Phase-5 的短暂双收机制兜底处理空中在途帧。

理解了 Underlay 与 Overlay 的协作之后，我们将在下一章进入 URWB 的另一项独特技术——多路径操作（MPO），它在 Overlay 层之上进一步利用多条独立射频路径实现带宽聚合与冗余保护。

第七章 · MPO 多路径操作（Multi-Path Operations）技术深度解析

7.1 为什么需要 MPO？——单路径的天花板

在前面章节中，我们已经了解到 URWB 的 Mesh / Mobility 拓扑能够建立多条无线链路，而 Overlay 的 MPLS LSP 可以沿任意一条路径转发数据。然而，在许多工业场景中，仅使用单一最优路径仍然面临以下瓶颈：

#痛点维度具体描述
1带宽上限单条 802.11ac/ax 链路的物理层速率虽然可达数百 Mbps，但受限于调制方式、信道宽度、距离衰减等因素，实际吞吐往往远低于理论值。对于高清视频回传（如港口多路 720p/1080p 摄像头）场景，单链路可能成为瓶颈。
2单点故障当唯一路径因干扰、遮挡或设备故障而中断时，即使 URWB 的快速重路由能力可以在数十毫秒内切换，仍会产生短暂中断。对于安全关键型应用（SIL-2 等级的列车通信、矿井紧急停机信号），任何中断都不可接受。
3延迟抖动无线信道的瞬时质量波动导致单路径的延迟出现毛刺（Jitter），对于运动控制、远程遥操作等对延迟一致性要求极高的应用造成影响。
4频谱效率浪费在部署了多台基站的环境中，Vehicle 设备可能同时"看到"2-3 个可用基站，但传统单路径模式下只使用其中一个，其余射频资源被白白浪费。

MPO（Multi-Path Operations，多路径操作）正是 URWB 为解决上述痛点而设计的核心技术。它允许数据同时通过多条独立的射频路径进行传输，从而实现带宽聚合、冗余保护和抖动平滑三重目标。

概念定位：MPO 工作在 URWB 协议栈的 Overlay 层与 Underlay 层之间的调度层。它不改变底层射频链路的建立方式（Underlay 负责），也不改变业务流的封装格式（Overlay 负责），而是在两者之间插入一个智能流量分配与路径冗余引擎。可以将其类比为高速公路上的"智能车道分配系统"——路本身由 Underlay 修建，车辆由 Overlay 装载，而 MPO 决定每辆车走哪条车道。

7.2 MPO 的工作原理

7.2.1 多路径建立

MPO 的基础是同时维护多条活跃路径（Active Path）。在典型的 Mobility 拓扑中，一台 Vehicle（IW9165E）拥有两个 5 GHz 射频接口（Radio-1 和 Radio-2），当它处于多个 Infrastructure 基站的覆盖重叠区时：

  1. Radio-1 与当前主基站（Primary Base）保持数据连接，承载正常业务流量。
  2. Radio-2 在非 MBB 漫游期间，通常用于扫描或作为备用。当 MPO 启用时，Radio-2 不再仅仅"待命"，而是主动与次优基站（Secondary Base）建立第二条数据链路。
  3. Coordinator 感知到 Vehicle 拥有两条活跃上行链路后，为每条链路分别建立独立的 MPLS LSP（Label Switched Path），形成两条从 Vehicle 到 Mesh End 的端到端隧道。
  4. 若环境中存在更多可用基站且设备支持，理论上可以扩展到三条甚至更多路径，但实际工程中以双路径为主，因为 IW9165E 的双射频架构天然支持两条独立路径。

7.2.2 数据分配策略

MPO 建立多条路径后，如何将数据流量分配到各路径上是核心问题。URWB 的 MPO 引擎支持以下分配模式：

分配模式英文名称工作方式适用场景
复制模式Duplication每一帧数据同时在所有活跃路径上发送完整副本。接收端通过序列号去重，仅向上层交付第一个到达的副本。安全关键型应用（紧急停机、信号控制）——追求极致可靠性，即使一条路径完全失效，另一条仍可无缝接管。带宽利用率降低 50 %，但可靠性接近 99.9999 %。
负载均衡模式Load Balancing基于路径质量（延迟、丢包率、带宽）动态将数据帧按权重分配到各路径。权重每 100-500 ms 根据实时度量更新。高带宽需求场景（多路视频回传、大文件传输）——追求吞吐量最大化。接收端通过重排序缓冲区（Reorder Buffer）恢复帧序。
主备模式Primary-Backup正常情况下所有流量走主路径（Primary），备用路径仅传输心跳探针。主路径劣化超过阈值时，流量在 <10 ms 内切换到备用路径。对延迟敏感但不需要带宽聚合的场景——追求快速故障切换而不引入重排序延迟。
工程建议：在大多数工业移动场景中，推荐使用复制模式（Duplication）作为默认选项。虽然它"浪费"了一半带宽，但对于典型的 PLC 控制数据（帧大小 64-256 字节、速率 <10 Mbps），带宽完全充足，而获得的可靠性提升是质的飞跃。只有当带宽确实成为瓶颈时（如需要回传多路 1080p 视频），才考虑切换到负载均衡模式。

7.2.3 路径质量评估

MPO 引擎持续监控每条路径的质量，以动态调整分配策略或触发路径切换。评估指标包括：

  • 往返延迟（RTT）：通过高频心跳探针（每 50-100 ms 一次）测量。若某路径 RTT 突增超过基线值的 2 倍，降低其权重。
  • 丢包率（Packet Loss Rate）：统计窗口为最近 1 秒内的发送/接收比。丢包率 >5 % 触发路径降级。
  • RSSI / SNR：来自 Underlay 层的射频质量指标，提供未来趋势预判（RSSI 持续下降 → 即将脱离覆盖 → 提前准备切换）。
  • 链路带宽（Available Bandwidth）：通过 MCS 索引和信道利用率估算。用于负载均衡模式的权重计算。
  • 路径跳数（Hop Count）：经过中继节点越多，延迟越高且稳定性越差。MPO 倾向于选择跳数少的路径。

所有指标被综合为一个复合路径分数（Composite Path Score, CPS），计算公式概念上可表示为：

CPS = w₁ × (1/RTT) + w₂ × (1 - LossRate) + w₃ × SNR_normalized + w₄ × BW_normalized − w₅ × HopCount
其中 w₁…w₅ 为可调权重因子，默认配置已针对工业移动场景优化。

7.4 MPO 与 MBB 的协同

MPO 与 Make-Before-Break 漫游机制之间存在精巧的协作关系，两者共同确保移动场景下的零中断体验：

正常运行阶段（非漫游）

Vehicle 的两个射频接口分别连接两个基站，MPO 引擎在两条路径上执行数据复制或负载均衡。此时系统处于稳态双路径状态，可靠性最高。

漫游触发阶段

当 Vehicle 移动导致其中一个基站（如 Base-A）的信号持续衰减时：

  1. MPO 引擎检测到 Path-A 的 CPS 分数下降，在复制模式下自动增加对 Path-B 的依赖权重。
  2. Coordinator 的漫游决策模块同步启动 MBB 流程，指示 Vehicle 的 Radio-1（当前连接衰减中的 Base-A）开始扫描新的候选基站 Base-C。

MBB 执行阶段

  1. Radio-1 与 Base-C 建立新连接（MBB Phase-2/3），此时 Path-B（经 Base-B）仍在正常承载所有业务流量——这就是零中断的保障。
  2. Radio-1 → Base-C 的新链路（Path-C）建立成功后，MPO 引擎将其纳入活跃路径组。
  3. 旧的 Path-A（Radio-1 → Base-A）被优雅拆除。
  4. 系统进入新的稳态：Path-B（Radio-2 → Base-B）+ Path-C（Radio-1 → Base-C）。
关键洞察：在没有 MPO 的纯 MBB 场景中，漫游期间只有一条路径承载流量（旧路径或新路径，取决于阶段），存在短暂的"单路径窗口"。而 MPO + MBB 的组合确保在漫游过程中，始终至少有一条路径不受影响，实现了真正意义上的"漫游对业务完全透明"。

7.5 MPO 与频率规划

MPO 的有效性高度依赖于路径的射频独立性。如果两条路径使用相同或相邻信道，它们会共享干扰源，降低冗余价值。因此，MPO 部署需要精心的频率规划：

规划要素建议与说明
信道间隔MPO 的两条路径应使用完全不重叠的信道。例如 Path-A 使用 UNII-1（信道 36-48），Path-B 使用 UNII-3（信道 149-165）。避免使用相邻信道（如 36 和 40，在 80 MHz 宽度下会重叠）。
基站信道分配沿移动路径交替部署的基站应采用双信道交替模式：Base-1 → 信道 36，Base-2 → 信道 149，Base-3 → 信道 36，Base-4 → 信道 149 ……确保 Vehicle 在任意位置都能同时看到两个不同信道的基站。
信道宽度权衡使用 80 MHz 信道宽度可获得更高吞吐量，但可用的不重叠信道数减少（5 GHz 频段约 2-3 个 80 MHz 信道）。如果基站密度高，可考虑使用 40 MHz 信道宽度以获得更多不重叠信道选项。
Wi-Fi 6E（6 GHz）若使用支持 6 GHz 的设备（如 CW9178I），则频谱资源大幅增加（最多 7 个 160 MHz 信道），MPO 的多路径独立性更易保障。
DFS 信道UNII-2 / UNII-2 Extended 信道受 DFS（动态频率选择）约束，雷达检测会导致信道突然切换。建议 MPO 的至少一条路径使用非 DFS 信道（UNII-1 或 UNII-3），确保"保底"路径的稳定性。

7.6 MPO 在工业场景中的可靠性提升——量化分析

以一个典型的港口自动化 AGV 场景为例，量化说明 MPO 对可靠性的提升：

场景假设

  • 单条无线路径的可用性为 99.95 %（即每年约 4.4 小时不可用，因干扰、遮挡、设备瞬时故障等）。
  • 两条路径使用不同信道、不同基站，故障完全独立（这是合理假设，因为射频环境不同）。

可靠性计算

  • 无 MPO（单路径）：可用性 = 99.95 % → 年不可用 ≈ 4.38 小时
  • MPO 复制模式（双路径）：两条路径同时失效的概率 = (1 − 0.9995)² = 0.00000025 → 可用性 = 99.999975 % → 年不可用 ≈ 7.9 秒

结论

MPO 将年不可用时间从 4.38 小时降至不到 8 秒，提升了三个数量级。这一水平已接近电信级"五个 9"（99.999 %）的要求，而且是在无线环境中实现的——这在传统 Wi-Fi 架构中是不可想象的。

前提条件：以上计算假设两条路径的故障相互独立。如果两条路径使用相同信道或共享同一基站（未正确规划），则故障会出现相关性，冗余增益将大打折扣。因此，频率规划和基站布局是 MPO 效果的决定性因素。

7.8 本章小结

MPO 是 URWB 技术栈中将无线可靠性推向工业级的关键拼图。它通过在多条射频独立路径上复制或分配数据，实现了：

  1. 极致可靠性：复制模式下，双路径独立故障场景的可用性达到 99.999975 %。
  2. 带宽聚合：负载均衡模式下，理论可用带宽翻倍。
  3. 与 MBB 无缝协同：漫游期间始终至少一条路径不受影响，消除了传统"单路径窗口"。
  4. 对应用透明：MPO 的所有操作发生在 URWB 协议栈内部，上层应用看到的仍然是一条稳定的以太网连接。

MPO 的代价是需要更精心的频率规划（确保路径射频独立性）以及在复制模式下的带宽开销。但对于工业关键应用而言，这一代价完全值得。下一章，我们将从多个维度综合分析 URWB 实现超低延迟的技术手段。

8.1 延迟的重要性——为什么工业场景对延迟如此敏感？

8.2 延迟优化的六个维度

URWB 的超低延迟并非依赖单一技术，而是在协议栈的多个层次协同优化的结果。我们从六个维度逐一分析：

8.2.1 维度一：PHY 层——802.11 物理层的天然优势与增强

URWB 采用标准 802.11ac/ax PHY 层，本身就具备以下低延迟特性：

  • OFDM 调制：高频谱效率意味着数据可以在更短的时间内传输完成。以 802.11ax、80 MHz 信道宽度、MCS 9 为例，单个 OFDM 符号持续时间仅为 12.8 μs（含保护间隔），一帧 1500 字节的数据在 256-QAM 调制下仅需 ~28 μs 的空中传输时间。
  • 5 GHz / 6 GHz 频段：相比 2.4 GHz，高频段的信道更多、干扰更少（避开蓝牙、微波炉等干扰源），竞争退避时间更短。
  • URWB 对 PHY 的增强：虽然 URWB 不修改 PHY 标准，但通过 Prodigy MAC 的智能调度，确保 PHY 层始终在最优参数下工作——自动选择最佳 MCS、动态调整发射功率、快速响应信道质量变化。

8.2.2 维度二：Prodigy MAC 层——确定性时隙调度

这是 URWB 低延迟的最核心差异化所在。传统 802.11 MAC（DCF / EDCA）基于 CSMA/CA 竞争机制，节点必须"先听后说"，在高负载或多节点环境下，竞争退避可能导致帧等待数毫秒甚至数十毫秒才能获得发送机会。

8.2.3 维度三：Overlay 层——MPLS 标签交换的硬件级转发

传统 IP 路由的每一跳都需要查询路由表（最长前缀匹配），计算复杂度为 O(log n)。而 MPLS 标签交换的核心操作是：

  1. 读取 4 字节标签（固定位置、固定长度）
  2. 查询 LFIB 表（精确匹配，O(1) 复杂度）
  3. 替换标签（写入新标签值）
  4. 转发到出接口

整个过程可以在 ASIC / FPGA 中以硬件速度完成，单跳处理延迟 < 5-10 μs。相比之下，软件路由查询通常需要 50-200 μs。

在 URWB 节点中，虽然 MPLS 转发是由嵌入式处理器而非专用 ASIC 完成的，但由于标签交换的简单性，处理延迟仍然非常低——典型值约 20-50 μs，远低于无线传输本身的延迟。

8.2.4 维度四：帧调度与优先级——确保关键流量"插队"

即使 MAC 层提供了确定性时隙，如果发送队列中的帧排列不当，关键数据仍然可能在缓冲区中等待。URWB 在帧调度层面实现了多级优先级机制：

  • Strict Priority Queuing：将流量分为 4-8 个优先级队列。最高优先级的控制帧（如 MBB 信令、紧急停机信号）始终优先发送，不受其他流量影响。
  • DSCP / 802.1p 映射：来自有线侧的 QoS 标记被映射到 URWB 内部优先级，确保端到端 QoS 一致性。
  • 帧聚合优化：对于小帧（如 PLC 控制指令，通常 64-128 字节），URWB 会根据延迟要求决定是否等待聚合。对高优先级小帧，立即发送而不等待聚合，牺牲些许频谱效率换取最低延迟。
  • MBB 信令超级优先级：漫游相关的控制帧（Handoff 请求/响应、LSP 建立信令）拥有最高优先级，确保漫游过程不会被业务流量延迟。

8.2.5 维度五：零中断漫游——MBB 消除移动性延迟

在移动场景中，漫游延迟往往是端到端延迟中最大的"毛刺"。传统 Wi-Fi 即使使用 802.11r 快速漫游，仍有 30-50 ms 的中断，对于 10 ms 延迟要求的应用来说是灾难性的。

8.2.6 维度六：多路径冗余——MPO 平滑延迟抖动

即使在非漫游状态下，无线信道的瞬时质量波动也会导致延迟抖动。MPO 通过以下方式平滑抖动：

  • 复制模式下的"最先到达"策略：同一帧的两个副本分别经两条路径传输，接收端取先到达的那个。由于两条路径的延迟波动通常不相关（不同信道、不同基站），综合后的延迟取两个随机变量的最小值，其方差显著小于单路径。
  • 量化示例：假设单路径延迟服从均值 3 ms、标准差 1 ms 的正态分布。双路径"取最小"后，等效均值降至约 2.4 ms，标准差降至约 0.6 ms。延迟的 99 百分位从 5.3 ms 降至约 3.8 ms。

8.6 延迟测量方法

部署完成后，如何准确测量端到端延迟以验证是否达标？以下是推荐的测量方法：

方法一：硬件打时间戳（最精确）

使用 TAP 设备（如 ProfiShark）分别在 AGV PLC 以太网口和服务器网卡处抓包，两端通过 PTP（IEEE 1588）或 GPS 精确同步时钟。比较同一帧在两端的时间戳差即为单向延迟。精度可达 ±1 μs。

方法二：ICMP Ping（简易）

从 AGV 端设备运行 ping -i 0.01 -s 128 <服务器IP>（10 ms 间隔、128 字节载荷），采集 1000+ 个 RTT 样本，统计平均值、P95、P99。注意 ICMP 的处理优先级可能低于实际业务流量，测量结果偏保守。

方法三：iPerf3 + 延迟测量

使用 iperf3 -c <服务器> -u -b 10M -l 128 --timestamps，产生 UDP 流量并记录每帧延迟。适合测量负载下的延迟表现。

方法四：URWB 内建诊断

URWB 设备提供 CLI 命令 show wireless stats latency（具体命令因固件版本而异），显示每条 LSP 的实时延迟统计，包括最小、最大、平均值和抖动。这是最直接的无线段延迟数据来源。

8.7 本章小结

URWB 的超低延迟并非来自单一"黑科技"，而是在协议栈的每一层系统性优化的结果：

  1. PHY 层：继承 802.11ac/ax 的高效 OFDM 调制，空口传输仅需数十微秒。
  2. Prodigy MAC：确定性时隙调度取代 CSMA/CA 竞争，将 MAC 层延迟从毫秒级降至亚毫秒级，且有界可预测。
  3. Overlay MPLS：标签交换的 O(1) 复杂度确保转发延迟极低。
  4. 帧调度：多级优先级队列保障关键流量不被低优先级流量阻塞。
  5. MBB 漫游：将移动性延迟从数百毫秒降至零。
  6. MPO 冗余："最先到达"策略有效平滑延迟抖动。

端到端延迟预算分析显示，在典型 AGV 场景下，URWB 可实现 ~1.7 ms 的 RTT，对比传统 Wi-Fi 的 5-500 ms（含漫游）有一到两个数量级的优势。这使得原本"只有有线才能做"的实时控制应用，第一次可以安全地迁移到无线平台上。

9.2 案例一：矿井自动化通信

🏭 场景描述

某大型地下金属矿山，巷道总长度超过 50 公里，部署了 30+ 台无人矿卡（Autonomous Haul Truck）和 10+ 台远程遥操作铲运机（Teleremote LHD）。矿卡在地下巷道中以 15-30 km/h 速度行驶，需要实时与地面控制中心通信。

需求维度具体要求
覆盖范围主运输巷道、采矿工作面、卸矿站、充电区域——总覆盖面积约 20 km 线性距离。
延迟要求矿卡自动驾驶控制：RTT < 20 ms。遥操作铲运机：RTT < 15 ms（含多路视频回传）。
带宽要求每台矿卡：~5 Mbps（控制 + 遥测 + 1 路 720p 视频）。每台遥操作 LHD：~25 Mbps（控制 + 4 路 720p 视频 + 1 路 LIDAR 点云）。
可靠性紧急停机信号必须在 50 ms 内到达。单点故障不得导致通信完全中断。
环境挑战巷道弯曲、交叉口多、岩壁反射强烈导致多径效应严重。粉尘、潮湿、温度变化大（10-40°C）。空间狭窄，天线安装受限。

参考设计方案

  • 基站：沿主巷道每 200-300 米安装一台 IW9165D（IP67，适合矿井潮湿环境），使用定向天线沿巷道方向覆盖。在巷道交叉口和弯道处加密部署。总计约 80-100 台基站。
  • 车载设备：每台矿卡/LHD 安装一台 IW9165E，双 5 GHz 射频配合车顶全向天线。
  • 拓扑：Mobility（Fluidity）拓扑 + MPO 复制模式。L2 Fluidity（矿卡数量 < 200，单一广播域可行）。
  • 有线骨干：巷道顶部敷设工业级光纤环网（Catalyst IE3x00 交换机），每 4-6 台基站汇聚到一台交换机，交换机通过光纤环网连接到地面控制中心。
  • Coordinator：部署在地面控制中心的机房中，两台 IW9165E（或服务器上的虚拟 Coordinator）互为热备（TITAN failover）。
  • 信道规划：使用两个非 DFS 信道交替部署（如信道 36 和 149），确保 MPO 双路径独立。
效果：该设计在实际类似项目中验证了以下性能指标——RTT < 8 ms（P99），漫游零丢包（MBB），通信可用性 > 99.999 %（MPO 复制模式），支持矿卡以 30 km/h 速度连续行驶无中断。紧急停机信号端到端延迟 < 15 ms。

9.3 案例二：港口自动化 AGV

🚢 场景描述

某大型集装箱港口，码头面积约 2 km × 0.8 km，部署 60+ 台 AGV（自动导引车）在堆场与岸桥之间运输集装箱。AGV 运行速度 5-25 km/h，需要实时接收调度指令、上报位置、传输安全传感器数据。

需求维度具体要求
覆盖范围整个码头堆场（开阔区域），包括堆场通道、岸桥下方、转弯区。需全覆盖无死角。
延迟要求AGV 运动控制：RTT < 10 ms。调度指令：< 100 ms。
带宽要求每台 AGV：~2-3 Mbps（控制 + 传感器 + 1 路低码率视频）。总计约 200 Mbps。
密度挑战高峰时段 60+ 台 AGV 同时在线，部分区域（如岸桥下方排队区）可能有 10+ 台 AGV 在同一基站覆盖范围内。
环境挑战集装箱堆场金属反射严重，堆叠的集装箱（高 15-20 米）造成遮挡。海风、盐雾、高温。

参考设计方案

  • 基站：在堆场灯杆顶部（高度 25-30 米）安装 IW9167E（多射频、高密度能力），每台覆盖约 150-200 米半径。总计约 30-40 台基站，确保全覆盖且重叠区 ≥30 %。
  • 车载设备：每台 AGV 安装 IW9165E，天线安装在 AGV 顶部（高于集装箱锁扣，约 1.5 米高度），双极化全向天线。
  • 拓扑：Mobility（Fluidity）+ MPO。由于 AGV 数量 > 50，推荐 L3 Fluidity（每台 AGV 一个 /30 子网），避免广播域过大。
  • Coordinator：部署在码头中心机房，双机热备。通过 Catalyst 9500 核心交换机连接所有基站。
  • 信道规划：利用 5 GHz 的 4 个 80 MHz 不重叠信道（36、52、100、149）对基站进行四色着色，最小化同频干扰。MPO 双路径分别使用不同颜色的基站。
  • 管理：通过 Cisco Catalyst 9800 WLC（17.18.1+固件）统一管理所有基站和车载设备，与港口 TOS（Terminal Operating System）集成监控。
效果：类似港口项目实测指标——RTT < 5 ms（P99），AGV 从岸桥到堆场全程零丢包漫游，单基站稳定支持 12+ 台 AGV 并发连接。系统支持 AGV 队列模式运行（多台 AGV 编队以 20 km/h 行驶），编队间距缩短至 3 米（依赖低延迟通信实现安全避碰）。

9.4 案例三：轨道交通 CBTC / 车地通信

🚄 场景描述

某城市轨道交通新建线路，全长 35 公里，30 个站点，设计最高运行速度 120 km/h。需要建设可靠的车地无线通信系统，承载 CBTC（基于通信的列车控制）信号以及乘客信息系统（PIS）视频。

需求维度具体要求
覆盖范围全线 35 km 无缝覆盖，包括地上段、地下隧道段、车站、道岔区。
延迟要求CBTC 控制信号：单向 < 50 ms，RTT < 100 ms（含冗余处理）。PIS 视频：< 500 ms。
带宽要求CBTC：< 1 Mbps（控制指令，帧小但频率高）。PIS：10-20 Mbps（多路视频）。
速度挑战120 km/h → 每秒移动 33.3 米。基站间距 ~500 米时，约每 15 秒漫游一次。
可靠性要求CBTC 系统要求通信可用性 > 99.999 %，丢包率 < 0.01 %，漫游中断 < 50 ms（实际越低越好）。
安全等级CBTC 属于 SIL-2 安全等级系统，通信中断超过 3 秒将触发紧急制动。

参考设计方案

  • 轨旁基站：沿轨道每 300-500 米安装一台 IW9165D，定向天线沿轨道方向覆盖。隧道段使用漏缆或高增益定向天线延伸覆盖。总计约 80-120 台基站。
  • 车载设备：每列车安装 2 台 IW9165E（车头和车尾各一台，互为冗余）。每台 IW9165E 的双射频接口同时连接不同基站（MPO + MBB）。
  • 拓扑：Mobility（Fluidity）+ MPO 复制模式。L2 Fluidity（列车数量通常 < 50，广播域可控）。
  • 冗余架构：
    • 车载双设备冗余：车头 IW9165E-A + 车尾 IW9165E-B，通过车内以太网互联，任一设备故障另一台立即接管。
    • MPO 复制模式：每帧数据同时经两条路径传输。
    • Coordinator 双机热备（TITAN failover < 500 ms）。
    • 有线骨干光纤环网冗余（ERPS < 50 ms 切换）。
  • 信道规划：全线使用两个非 DFS 信道交替部署（信道 36、149），隧道段由于无外部干扰，可额外使用 DFS 信道增加容量。
效果：URWB 已在全球多条轨道交通线路中验证（含 Fluidmesh 时代的部署）。典型指标——120 km/h 速度下漫游零丢包（MBB），RTT < 20 ms（P99），通信可用性 > 99.9999 %（双设备 + MPO + TITAN 综合冗余），满足 SIL-2 安全要求。Cisco 已将 URWB 解决方案在速度高达 350+ km/h 的高铁场景中进行过概念验证测试，展示了技术的速度极限能力。

10.2 物理安装要点

10.2.1 路侧设备（IW9165D）安装

  1. 安装高度：建议天线中心距地面 4 ~ 6 m，确保菲涅尔区第一区无遮挡。
  2. 天线朝向：两台路侧设备的天线主瓣应朝向导轨方向，使车辆全程处于主瓣覆盖内。若使用定向天线，水平波束宽度应 ≥ 90°。
  3. 间距规划：两台路侧设备间距建议 150 ~ 250 m（5 GHz 全向天线 + 20 dBm 发射功率条件下），确保中间有至少 30 ~ 50 m 的重叠覆盖区以完成 Make-Before-Break 切换。
  4. 接地：将设备接地端子通过 ≤ 10 AWG 铜线连接至建筑等电位接地排（阻抗 < 5 Ω）。
  5. 网线走线：使用室外级屏蔽网线，避免与高压电缆平行敷设；弯曲半径 ≥ 8 倍线缆外径。

10.2.2 车载设备（IW9165E）安装

  1. 位置选择：安装在车顶最高点，尽量远离金属遮挡物和其他 RF 发射源。
  2. 天线选择：推荐使用全向天线（Omni），增益 3 ~ 5 dBi；若车辆仅沿固定方向行驶，可使用 120° 扇区天线以提升链路裕量。
  3. 防振处理：使用防振垫或弹性支架，避免长期振动导致连接器松动或天线辐射模式畸变。
  4. 供电：车载设备可通过车载 PoE 交换机或 DC 电源适配器（48 VDC）供电。确保电源纹波 ≤ 200 mVpp。
⚠️ 安全警告：高空作业请遵守当地职业安全法规，佩戴安全带与安全帽。室外安装必须先确认设备符合当地无线电频率法规，并按需申请射频许可。

10.6 漫游功能测试

10.6.1 测试方法

  1. 持续 Ping：在管理笔记本上运行 ping 192.168.10.3 -t -l 1000（Windows）或 ping -i 0.01 192.168.10.3（Linux），以 10 ms 间隔持续 Ping Vehicle。
  2. 移动 Vehicle：将搭载 IW9165E 的 AGV 或测试推车从 IW9165D #1 附近缓慢推向 IW9165D #2。
  3. 观察切换：在 Vehicle 上执行 show fluidity config，观察 "Current Base" 字段从 192.168.10.1 变为 192.168.10.2。
  4. 记录丢包：切换瞬间 Ping 序列中应无丢包或最多 1 个包丢失（Make-Before-Break 机制保证连续性）。
  5. 反向测试：将车辆推回 #1 附近，确认反向漫游同样无缝。

10.6.2 iPerf3 吞吐量 + 漫游测试

在管理笔记本（192.168.10.100）上运行 iPerf3 Server，在 Vehicle 下挂设备上运行 Client：

# 笔记本端（Server）：
iperf3 -s -p 5201

# 车载端（Client，通过 Vehicle 下挂以太网连接的设备）：
iperf3 -c 192.168.10.100 -p 5201 -t 120 -i 1 -b 200M

在 120 秒测试过程中推动车辆完成漫游。观察 iPerf3 逐秒输出：

  • 非漫游时段：吞吐量应稳定在 150 ~ 300 Mbps（视信道条件与距离）。
  • 漫游瞬间：吞吐量可能短暂下降至 50 ~ 100 Mbps（约 1 ~ 2 秒），但不会归零。
  • 漫游后：吞吐量恢复至正常水平。

10.6.3 验证 MPLS 隧道迁移

# 漫游前在 IW9165D #1 上查看：
Device-1> show mpls config
  LSP to Vehicle (192.168.10.3): Label-In=1001, Label-Out=2001, Status=Active

# 漫游后在 IW9165D #2 上查看：

💡 调试技巧：如果漫游测试中出现持续丢包（> 3 秒），请检查以下几点：
  • 两台 Base 的有线连接是否都在同一 VLAN（L2 可达）
  • 上行交换机 MTU 是否 ≥ 1544（推荐 1600）
  • 两台 Base 的无线口令是否完全一致（区分大小写）
  • 覆盖重叠区是否足够（建议 ≥ 30 m）
  • 射频信道是否有外部干扰（使用 show eng-stats 查看 Noise Floor）

本术语表汇总了全文涉及的所有关键技术术语，按英文字母顺序排列，附中文释义。

英文术语 / 缩写 中文译名 释义
AES (Advanced Encryption Standard) 高级加密标准 对称加密算法，URWB 使用 AES-128 对无线帧进行加密，确保 Over-the-Air 数据机密性。
AGV (Automated Guided Vehicle) 自动导引车 在工厂、仓库或港口中沿预设路径或自主导航运行的无人运输车辆，是 URWB 的典型移动终端载体。
ARP (Address Resolution Protocol) 地址解析协议 将 IPv4 地址映射为 MAC 地址的二层协议。URWB Overlay 层提供 ARP 优化以减少广播风暴。
AutoTap 自动流量监听 URWB 内置的网络诊断功能，可在不中断业务的情况下镜像特定流量供分析工具捕获。
BBM (Break-Before-Make) 先断后连 传统 Wi-Fi 漫游方式：先断开旧连接，再建立新连接。切换期间存在数十至数百毫秒的中断。
BPDU (Bridge Protocol Data Unit) 桥接协议数据单元 STP 协议的控制帧。URWB 设备端口建议启用 BPDUGuard 以防止拓扑震荡。
CBTC (Communications-Based Train Control) 基于通信的列车控制 利用无线通信实现列车与轨旁设备实时信息交互的信号系统，对延迟和可靠性要求极高（RTT < 10 ms）。
CLI (Command-Line Interface) 命令行接口 通过文本命令管理设备的接口。URWB Standalone 模式完全通过 CLI 配置。
Coordinator 协调器 URWB 网络中的核心控制节点，负责 MPLS LSP 建立、拓扑管理和 Fluidity 漫游决策。通常由一台 Mesh End 自动选举承担。
CPS (Composite Path Score) 综合路径得分 MPO 路径质量评估的核心指标，综合考虑 RSSI、SNR、PER、延迟等多维参数。
DFS (Dynamic Frequency Selection) 动态频率选择 5 GHz 频段中用于避让雷达信号的强制性机制。设备检测到雷达后必须在 10 秒内切换信道。
EIRP (Equivalent Isotropically Radiated Power) 等效全向辐射功率 发射功率与天线增益的总和，需符合各国无线电法规上限。
Fluidity 流动性（Cisco 专有协议名） Cisco URWB 的核心移动性协议，实现 Make-Before-Break 无缝漫游、路径优化和拓扑自愈。
FluidMAX 增强流动性（Cisco 专有） Fluidity 的增强版本，支持 MU-MIMO 与 OFDMA 调度，进一步提升多车场景下的频谱效率。
IGMP (Internet Group Management Protocol) 因特网组管理协议 管理组播组成员关系的协议。URWB 在 Overlay 层优化组播转发以减少不必要的无线广播。
IoT OD (IoT Operations Dashboard) 物联网运维仪表盘 Cisco 云端管理平台，可对 URWB 设备进行远程配置、监控和固件升级。
IW9165D — Cisco Catalyst IW9165D 系列重型工业级无线接入点，支持双 5 GHz 射频，IP67 防护，适用于路侧基础设施部署。
IW9165E — Cisco Catalyst IW9165E 系列轻型工业级无线客户端，设计用于车载或移动平台安装。
IW9167E — Cisco Catalyst IW9167E 系列，支持 Wi-Fi 6E (6 GHz)，具有三射频架构和更高吞吐量，面向下一代 URWB 部署。
L2 Fluidity 二层流动性模式 所有设备处于同一广播域 / VLAN，MPLS 隧道承载原始 L2 帧，漫游时 IP 地址不变。适用于简单拓扑。
L3 Fluidity 三层流动性模式 路侧设备和车载设备可处于不同子网，MPLS 隧道在 L3 层路由。适用于大规模多区域部署。
LACP (Link Aggregation Control Protocol) 链路聚合控制协议 有线以太网的链路捆绑协议。URWB 的 MPO 在无线域中实现了类似 LACP 的多路径冗余功能。
LSP (Label Switched Path) 标签交换路径 MPLS 网络中数据包沿标签转发的预定路径。URWB 为每个终端 / VLAN 建立专用 LSP。
MBB (Make-Before-Break) 先连后断 URWB 的核心漫游策略：新链路建立并验证成功后，才释放旧链路，实现零感知切换。
Mesh End 网格末端 URWB 网络中的固定基础设施节点，通常连接有线网络，等同于传统网络中的"接入点 + 交换机"角色。
Mesh Point 网格节点 URWB 网络中的无线接入节点或中继节点，可以是固定的（扩展覆盖）或移动的（Vehicle）。
MIMO (Multiple-Input Multiple-Output) 多输入多输出 利用多根天线同时发送和接收多个数据流以提高吞吐量和可靠性的射频技术。
MLO (Multi-Link Operation) 多链路操作 Wi-Fi 7 的核心特性，允许单一关联跨多频段同时收发，与 URWB MPO 架构具有天然协同潜力。
MPLS (Multi-Protocol Label Switching) 多协议标签交换 基于标签（而非 IP 地址）转发数据包的网络技术。URWB 在无线域中创新性地使用 MPLS 作为 Underlay 传输机制。
MPO (Multi-Path Optimization) 多路径优化 URWB 利用双射频在不同信道建立两条独立路径，实现负载分担与快速故障切换。
MTU (Maximum Transmission Unit) 最大传输单元 网络接口一次可发送的最大数据包字节数。URWB MPLS 封装要求上行交换机 MTU ≥ 1544，建议 1600。
MU-MIMO (Multi-User MIMO) 多用户多输入多输出 允许接入点同时向多个客户端发送 / 接收不同数据流，显著提升多终端密集场景的频谱效率。
NTP (Network Time Protocol) 网络时间协议 用于网络设备时间同步的协议。URWB 设备配置 NTP 可确保日志时间戳一致性和协议定时精度。
OFDMA (Orthogonal Frequency-Division Multiple Access) 正交频分多址 Wi-Fi 6/6E/7 引入的多址技术，将信道划分为多个资源单元（RU）分配给不同用户，降低延迟。
Overlay 覆盖层 / 逻辑层 运行在 Underlay 之上的虚拟网络层。URWB Overlay 负责 VLAN 透传、ARP 优化和业务流量的逻辑转发。
P2MP (Point-to-Multipoint) 点对多点 一个中心节点与多个远端节点通信的拓扑。URWB 中 Mesh End 到多个 Mesh Point 的连接即为 P2MP。
P2P (Point-to-Point) 点对点 两个节点之间直接连接的最简拓扑，常用于建筑间无线桥接。
PER (Packet Error Rate) 误包率 接收端检测到错误的数据包占总接收包数的比例，是衡量无线链路质量的关键指标。
PHY (Physical Layer) 物理层 OSI 七层模型的第一层，负责比特流的调制、编码和无线传输。URWB PHY 基于 IEEE 802.11ac/ax 标准。
PLC (Programmable Logic Controller) 可编程逻辑控制器 工业自动化的核心控制设备，通常通过以太网与上位系统通信，是 URWB 车载终端下挂的典型设备。
PoE (Power over Ethernet) 以太网供电 通过以太网线缆同时传输数据和电力的技术。URWB 设备要求 802.3bt（PoE++，60W+）。
QoS (Quality of Service) 服务质量 网络优先级调度机制，确保关键业务流量（如控制指令）获得低延迟和高优先级转发。
RSSI (Received Signal Strength Indicator) 接收信号强度指示 衡量无线接收信号功率的指标，单位为 dBm。URWB 漫游决策的关键输入参数之一。
RTT (Round-Trip Time) 往返延迟 数据包从源端发出到收到响应的总时间。URWB 典型 RTT < 2 ms。
SNR (Signal-to-Noise Ratio) 信噪比 有用信号功率与背景噪声功率的比值（dB），直接影响调制方式和吞吐量。≥ 25 dB 为良好。
SPD (Surge Protective Device) 浪涌保护器 / 防雷器 保护设备免受雷击和电涌损坏的器件，室外部署 URWB 设备时必须安装。
STP (Spanning Tree Protocol) 生成树协议 防止二层网络环路的协议。连接 URWB 设备的交换机端口应配置 PortFast 以加速收敛。
TITAN — Cisco URWB MPLS 快速故障切换机制的内部代号，可在 < 500 ms 内完成备用路径切换。
TSN (Time-Sensitive Networking) 时间敏感网络 IEEE 802.1 系列标准定义的确定性以太网技术，保证有界延迟和零丢包。URWB 被视为 TSN-over-Wireless 的潜在承载层。
Underlay 底层网络 承载 Overlay 流量的基础传输网络。URWB Underlay 包含 PHY、MAC、MPLS 三个核心子层。
URWB (Ultra-Reliable Wireless Backhaul) 超可靠无线回传 Cisco 面向工业移动场景的无线回传技术体系，前身为 Fluidmesh 技术，于 2020 年被 Cisco 收购并整合。
VLAN (Virtual Local Area Network) 虚拟局域网 在物理网络上划分逻辑广播域的技术。URWB 支持跨无线链路的完整 VLAN 透传。
WGB (Workgroup Bridge) 工作组桥 将有线设备桥接到无线网络的功能模式。IW9165E 支持 WGB/uWGB 模式连接传统 Cisco AP。
Wi-Fi 6 / 6E / 7 — 分别对应 IEEE 802.11ax（Wi-Fi 6/6E）和 802.11be（Wi-Fi 7）标准。URWB 当前基于 Wi-Fi 5/6，下一代将支持 Wi-Fi 7。
