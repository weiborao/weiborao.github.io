🔎 Overview 🔎 概述

802.1X is the IEEE standard for Port-Based Network Access Control (IEEE 802.1X-2020, Clause 6). It defines an architecture comprising three roles: the Supplicant (the endpoint requesting access, e.g., an AGV), the Authenticator (the network device enforcing access, e.g., an AP or switch), and the Authentication Server (the policy engine, e.g., Cisco ISE). Communication between the Supplicant and Authenticator uses EAPOL (EAP over LAN, EtherType 0x888E), while the Authenticator relays EAP messages to the Authentication Server via RADIUS (RFC 2865 / RFC 3579).

802.1X 是 IEEE 制定的基于端口的网络访问控制标准（IEEE 802.1X-2020，第6章）。它定义了三个角色：请求者（Supplicant）（请求接入的终端，如 AGV）、认证者（Authenticator）（执行访问控制的网络设备，如 AP 或交换机）和认证服务器（Authentication Server）（策略引擎，如 Cisco ISE）。请求者与认证者之间使用 EAPOL（EAP over LAN，以太类型 0x888E）通信，认证者通过 RADIUS（RFC 2865 / RFC 3579）将 EAP 消息中继到认证服务器。

EAP-TLS (EAP Type 13, defined in RFC 5216 and updated by RFC 9190 for TLS 1.3) is the gold-standard EAP method. It provides mutual authentication using X.509 certificates on both sides — no passwords are transmitted. The TLS handshake establishes a secure tunnel, derives keying material (MSK and EMSK), and upon success, the Authenticator opens the controlled port, granting the Supplicant network access.

EAP-TLS（EAP 类型 13，由 RFC 5216 定义，RFC 9190 更新以支持 TLS 1.3）是业界公认的最高安全级别 EAP 方法。它使用双方的 X.509 证书进行双向认证——不传输密码。TLS 握手建立安全隧道，派生密钥材料（MSK 和 EMSK），认证成功后，认证者开放受控端口，授予请求者网络访问权限。

🏭 Industrial Context: AGV Deployments
🏭 工业场景：AGV 部署

Automated Guided Vehicles (AGVs) in warehouses and factories require secure, seamless wireless authentication. EAP-TLS with Cisco ISE ensures that each AGV is uniquely identified by its device certificate, enabling fine-grained authorization policies (VLAN assignment, dACL, SGT) while supporting fast roaming across APs.

仓库和工厂中的自动导引车（AGV）需要安全、无缝的无线认证。使用 Cisco ISE 的 EAP-TLS 确保每台 AGV 通过其设备证书被唯一识别，支持细粒度授权策略（VLAN 分配、dACL、SGT），同时支持跨 AP 快速漫游。

📌 Controlled vs. Uncontrolled Port
📌 受控端口与非受控端口

IEEE 802.1X-2020 §6.3.2 defines the PAE (Port Access Entity) as having two virtual access points: the Uncontrolled Port (always open, carries only EAPOL/MKA traffic) and the Controlled Port (initially blocked, opened only after successful authentication). This separation ensures that authentication frames can always transit even when the network is "locked."

IEEE 802.1X-2020 §6.3.2 定义 PAE（端口访问实体）具有两个虚拟接入点：非受控端口（始终开放，仅承载 EAPOL/MKA 流量）和受控端口（初始阻塞，仅在认证成功后开放）。这种分离确保即使网络被"锁定"，认证帧也能始终通过。

Authenticator behavior: Upon receiving EAPOL-Start, the Authenticator's PACP (Port Access Control Protocol) state machine transitions from DISCONNECTED → CONNECTING and sends an EAP-Request/Identity to the Supplicant (IEEE 802.1X-2020 §8.2). Alternatively, the Authenticator may proactively send an EAP-Request/Identity upon detecting link-up, without waiting for EAPOL-Start.

认证者行为：收到 EAPOL-Start 后，认证者的 PACP（端口访问控制协议）状态机从 DISCONNECTED → CONNECTING 转换，并向请求者发送 EAP-Request/Identity（IEEE 802.1X-2020 §8.2）。另外，认证者也可能在检测到链路连接后主动发送 EAP-Request/Identity，无需等待 EAPOL-Start。

⚠️ RADIUS Relay Mechanics
⚠️ RADIUS 中继机制

The Authenticator (AP/Switch) acts as a pass-through for EAP messages. It does NOT interpret EAP methods — it simply wraps EAP payloads into RADIUS EAP-Message attributes (Type 79, RFC 3579 §3.1) within Access-Request packets and unwraps them from Access-Challenge / Access-Accept / Access-Reject responses from ISE. The RADIUS shared secret protects the RADIUS transport, but the EAP payload inside is opaque to the Authenticator.

认证者（AP/交换机）充当 EAP 消息的透传角色。它不解析 EAP 方法——只是将 EAP 载荷封装到 RADIUS EAP-Message 属性（类型 79，RFC 3579 §3.1）中的 Access-Request 包中，并从 ISE 的 Access-Challenge / Access-Accept / Access-Reject 响应中解封装。RADIUS 共享密钥保护 RADIUS 传输，但其中的 EAP 载荷对认证者来说是不透明的。

⚡ EAP-TLS Method Selection EAP-TLS 方法选择

Upon receiving the EAP-Response/Identity, Cisco ISE looks up its authentication policy to determine which EAP method to use. When the policy specifies EAP-TLS, ISE responds with a RADIUS Access-Challenge containing an EAP-Request with Type=13 (EAP-TLS) and the TLS Start flag set (S=1). This transitions the conversation from the generic EAP identity phase into the TLS handshake (Phase 3).

收到 EAP-Response/Identity 后，Cisco ISE 查找其认证策略以确定使用哪种 EAP 方法。当策略指定 EAP-TLS 时，ISE 以 RADIUS Access-Challenge 响应，其中包含一个类型为 13（EAP-TLS）、且设置了 TLS Start 标志（S=1）的 EAP-Request。这将会话从通用 EAP 身份阶段过渡到 TLS 握手（阶段 3）。

3c. Server Certificate Chain Validation (by Supplicant) 3c. 服务器证书链验证（由请求方执行）

Before proceeding, the supplicant (AGV's 802.1X software) MUST validate the ISE server certificate chain. This is a critical security step — failure to properly validate opens the door to man-in-the-middle attacks. The validation follows standard X.509 path validation (RFC 5280).

The root CA certificate must exist in the supplicant's local trust store. In industrial AGV deployments, this is typically provisioned during initial device setup or via MDM/SCEP enrollment.

Verify each certificate's digital signature using the issuer's public key. The Root CA is self-signed (its own public key verifies its own signature).

Server Identity Verification 服务器身份验证

Optionally verify the ISE server's identity by checking the certificate's Subject CN or Subject Alternative Name (SAN) against a configured expected server name. This prevents rogue RADIUS server attacks.

4c. Port State Transition → Authorized 4c. 端口状态转换 → 已授权

1
Controlled Port Opens 受控端口打开

The authenticator's PAE (Port Access Entity) transitions the controlled port from Unauthorized to Authorized state (IEEE 802.1X-2020 §6.3.2, §12.3). Data traffic from the AGV can now flow through the controlled port, subject to any applied VLAN assignment, dACL, or SGT policy.

认证者的 PAE（端口访问实体）将受控端口从 未授权 转换为 已授权 状态（IEEE 802.1X-2020 §6.3.2、§12.3）。AGV 的数据流量现在可以通过 受控端口传输，受已应用的 VLAN 分配、dACL 或 SGT 策略约束。

2
802.11 4-Way Handshake (Wireless Only) 802.11 四次握手（仅无线）

For wireless AGV connections via Cisco Catalyst 9800 WLC, the MSK derived from EAP-TLS becomes the PMK (Pairwise Master Key). The AP and supplicant then perform the IEEE 802.11i 4-Way Handshake to derive the PTK (Pairwise Transient Key) and GTK (Group Temporal Key) for wireless frame encryption (AES-CCMP / GCMP).

对于通过 Cisco Catalyst 9800 WLC 的无线 AGV 连接，从 EAP-TLS 派生的 MSK 成为 PMK（成对主密钥）。AP 和请求方随后执行 IEEE 802.11i 四次握手以派生 PTK（成对临时密钥）和 GTK（组临时密钥）， 用于无线帧加密（AES-CCMP / GCMP）。

3
PMKSA Caching (for Fast Roaming) PMKSA 缓存（用于快速漫游）

The PMK and its associated PMKID are cached as a PMKSA (PMK Security Association) at both the supplicant and the authenticator (AP/WLC). On subsequent associations to the same AP or within the same mobility domain, the supplicant can reference the PMKID in the (Re)Association Request to skip the full EAP-TLS handshake, performing only the 4-Way Handshake. See also: 802.11r Fast Transition (FT) and OKC (Opportunistic Key Caching) in the Deployment Notes section.

PMK 及其关联的 PMKID 作为 PMKSA（PMK 安全关联）被缓存在请求方和认证者（AP/WLC）双方。 在后续关联到相同 AP 或同一移动域内时，请求方可在（重新）关联请求中引用 PMKID 以跳过完整的 EAP-TLS 握手，仅执行四次握手。另见部署说明部分的 802.11r 快速转换（FT）和 OKC（机会性密钥缓存）。

4
AGV Network Access Active AGV 网络访问已激活

The AGV now has full network access according to its authorization policy. It can communicate with fleet management servers, receive navigation commands, report telemetry data, and coordinate with other AGVs — all within the security boundaries defined by the VLAN, dACL, and/or SGT policies pushed by Cisco ISE.

AGV 现在根据其授权策略拥有完整的网络访问权限。它可以与车队管理服务器通信、 接收导航命令、报告遥测数据，并与其他 AGV 协调——所有操作均在 Cisco ISE 推送的 VLAN、dACL 和/或 SGT 策略定义的安全边界内。

🏭 Deployment Notes — Industrial AGV with Cisco ISE 🏭 部署注意事项 — 工业 AGV 与 Cisco ISE

The following operational guidelines address certificate lifecycle, roaming performance, session management, cipher selection, revocation checking, and RADIUS infrastructure design for AGV 802.1X EAP-TLS deployments. 以下运营指南涵盖 AGV 802.1X EAP-TLS 部署中的证书生命周期、漫游性能、会话管理、密码套件选择、吊销检查和 RADIUS 基础设施设计。

📜 Certificate Management证书管理

  • Use SCEP (Simple Certificate Enrollment Protocol) or EST (Enrollment over Secure Transport, RFC 7030) for automated AGV certificate enrollment and renewal. 使用 SCEP（简单证书注册协议）或 EST（安全传输注册，RFC 7030）实现 AGV 证书的自动注册和续期。
  • Ensure the CA trust chain is consistent across all AGVs, ISE nodes, and authenticators. Import intermediate CA certificates into the ISE Trusted Certificates store. 确保所有 AGV、ISE 节点和认证方之间的 CA 信任链 一致。将中间 CA 证书导入 ISE 受信任证书库。
  • Set certificate validity to 1–2 years for device certificates; automate renewal 30 days before expiry to prevent authentication failures during AGV operation. 设备证书有效期设为 1–2 年；在到期前 30 天自动续期，防止 AGV 运行期间认证失败。
  • Use SAN (Subject Alternative Name) fields (dNSName or iPAddress) for machine identity; avoid relying solely on CN for certificate matching in ISE policies. 使用 SAN（主题备用名称） 字段（dNSName 或 iPAddress）标识设备身份；避免在 ISE 策略中仅依赖 CN 进行证书匹配。
  • Store private keys in a TPM (Trusted Platform Module) or hardware secure element on the AGV where possible to prevent key extraction. 尽可能将私钥存储在 AGV 的 TPM（可信平台模块）或硬件安全元件中，以防止密钥被提取。

⚡ Low-Latency Requirements & Roaming低延迟要求和漫游

  • Enable PMKSA Caching on the Catalyst 9800 WLC to allow AGVs to skip full EAP-TLS re-authentication when roaming between APs within the same mobility domain. 在 Catalyst 9800 WLC 上启用 PMKSA 缓存，使 AGV 在同一移动域内的 AP 之间漫游时跳过完整的 EAP-TLS 重新认证。
  • Configure OKC (Opportunistic Key Caching) as a fallback for PMKSA distribution across WLC-managed APs. 配置 OKC（机会性密钥缓存） 作为 WLC 管理的 AP 之间 PMKSA 分发的备用方案。
  • Deploy 802.11r (Fast BSS Transition) for sub-50ms roaming. Ensure over-the-DS mode is used in dense industrial RF environments for reliability. 部署 802.11r（快速 BSS 转换） 实现低于 50ms 的漫游。在密集的工业 RF 环境中使用 over-the-DS 模式以提高可靠性。
  • Full EAP-TLS handshake typically takes 6–12 round trips (100–500ms on WLAN). Target roaming scenarios should use PMKSA/OKC/FT to avoid this latency. 完整的 EAP-TLS 握手通常需要 6–12 次往返（WLAN 上 100–500ms）。漫游场景应使用 PMKSA/OKC/FT 来避免此延迟。
  • Consider pre-authentication (802.11i) to neighboring APs when AGV travel paths are predictable in warehouse/factory floor layouts. 当 AGV 在仓库/工厂车间的行驶路径可预测时，考虑对邻近 AP 进行 预认证（802.11i）。

🔄 Re-authentication & Session Management重新认证和会话管理

  • Set Session-Timeout ≥ 3600 seconds in the ISE Authorization Profile to minimize re-authentication frequency for continuously operating AGVs. 在 ISE 授权配置文件中设置 Session-Timeout ≥ 3600 秒，以减少持续运行的 AGV 的重新认证频率。
  • Set Termination-Action = RADIUS-Request (1) to ensure re-authentication occurs via a new RADIUS exchange rather than session teardown. 设置 Termination-Action = RADIUS-Request (1)，确保通过新的 RADIUS 交换进行重新认证而非终止会话。
  • Leverage RADIUS CoA (Change of Authorization) for dynamic policy updates (e.g., moving an AGV to quarantine VLAN upon security alert) without requiring re-authentication. 利用 RADIUS CoA（授权变更） 实现动态策略更新（例如在安全警报时将 AGV 移至隔离 VLAN），无需重新认证。
  • Monitor EAPOL-Logoff frames; AGVs powering down should trigger session cleanup on the authenticator and ISE. 监控 EAPOL-Logoff 帧；AGV 关机时应触发认证方和 ISE 上的会话清理。

🔐 Cipher Suite Selection密码套件选择

  • TLS 1.2 Preferred: TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384 or TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256 for optimal balance of security and performance on embedded AGV hardware. TLS 1.2 首选：TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384 或 TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256，在嵌入式 AGV 硬件上实现安全性和性能的最佳平衡。
  • TLS 1.3 Preferred: TLS_AES_256_GCM_SHA384 for maximum security; TLS_AES_128_GCM_SHA256 for resource-constrained AGV controllers. TLS 1.3 首选：TLS_AES_256_GCM_SHA384 提供最大安全性；TLS_AES_128_GCM_SHA256 适用于资源受限的 AGV 控制器。
  • Disable legacy cipher suites: CBC-mode ciphers, RC4, 3DES, SHA-1-based MACs, and static RSA key exchange must be disabled on both ISE and supplicant. 禁用旧版密码套件：必须在 ISE 和申请方上禁用 CBC 模式密码、RC4、3DES、基于 SHA-1 的 MAC 和静态 RSA 密钥交换。
  • For AGVs with hardware crypto acceleration (AES-NI equivalent), prefer AES-256-GCM. For software-only TLS stacks, AES-128-GCM offers lower CPU overhead. 对于具有 硬件加密加速（AES-NI 等效）的 AGV，优先使用 AES-256-GCM。对于仅软件 TLS 栈，AES-128-GCM 提供更低的 CPU 开销。

📋 CRL / OCSP Revocation CheckingCRL / OCSP 吊销检查

  • Configure ISE to check OCSP (Online Certificate Status Protocol) as the primary revocation method for real-time status. Set CRL as fallback. 配置 ISE 使用 OCSP（在线证书状态协议） 作为主要吊销检查方法以获取实时状态。将 CRL 设为备用。
  • Set CRL download interval appropriately (e.g., every 4–8 hours) to balance freshness against ISE resource usage. Cache CRL in ISE local store. 适当设置 CRL 下载间隔（例如每 4–8 小时），以平衡新鲜度和 ISE 资源使用。在 ISE 本地存储中缓存 CRL。
  • Enable OCSP Nonce to prevent replay attacks if the OCSP responder supports it. If not, use short-lived OCSP response caching (≤ 5 min). 如果 OCSP 响应程序支持，启用 OCSP Nonce 以防止重放攻击。否则，使用短期 OCSP 响应缓存（≤ 5 分钟）。
  • In ISE → Administration → Certificates → Certificate Store, configure the OCSP responder URL and CRL Distribution Point (CDP) for each trusted CA. 在 ISE → 管理 → 证书 → 证书库 中，为每个受信任的 CA 配置 OCSP 响应程序 URL 和 CRL 分发点（CDP）。
  • Decide ISE behavior when revocation check is unreachable: "Allow" (less secure, more available) vs. "Reject" (more secure, risk of false negatives). For safety-critical AGV operations, consider "Reject" with RADIUS redundancy. 决定吊销检查不可达时的 ISE 行为："允许"（安全性较低，可用性更高）vs. "拒绝"（安全性更高，存在误判风险）。对于安全关键的 AGV 操作，建议配合 RADIUS 冗余使用"拒绝"。

🌐 RADIUS Redundancy & InfrastructureRADIUS 冗余与基础设施

  • Deploy at least two ISE PSN (Policy Service Nodes) — primary and secondary — for RADIUS high availability. Configure on the WLC / switch using radius server groups with failover. 部署至少 两个 ISE PSN（策略服务节点） — 主节点和备用节点 — 以实现 RADIUS 高可用性。在 WLC / 交换机上使用 radius server 组配置故障转移。
  • Use RADIUS load balancing on the Catalyst 9800 WLC (radius-server load-balance method least-outstanding) to distribute authentication requests across PSNs. 在 Catalyst 9800 WLC 上使用 RADIUS 负载均衡（radius-server load-balance method least-outstanding）将认证请求分配到各 PSN。
  • Set RADIUS dead-server detection with appropriate timeout (e.g., radius-server dead-criteria time 5 tries 3) to quickly failover without impacting AGV connectivity. 设置 RADIUS 死服务器检测（例如 radius-server dead-criteria time 5 tries 3），以便快速故障转移而不影响 AGV 连接。
  • Ensure RADIUS shared secrets are strong (≥ 22 characters, mixed case + symbols) and unique per authenticator/ISE pair. Use RADIUS/DTLS (RFC 7360) where supported for transport encryption. 确保 RADIUS 共享密钥 强度足够（≥ 22 个字符，大小写混合 + 符号）且每对认证方/ISE 唯一。在支持的设备上使用 RADIUS/DTLS（RFC 7360）进行传输加密。
  • Place ISE PSN nodes local to the factory/warehouse site to minimize RADIUS round-trip latency. Avoid WAN-dependent RADIUS paths for real-time AGV authentication. 将 ISE PSN 节点部署在 工厂/仓库站点本地，以最小化 RADIUS 往返延迟。避免实时 AGV 认证依赖 WAN 的 RADIUS 路径。

📖 Glossary 📖 术语表

802.1X
IEEE standard for port-based Network Access Control (PNAC). Defines three roles: supplicant, authenticator, and authentication server. Current revision: IEEE 802.1X-2020. IEEE 基于端口的网络访问控制 (PNAC) 标准。定义三个角色：申请方、认证方和认证服务器。当前版本：IEEE 802.1X-2020。
EAP
Extensible Authentication Protocol (RFC 3748). A flexible framework that transports authentication methods between supplicant and authentication server via the authenticator. 可扩展认证协议（RFC 3748）。一种灵活框架，通过认证方在申请方和认证服务器之间传输认证方法。
EAP-TLS
EAP Transport Layer Security (RFC 5216 for TLS 1.2, RFC 9190 for TLS 1.3). Provides mutual certificate-based authentication — both client and server present X.509 certificates. Considered the gold standard for EAP security. EAP 传输层安全（TLS 1.2 为 RFC 5216，TLS 1.3 为 RFC 9190）。提供基于证书的双向认证——客户端和服务器均出示 X.509 证书。被认为是 EAP 安全性的最高标准。
EAPOL
EAP over LAN (IEEE 802.1X). Layer 2 encapsulation protocol for carrying EAP frames between supplicant and authenticator using EtherType 0x888E. Includes Start, Logoff, Key, and Packet frame types. 局域网上的 EAP（IEEE 802.1X）。第 2 层封装协议，使用以太类型 0x888E 在申请方和认证方之间传输 EAP 帧。包括 Start、Logoff、Key 和 Packet 帧类型。
Supplicant 申请方 (Supplicant)
The entity (AGV, endpoint device) seeking network access. Runs 802.1X client software that initiates EAPOL exchanges and provides credentials (certificates in EAP-TLS). 寻求网络访问的实体（AGV、终端设备）。运行 802.1X 客户端软件，发起 EAPOL 交换并提供凭据（EAP-TLS 中为证书）。
Authenticator 认证方 (Authenticator)
The network access device (Catalyst 9800 WLC AP or Catalyst Switch) that enforces access control. Relays EAP messages between supplicant and authentication server; controls the port state (blocked/authorized). 执行访问控制的网络接入设备（Catalyst 9800 WLC AP 或 Catalyst 交换机）。在申请方和认证服务器之间中继 EAP 消息；控制端口状态（阻塞/授权）。
RADIUS
Remote Authentication Dial-In User Service (RFC 2865). Client-server protocol used between the authenticator and ISE for AAA (Authentication, Authorization, Accounting). Carries EAP messages in RADIUS attributes. 远程认证拨入用户服务（RFC 2865）。认证方与 ISE 之间用于 AAA（认证、授权、计费）的客户端-服务器协议。在 RADIUS 属性中承载 EAP 消息。
ISE
Cisco Identity Services Engine. A centralized policy platform providing AAA, posture assessment, profiling, guest access, and BYOD management. Serves as the RADIUS/EAP authentication server in 802.1X deployments. Cisco 身份服务引擎。提供 AAA、合规评估、设备分析、访客访问和 BYOD 管理的集中策略平台。在 802.1X 部署中充当 RADIUS/EAP 认证服务器。
dACL
Downloadable Access Control List. An ACL pushed from ISE to the authenticator via RADIUS during authorization. Provides per-session, per-user/device granular traffic filtering. Referenced by the Filter-Id or cisco-av-pair RADIUS attribute. 可下载访问控制列表。在授权期间通过 RADIUS 从 ISE 推送到认证方的 ACL。提供按会话、按用户/设备的精细流量过滤。通过 Filter-Id 或 cisco-av-pair RADIUS 属性引用。
SGT
Scalable Group Tag (Cisco TrustSec). A 16-bit security label assigned during authorization that enables software-defined segmentation. SGT-based policies (SGACLs) enforce access between groups without traditional ACLs. 可扩展组标签（Cisco TrustSec）。在授权期间分配的 16 位安全标签，实现软件定义的网络分段。基于 SGT 的策略（SGACL）无需传统 ACL 即可在组之间实施访问控制。
PMKSA
Pairwise Master Key Security Association. Cached result of a successful 802.1X/EAP authentication in wireless networks. Allows fast reconnection and roaming by skipping full EAP re-authentication. Identified by PMKID. 成对主密钥安全关联。无线网络中成功 802.1X/EAP 认证的缓存结果。通过跳过完整的 EAP 重新认证实现快速重连和漫游。由 PMKID 标识。
802.11r (FT)
IEEE 802.11r — Fast BSS Transition. An amendment to 802.11 that enables fast, secure handoffs between APs by pre-distributing PMK-R1 keys. Targets < 50 ms roaming for latency-sensitive applications like AGV navigation. IEEE 802.11r — 快速 BSS 转换。802.11 的修订版，通过预分发 PMK-R1 密钥实现 AP 之间的快速安全切换。目标漫游时间 < 50ms，适用于 AGV 导航等延迟敏感应用。
